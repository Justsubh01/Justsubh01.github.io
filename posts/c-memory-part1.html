C Memory Internals #1: Stack, Heap, Pointers & Segments

Before you write CUDA kernels, you need to understand how C manages memory — not at the "I can copy-paste malloc" level, but at the "I know exactly which byte lives where and why" level. Part 1 of 3: stack vs heap with real addresses, pointer arithmetic from first principles, all 5 memory segments, and the fragmentation trap hidden inside dynamic 2D arrays.

I spent two weeks confused by CUDA memory management before I realised the problem had nothing to do with CUDA. The problem was that I didn't really understand C memory. I knew the syntax. I could write malloc and free. But I couldn't tell you where memory lived, why pointers worked the way they did, or what the five distinct memory regions of a running process actually were.

So I stopped. I went back to basics and wrote 10 small focused programs — each one designed to make one memory concept completely visible. No handwaving. Real addresses printed to the terminal. Real byte counts. This is the first four of those programs, covering what I now think of as the four prerequisites for GPU programming.

GPU memory is just heap memory on a different chip. If you don't understand CPU heap, you won't understand cudaMalloc. Every concept in this post maps directly to something in CUDA — I'll call those out as we go.

Program 1 — Stack vs Heap

The mental model people start with (and why it's incomplete)

Most beginners think of memory as one big box — declare a variable, it goes in the box, use it, done. That model works until you hit malloc, or until you try to return a local variable's address from a function, or until your server starts eating RAM overnight. The real picture is two boxes with very different rules, and understanding which box a variable lives in explains almost every memory bug in C.

The Stack

Every function call gets a block of memory called a stack frame. This frame is reserved the moment the function is entered and destroyed the moment it returns. All local variables you declare inside a function — the int i in your loop, the float temp you compute midway — live in that frame.

The OS allocates stack space for your process upfront. On Linux it's typically 8 MB. That 8 MB is it — there's no growing it at runtime. Allocation on the stack is essentially free: it's just a register decrement. The CPU keeps a register called rsp (stack pointer) that points to the current top of the stack. "Allocating" a local variable means subtracting its size from rsp. That's one instruction.

The Heap

The heap is the opposite of the stack in almost every way. It's not bounded at 8 MB — it can grow to fill all available RAM. Variables on the heap don't disappear automatically — they live until you explicitly call free(). And allocation isn't free: malloc() has to search for a suitable free block, update bookkeeping data structures, and return you a pointer. It's measurably slower than stack allocation.

But the heap's lifetime flexibility is what makes it essential. You can allocate memory in one function and return it to be used in another. You can allocate gigabytes. You can resize allocations. None of that is possible on the stack.

The stack grows downward in virtual memory — each new function call pushes a frame at a lower address. The heap grows upward — each malloc() gives you a block at a progressively higher address. They grow toward each other; in the middle is unmapped virtual address space that will cause a segfault if accessed.

Here's the program. Compile it and look at the actual addresses:

#include <stdio.h>
#include <stdlib.h>

int main() {
    // ── STACK variables ──────────────────────────────
    // These are declared locally. The OS allocates their
    // space in main()'s stack frame before main() even runs.
    int   stack_int   = 42;
    float stack_float = 3.14f;
    char  stack_char  = 'A';

    // ── HEAP variables ───────────────────────────────
    // malloc() asks the OS for a fresh block of memory.
    // The pointer (heap_arr) is on the STACK.
    // The data it points to is on the HEAP.
    int *heap_arr = (int *)malloc(5 * sizeof(int));
    if (!heap_arr) { fprintf(stderr, "malloc failed\n"); return 1; }

    for (int i = 0; i < 5; i++)
        heap_arr[i] = (i + 1) * 10;  // 10, 20, 30, 40, 50

    // Print both sets of addresses
    printf("── STACK ──\n");
    printf("  stack_int   @ %p  value=%d\n",  (void*)&stack_int,   stack_int);
    printf("  stack_float @ %p  value=%.2f\n",(void*)&stack_float, stack_float);
    printf("  stack_char  @ %p  value=%c\n",  (void*)&stack_char,  stack_char);

    printf("\n── HEAP ──\n");
    printf("  heap_arr ptr stored @ %p  (this pointer is ON the stack)\n",
           (void*)&heap_arr);
    printf("  heap_arr data starts@ %p  (this block is ON the heap)\n",
           (void*)heap_arr);
    for (int i = 0; i < 5; i++)
        printf("  heap_arr[%d] @ %p  value=%d\n",
               i, (void*)&heap_arr[i], heap_arr[i]);

    free(heap_arr);  // ALWAYS free what you malloc
    return 0;        // stack_int, stack_float, stack_char are gone here
}

When you run this, you'll see something like:

── STACK ──
  stack_int @ 0x7ffd8a3c1dfc value=42
  stack_float @ 0x7ffd8a3c1e00 value=3.14
  stack_char @ 0x7ffd8a3c1dfb value=A

── HEAP ──
  heap_arr ptr stored @ 0x7ffd8a3c1e08 (this pointer is ON the stack)
  heap_arr data starts@ 0x5631d4a392b0 (this block is ON the heap)
  heap_arr[0] @ 0x5631d4a392b0 value=10
  heap_arr[1] @ 0x5631d4a392b4 value=20
  ... (each element 4 bytes apart)

The pattern is unmistakable. Stack addresses all start with 0x7fff — they're in the top few gigabytes of virtual address space. Heap addresses start with something like 0x563 or 0x55a — much lower. They genuinely live in different regions. And notice: heap_arr (the pointer variable itself) is on the stack at 0x7ffd... — but what it points to is on the heap at 0x5631.... Pointers live in one place, their targets in another.

Stack variables are close to each other. Look at the three stack addresses above — they're within a few bytes of each other. That's the stack frame: one contiguous block where all of main()'s locals live. Heap allocations are scattered — each malloc() can land anywhere the allocator finds free space.

The critical mistakes to avoid

1. Returning a pointer to a stack variable. This is one of the most common C bugs and one of the most confusing to debug: 

int *bad_function() {
    int x = 42;
    return &x;   // x lives on this function's stack frame
}                // frame is DESTROYED when we return
                 // the pointer now points to garbage

int main() {
    int *p = bad_function();
    printf("%d\n", *p);  // undefined behaviour — could print 42,
                          // could print 0, could crash. Anything.
}

The fix is to either return the value (not a pointer to it), or allocate on the heap and return that pointer — which the caller must then free.

2. Forgetting to free. Every malloc must have a matching free. If you lose the pointer before calling free, that memory is gone for the lifetime of the process. Run this in a loop and your process bloats: 

while (1) {
    char *buf = malloc(1024 * 1024);  // allocate 1 MB
    do_work(buf);
    // forgot free(buf) ← memory leak: 1 MB lost per iteration
}

3. Double free. Calling free(ptr) twice on the same pointer corrupts the allocator's internal data structures and typically causes a crash or exploitable behaviour: 

int *p = malloc(40);
free(p);
free(p);  // CRASH or heap corruption
// Fix: set p = NULL after free
free(p);  // free(NULL) is always safe — it does nothing
CUDA parallel: cudaMalloc(&d_ptr, size) is heap allocation on the GPU. cudaFree(d_ptr) is the free. The same rules apply: every cudaMalloc needs a cudaFree, and losing the pointer means a GPU memory leak. Unlike CPU leaks, GPU memory is far scarcer — a 16 GB GPU with a leak in a long-running inference loop will run out of VRAM much sooner than you'd expect.

Program 2 — Pointer Arithmetic

What a pointer actually is

A pointer is a variable that holds a memory address. That's it. When you write int *ptr = arr, the variable ptr contains the number 0x7ffd8a3c1e00 (or whatever the address of arr[0] is). The * in the type declaration doesn't mean ptr contains a special value — it just tells the compiler "treat the value stored in ptr as a memory address, and know that at that address you'll find an int."

On a 64-bit machine, a pointer is always 8 bytes — regardless of what it points to. sizeof(int*) == sizeof(double*) == sizeof(char*) == 8. All of them just hold an address.

Why ptr+1 isn't always +1 byte

This is the part that trips people up. When you increment a pointer by 1, the compiler doesn't add 1 to the stored address. It adds sizeof(pointed-to type). This is called pointer scaling, and the compiler does it automatically. 

#include <stdio.h>

int main() {
    // Demonstrate pointer scaling for different types
    char   c_arr[4] = {1, 2, 3, 4};
    short  s_arr[4] = {10, 20, 30, 40};
    int    i_arr[4] = {100, 200, 300, 400};
    double d_arr[4] = {1.1, 2.2, 3.3, 4.4};

    char   *cp = c_arr;
    short  *sp = s_arr;
    int    *ip = i_arr;
    double *dp = d_arr;

    printf("sizeof char  = %zu  |  cp+1 jump = %ld bytes\n",
           sizeof(char),  (long)(cp+1) - (long)cp);
    printf("sizeof short = %zu  |  sp+1 jump = %ld bytes\n",
           sizeof(short), (long)(sp+1) - (long)sp);
    printf("sizeof int   = %zu  |  ip+1 jump = %ld bytes\n",
           sizeof(int),   (long)(ip+1) - (long)ip);
    printf("sizeof double= %zu  |  dp+1 jump = %ld bytes\n",
           sizeof(double),(long)(dp+1) - (long)dp);

    return 0;
}
sizeof char = 1 | cp+1 jump = 1 bytes
sizeof short = 2 | sp+1 jump = 2 bytes
sizeof int = 4 | ip+1 jump = 4 bytes
sizeof double= 8 | dp+1 jump = 8 bytes

The jump is always exactly sizeof(type). The compiler guarantees this. It's not magic — the CPU instruction uses a scaled index under the hood. But from your perspective as the programmer, pointer arithmetic always moves in units of the pointed-to type, never in bytes.

ptr starts at arr[0]. Each step advances the pointer by sizeof(int) = 4 bytes, landing on the next element. The yellow highlight shows the current element. Note the hex addresses in the bottom row — they increment by 4 each time. arr[i] and *(arr+i) are identical — the compiler generates the exact same machine instruction for both.

The full pointer program

#include <stdio.h>

int main() {
    int arr[6] = {10, 20, 30, 40, 50, 60};
    int *ptr   = arr;  // ptr = address of arr[0]

    printf("arr name  = %p\n", (void*)arr);    // address of first element
    printf("&arr[0]   = %p\n", (void*)&arr[0]);// same thing
    printf("ptr       = %p\n", (void*)ptr);    // also the same

    printf("\nWalking with ptr+i:\n");
    for (int i = 0; i < 6; i++) {
        printf("  ptr+%d  addr=%-14p  *(ptr+%d)=%-4d  arr[%d]=%d\n",
               i, (void*)(ptr+i), i, *(ptr+i), i, arr[i]);
    }
    // *(ptr+i) and arr[i] always give the same result
    // because arr[i] IS defined as *(arr+i) — they are literally
    // the same expression, just written differently

    // You can also do SUBTRACTION to get distance between elements
    int *first = &arr[0];
    int *last  = &arr[5];
    printf("\nDistance from arr[0] to arr[5]:\n");
    printf("  in elements: %ld\n",  last - first);  // = 5
    printf("  in bytes:    %ld\n",  (long)last - (long)first);  // = 20

    return 0;
}

Pointer arithmetic and array indexing are the same thing

The C standard defines arr[i] as exactly *(arr + i). Not "equivalent to" — literally defined as. The compiler never sees arr[i] as a special operation. It sees a pointer, adds i * sizeof(int) to it, then dereferences the result. There is no runtime overhead for the [] syntax vs the *(ptr + i) syntax — they compile to the same instruction.

This has a fun consequence: arr[i] and i[arr] are both legal C and produce the same result. arr[5] == *(arr + 5) == *(5 + arr) == 5[arr]. Don't write it that way. But it reveals the truth about what indexing actually is.

CUDA parallel: Every CUDA kernel does exactly this. The standard kernel pattern is: int idx = blockIdx.x * blockDim.x + threadIdx.x; if (idx < N) output[idx] = input[idx] * 2.0f;

That output[idx] is *(output + idx), which is *(output + idx * sizeof(float)) in bytes. Thread 0 gets element 0, thread 1 gets element 1, etc. Each thread accesses its element by pointer arithmetic — which is exactly what we just learned.

Program 3 — The 5 Memory Segments

The full picture of process memory

Stack and heap are two of five memory regions every running C program has. The other three explain behaviour that otherwise looks inexplicable — like why global variables are automatically zero, or why you get a segfault when you accidentally write to a string literal, or why two different processes running the same binary share the same code in physical RAM.

The five segments arranged from low address (bottom) to high address (top) in virtual memory. They appear in order: TEXT (read-only code), DATA (initialised globals), BSS (zeroed globals), HEAP (grows up), STACK (grows down). Each segment has completely different lifetime and access rules. 

#include <stdio.h>
#include <stdlib.h>

// ── DATA segment ─────────────────────────────────────────────
// Initialised globals. Their VALUES are stored in the binary file
// on disk. When the OS loads the program, it maps this data into
// the process's DATA segment. Size is fixed at compile time.
int   global_init_int   = 100;
float global_init_float = 3.14f;
char  global_init_char  = 'X';

// ── BSS segment ──────────────────────────────────────────────
// UNinitialised globals (or those explicitly set to 0).
// These are NOT stored in the binary — only their sizes are.
// When the program starts, the OS zeroes this entire segment.
// This is why global variables are always 0 by default.
int   global_uninit_int;    // = 0 automatically
float global_uninit_float;  // = 0.0 automatically

// ── TEXT segment ─────────────────────────────────────────────
// Your compiled machine code. Read-only. If two processes run
// the same binary, the OS maps the same physical pages into
// both — only one copy of the code exists in RAM.
int helper(int x) { return x * 2; }

int main() {
    // ── STACK ──
    int   local_int   = 42;
    float local_float = 1.23f;

    // ── HEAP ──
    int *heap_ptr = (int *)malloc(4 * sizeof(int));
    heap_ptr[0] = 9; heap_ptr[1] = 8;
    heap_ptr[2] = 7; heap_ptr[3] = 6;

    // Print addresses for all 5 segments
    printf("SEGMENT      VARIABLE              ADDRESS\n");
    printf("─────────────────────────────────────────────\n");
    printf("TEXT         main() function      %p\n", (void*)main);
    printf("TEXT         helper() function    %p\n", (void*)helper);
    printf("DATA         global_init_int      %p  = %d\n",
           (void*)&global_init_int, global_init_int);
    printf("DATA         global_init_float    %p  = %.2f\n",
           (void*)&global_init_float, global_init_float);
    printf("BSS          global_uninit_int    %p  = %d\n",
           (void*)&global_uninit_int, global_uninit_int);
    printf("BSS          global_uninit_float  %p  = %.1f\n",
           (void*)&global_uninit_float, global_uninit_float);
    printf("HEAP         heap_ptr[0]          %p  = %d\n",
           (void*)&heap_ptr[0], heap_ptr[0]);
    printf("STACK        local_int            %p  = %d\n",
           (void*)&local_int, local_int);

    int unused = helper(5);  // keep helper in TEXT
    (void)unused;

    free(heap_ptr);
    return 0;
}

─────────────────────────────────────────────
TEXT main() 0x55a3f1201189
TEXT helper() 0x55a3f1201175
DATA global_init_int 0x55a3f1204010 = 100
DATA global_init_float 0x55a3f1204014 = 3.14
BSS global_uninit_int 0x55a3f1204020 = 0
BSS global_uninit_float 0x55a3f1204024 = 0.0
HEAP heap_ptr[0] 0x55a3f52392b0 = 9
STACK local_int 0x7ffcd3a21e3c = 42

Each segment explained in detail

1. TEXT (code segment) — Your compiled machine instructions. When you call main(), the CPU jumps to this address and starts executing. This segment is mapped read-only: any attempt to write to it causes an immediate segfault. That's intentional — it prevents code injection attacks and ensures that the instructions can't accidentally overwrite themselves. On modern systems, this segment is also marked executable; other segments are not.

2. DATA segment — Global and static variables that have explicit initial values. These are physically stored in the binary file on disk. When you open any compiled ELF binary with a hex editor, you'll find the actual bytes of your initialised globals in there. The OS reads them off disk and maps them into the process's DATA segment at startup. If you have a 1 MB initialised global array, your binary is 1 MB larger.

3. BSS segment — Global and static variables that either have no initialiser or are explicitly zero. These are not stored in the binary — only the total size is recorded. The OS sets aside that much RAM and zeroes it before the program starts. This is the reason int global; (no initialiser) reliably starts at 0 — the OS guarantees it. BSS also explains why a program with a large uninitialised global array doesn't have a proportionally large binary file.

4. HEAP — Runtime-allocated memory. Starts at a base address just above the BSS segment and grows upward. Managed by the C runtime's allocator (malloc/free). The allocator itself requests pages from the OS using the brk() or mmap() syscall and then subdivides them for individual allocations.

5. STACK — Automatic storage for local variables and function call mechanics. Starts near the top of virtual memory and grows downward. Each thread gets its own stack. The main thread's stack is typically 8 MB on Linux; you can check with ulimit -s.

String literals live in TEXT, not DATA. When you write char *s = "hello";, the string "hello" is in the read-only TEXT segment. The pointer s is on the stack. Trying to modify through s — like s[0] = 'H'; — is a segfault. To get a modifiable string, write char s[] = "hello"; which copies the literal onto the stack as a char array you own.

Program 4 — Dynamic 2D Array: The Fragmentation Trap

What people expect vs what actually happens

Ask a beginner what a 3×4 dynamically allocated matrix looks like in memory and they'll draw a grid: 12 integers, all in a row, nicely contiguous. That's what a static 2D array (int mat[3][4]) looks like. But a dynamic 2D array — allocated with malloc — looks completely different.

What you actually get is an array of pointers, and each pointer points to a separately allocated row somewhere else on the heap. The rows can be hundreds or thousands of bytes apart. There is no guarantee of contiguity between rows.

Left: the pointer array — mat[0], mat[1], mat[2] store addresses. Right: the actual row data, each at a separate heap location. The rows are NOT next to each other in memory. This is heap fragmentation — each malloc can land anywhere in the available heap space. 

#include <stdio.h>
#include <stdlib.h>

int main() {
    int rows = 3, cols = 4;

    // ── Step 1: allocate the pointer array ───────────────────
    // This gives you 'rows' pointers, each 8 bytes on 64-bit.
    // Total: 3 * 8 = 24 bytes on the heap.
    int **mat = (int **)malloc(rows * sizeof(int *));
    if (!mat) return 1;

    // ── Step 2: allocate each row separately ─────────────────
    // Each call to malloc() can return any free address on the heap.
    // There is NO guarantee that mat[0], mat[1], mat[2]
    // are adjacent.
    for (int i = 0; i < rows; i++) {
        mat[i] = (int *)malloc(cols * sizeof(int));
        if (!mat[i]) return 1;
        printf("Row %d allocated at: %p\n", i, (void*)mat[i]);
    }

    // ── Fill the matrix ───────────────────────────────────────
    for (int i = 0; i < rows; i++)
        for (int j = 0; j < cols; j++)
            mat[i][j] = i * cols + j + 1;

    // ── Print with addresses ──────────────────────────────────
    printf("\nMatrix contents and element addresses:\n");
    for (int i = 0; i < rows; i++) {
        printf("  Row %d @ %p  | ", i, (void*)mat[i]);
        for (int j = 0; j < cols; j++)
            printf("[%d]@%p  ", mat[i][j], (void*)&mat[i][j]);
        printf("\n");
    }

    // ── Show the gap between rows ─────────────────────────────
    printf("\nGaps between rows:\n");
    for (int i = 0; i < rows - 1; i++) {
        long gap = (long)mat[i+1] - (long)mat[i];
        printf("  Row %d to Row %d: %ld bytes apart\n", i, i+1, gap);
    }

    // ── Step 3: FREE in the right order ──────────────────────
    // MUST free each row first, THEN the pointer array.
    // If you free mat first, you lose all the row pointers
    // and can never reach the row data to free it.
    for (int i = 0; i < rows; i++)
        free(mat[i]);   // free each row
    free(mat);          // then free the pointer array
    return 0;
}

Typical output makes the fragmentation obvious:

Row 0 allocated at: 0x5631d4a392b0
Row 1 allocated at: 0x5631d4a392e0
Row 2 allocated at: 0x5631d4a39310

Gaps between rows:
  Row 0 to Row 1: 48 bytes apart
  Row 1 to Row 2: 48 bytes apart
(in this case adjacent because heap was empty — in a real program with mixed allocations, these gaps can be hundreds of bytes or more)

Why this matters for performance

Modern CPUs are fast because of caches. When you access mat[0][3], the CPU doesn't just fetch 4 bytes — it fetches an entire 64-byte cache line. If the next thing you access is mat[1][0], with a contiguous layout that's right there in the cache. With a pointer-of-pointers layout, it's at a completely different address — the cache line you just loaded is useless, and you pay for another memory fetch.

For small matrices this is irrelevant. For large matrices in hot inner loops — like the kind you'd write to multiply two 1024×1024 matrices — pointer chasing can cut your throughput by 5–10x compared to a flat contiguous layout.

The flat 1D layout: the better alternative

// Instead of pointer-of-pointers, allocate one flat block:
int *mat_flat = (int *)malloc(rows * cols * sizeof(int));

// Access element [i][j] as:
mat_flat[i * cols + j] = value;

// This is:
//   - A single malloc (one heap allocation, no fragmentation)
//   - Perfectly contiguous (cache-friendly)
//   - One free at the end
free(mat_flat);
This is exactly why every CUDA matrix example uses the flat layout. cudaMalloc(&d_mat, rows * cols * sizeof(float)) gives you one contiguous block on the GPU. Access it as d_mat[row * cols + col]. The GPU's memory subsystem is optimised for coalesced access — consecutive threads reading consecutive addresses — and the flat layout is the only way to get that. A pointer-of-pointers matrix on the GPU is essentially unusable for parallel kernels.
