CUDA Journey #1: Building a CPU Baseline Before You GPU

Before you write a single line of CUDA, you need something to beat. Here's how to build a proper CPU baseline — summing 1 million floats, timing it with clock(), and understanding exactly what's happening under the hood at every line.

Every CUDA tutorial I've seen skips straight to the GPU kernel. Launch some threads, run the reduction, print the speedup number — impressive. But speedup over what? If your baseline is bad C code, your CUDA looks like a miracle. If your baseline is reasonable, the CUDA win is still real, but honest.

This post is about the baseline. We're going to write a C program that allocates 1 million floats, fills them with random values, sums them with a for loop, and times the whole thing with clock(). That's it. But we're going to understand every single line — the memory model, the timing API, the randomness, the float arithmetic, all of it. Because when you later parallelize this on the GPU, you need to know what you're comparing against and why.

The Full Program

Here's the complete code. We'll dissect it piece by piece below.

#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define N 1000000

int main() {
    // Step 1: Allocate memory on the heap
    float *arr = (float *)malloc(N * sizeof(float));
    if (arr == NULL) {
        fprintf(stderr, "Memory allocation failed\n");
        return 1;
    }

    // Step 2: Seed the random number generator
    srand(42);

    // Step 3: Fill the array with random float values
    for (int i = 0; i < N; i++) {
        arr[i] = (float)rand() / RAND_MAX;
    }

    // Step 4: Start the timer
    clock_t start = clock();

    // Step 5: Compute the sum
    float sum = 0.0f;
    for (int i = 0; i < N; i++) {
        sum += arr[i];
    }

    // Step 6: Stop the timer
    clock_t end = clock();

    // Step 7: Compute elapsed time in milliseconds
    double elapsed_ms = (double)(end - start) / CLOCKS_PER_SEC * 1000.0;

    // Step 8: Print results
    printf("Sum      = %.2f\n", sum);
    printf("Time     = %.3f ms\n", elapsed_ms);

    // Step 9: Free memory
    free(arr);

    return 0;
}

Compile and run it like this:

gcc -O2 -o cpu_sum cpu_sum.c
./cpu_sum

You'll see something like:

Sum      = 500042.78
Time     = 1.243 ms

That's your baseline. Now let's understand every decision that went into it.

The Headers: What You're Pulling In

#include <stdio.h>
#include <stdlib.h>
#include <time.h>

These three headers give you access to the C standard library functions used in this program. They're not optional — without them, the compiler doesn't know what malloc, rand, or clock are.

  • stdio.h — Standard Input/Output. This gives you printf and fprintf. printf writes to stdout (your terminal). fprintf(stderr, ...) writes to stderr — the error stream — which is useful because stdout and stderr can be redirected independently in a shell. If you redirect your program's output to a file (./cpu_sum > out.txt), error messages still appear in the terminal because they go to stderr, not stdout.
  • stdlib.h — Standard Library. Gives you memory management (malloc, free) and random number generation (rand, srand). This header is the gateway to heap memory in C — without it, you're stuck on the stack.
  • time.h — Time utilities. Gives you clock() and the clock_t type and the CLOCKS_PER_SEC constant. These are the primitives we'll use to measure execution time.

The #define: Your Array Size

#define N 1000000

This is a preprocessor macro. Before the compiler even sees your code, the C preprocessor does a text substitution — every occurrence of N in your source file is replaced with 1000000. The compiler never sees the name N, only the literal number.

Why a macro instead of const int N = 1000000;? In this case, either works. But macros are zero-cost — there's no variable, no memory address, no potential for the value to be modified at runtime. It's a compile-time constant in the truest sense.

One million floats. At 4 bytes each, that's exactly 4 MB of data. This is a meaningful size: large enough to see real timing differences, small enough to fit comfortably in RAM and even in L3 cache on modern CPUs (which are typically 8–32 MB). When you move to CUDA, you'll feel the PCIe transfer cost of these 4 MB going to the GPU.

Step 1: Heap Allocation with malloc

float *arr = (float *)malloc(N * sizeof(float));
if (arr == NULL) {
    fprintf(stderr, "Memory allocation failed\n");
    return 1;
}

This is the first and most important decision in the program: putting the array on the heap rather than the stack.

The stack is fast but small (typically 1–8 MB). The heap is large but requires explicit management. Four megabytes on the stack would likely cause a stack overflow.

Let's break down the malloc call:

  • N * sizeof(float) — We need N floats, and each float is sizeof(float) bytes. On every modern platform, sizeof(float) is 4 (IEEE 754 single-precision). So we're requesting 4,000,000 bytes = 4 MB. Always use sizeof(type) rather than hardcoding a number — it makes your code portable and self-documenting.
  • (float *) cast — malloc returns void *, a generic pointer with no type information. We cast it to float * so the compiler knows each element is a float and can do correct pointer arithmetic. In C++ this cast is mandatory; in C it's technically optional but good practice.
  • NULL check — malloc returns NULL if the allocation fails (out of memory, or you asked for a nonsensical size). Always check. Dereferencing a NULL pointer is undefined behavior — on most systems it immediately crashes with a segfault, but you can't rely on that. The check costs nothing and saves you from mysterious failures.

After malloc, arr is a pointer to the beginning of a contiguous block of 4 MB in heap memory. The memory is uninitialized — it contains whatever bytes happened to be there. That's why we fill it next.

Step 2: Seeding the Random Number Generator

srand(42);

C's rand() function is a pseudorandom number generator (PRNG) — it produces a deterministic sequence of integers that looks random. The sequence is determined entirely by the initial seed value.

srand(42) sets the seed to 42. Every time you run the program with this seed, you get the exact same sequence of random numbers. This is intentional for a baseline: reproducibility. When you later run this on the GPU, you want to be summing the same numbers so you can verify correctness by comparing the sums.

In production code where you want different random data each run, you'd seed with srand(time(NULL)) — using the current time as the seed so it differs between runs. For benchmarking, a fixed seed is better.

The quality of C's rand() is not great — it's not suitable for cryptography or serious statistical work. For scientific computing, you'd use something like the Mersenne Twister. But for filling a benchmark array, it's perfectly fine.

Step 3: Filling the Array with Random Floats

for (int i = 0; i < N; i++) {
    arr[i] = (float)rand() / RAND_MAX;
}

This loop runs one million times, setting each element to a float in the range [0.0, 1.0]. Let's unpack it:

  • rand() — returns a pseudorandom integer in the range [0, RAND_MAX]. RAND_MAX is at least 32767, and on most modern Linux systems it's 2,147,483,647 (2^31 - 1, i.e., INT_MAX).
  • (float)rand() — cast the integer to float before dividing. If you wrote rand() / RAND_MAX without the cast, you'd get integer division — the result would always be 0 (since rand() < RAND_MAX). The cast forces floating-point division.
  • / RAND_MAX — divides by the maximum possible value, normalizing the result to [0.0, 1.0]. When rand() returns 0, you get 0.0. When it returns RAND_MAX, you get exactly 1.0.
  • arr[i] = ... — array subscript notation. This is equivalent to *(arr + i) = .... The compiler computes the address as arr (base pointer) + i * sizeof(float) bytes, then writes the float value there.

With 1 million values uniformly distributed in [0, 1], the expected sum is 500,000. The actual sum will be close — typically within a few hundred of 500,000. You can see this in the output: Sum = 500042.78.

Note that this initialization loop is not timed. We only want to measure the summation, not the data setup. In real CUDA benchmarks, you'll similarly separate data preparation from the kernel you're actually measuring.

Step 4 & 6: The Timer — clock()

clock_t start = clock();
// ... work ...
clock_t end = clock();

clock() returns the amount of processor time used by the program since it started, measured in "clock ticks". The type clock_t is typically a long or long long.

Key things to understand about clock():

  • It measures CPU time, not wall time. If your process sleeps or waits for I/O, that time is not counted. For pure compute (which this is), CPU time ≈ wall time, so this is fine.
  • Resolution varies. CLOCKS_PER_SEC is 1,000,000 on Linux (microsecond resolution). On Windows it's only 1,000. For very fast operations, this matters.
  • It can overflow. On 32-bit systems with clock_t as a 32-bit type, it wraps after about 72 minutes. For a sub-millisecond benchmark this is irrelevant, but worth knowing.
  • It's not the most precise option. For very fine-grained benchmarking in production, you'd use clock_gettime(CLOCK_MONOTONIC, ...) on Linux, which gives nanosecond resolution and doesn't have the CPU-vs-wall-time ambiguity. But for our purposes, clock() is clear and portable.

We take the timestamp immediately before the summation loop and immediately after. Nothing else happens between them. This is deliberate: you want to measure only the work you care about.

Step 5: The Summation Loop

float sum = 0.0f;
for (int i = 0; i < N; i++) {
    sum += arr[i];
}

The simplest possible reduction: a sequential accumulation. Let's think carefully about what's happening.

The accumulator: float sum = 0.0f

We use float (not double) to match the array element type. The f suffix on 0.0f makes it a float literal — without it, 0.0 is a double, and the assignment would involve an implicit narrowing conversion. It's a minor detail but good practice.

There's a subtle numerical issue here. Adding 1 million floats sequentially accumulates floating-point rounding errors. Each += operation rounds the result to the nearest representable float. After a million such operations, the accumulated error can be significant — potentially hundreds of ULPs (Units in the Last Place) off from the true sum. If you needed high-precision summation, you'd use Kahan summation or switch to double. For our baseline purposes, the naive approach is fine and matches what a naive GPU kernel would do.

The memory access pattern

The loop accesses memory sequentially: arr[0], arr[1], arr[2], ... This is the best possible pattern for a CPU cache. The CPU's hardware prefetcher will detect the stride-1 access pattern immediately and start loading cache lines ahead of the loop. With a 64-byte cache line holding 16 floats, the CPU fetches data in bursts of 16 elements. The entire 4 MB array fits in L3 cache on most modern CPUs, so after the first pass, repeat runs would be even faster (served from cache, not DRAM).

Sequential memory access + cache-resident data = this loop is essentially running at memory bandwidth speed. The compute (one addition per element) is trivial; the bottleneck is feeding data to the CPU fast enough.

Loop overhead

With -O2 optimization, the compiler will vectorize this loop automatically. It'll use SIMD instructions (SSE/AVX on x86) to add 4, 8, or even 16 floats per clock cycle rather than one. The -O2 flag in our compile command is important — without it, the loop runs scalar and is significantly slower. Always benchmark with optimizations enabled, because your GPU will be competing against optimized CPU code, not naive scalar loops.

Step 7: Converting Clock Ticks to Milliseconds

double elapsed_ms = (double)(end - start) / CLOCKS_PER_SEC * 1000.0;

Let's walk through the arithmetic:

  • end - start — the number of clock ticks elapsed. Since both are clock_t, this subtraction gives a clock_t value representing elapsed ticks.
  • (double)(...) — cast to double before dividing. clock_t is an integer type. If you wrote (end - start) / CLOCKS_PER_SEC without casting, you'd get integer division and lose all sub-second precision. The cast ensures floating-point division.
  • / CLOCKS_PER_SEC — converts ticks to seconds. On Linux, CLOCKS_PER_SEC is 1,000,000, so if you observed 1,243 ticks, you get 1243 / 1000000 = 0.001243 seconds.
  • * 1000.0 — converts seconds to milliseconds. 0.001243 * 1000 = 1.243 ms.

We store the result as double (not float) because timing values need enough precision to represent fractional milliseconds accurately. A float has about 7 significant decimal digits; a double has about 15. For a 1.243 ms measurement, float is fine, but for a 0.001 ms measurement (which you'll see with GPU kernels), you want the extra precision.

Step 8: Printing the Results

printf("Sum      = %.2f\n", sum);
printf("Time     = %.3f ms\n", elapsed_ms);

printf format specifiers:

  • %.2f — print a floating-point number with 2 decimal places. The %f format works for both float and double (floats are promoted to double when passed to variadic functions like printf).
  • %.3f — print with 3 decimal places. This gives us microsecond-level precision in our millisecond timing: 1.243 ms.
  • \n — newline character. Also flushes the stdout buffer on most terminal configurations, so you see output immediately.

Step 9: Freeing Memory

free(arr);

Every malloc must be paired with a free. This returns the allocated memory to the heap for reuse.

In this specific program, free happens right before the process exits, and the OS will reclaim all memory anyway when the process terminates. So technically, this free does nothing observable. But you should always write it anyway, for two reasons:

  • Habit. In larger programs, forgetting to free memory causes leaks. The habit of always pairing malloc with free prevents this class of bugs.
  • Tools. Memory checkers like Valgrind will flag the missing free as a leak. Keeping your code clean makes these tools more useful for finding real leaks.

When you move to CUDA, you'll use cudaMalloc and cudaFree for GPU memory — the same pattern, different functions.

What the Numbers Tell You

On a typical modern CPU (Intel Core i7, AMD Ryzen 7), summing 1 million floats takes roughly 0.5–3 ms depending on:

  • Whether the data is hot in cache (first run vs. repeated runs)
  • Compiler optimization level (-O0 vs -O2 vs -O3)
  • SIMD vectorization (AVX-512 vs AVX2 vs SSE4)
  • CPU frequency scaling (power management)

The theoretical peak: 4 MB of data divided by memory bandwidth. If your CPU has 40 GB/s memory bandwidth, the minimum possible time is 4 MB / 40 GB/s = 0.1 ms. The actual time is higher because of loop overhead and the accumulator dependency chain (each addition depends on the previous result, limiting instruction-level parallelism).

The CPU is bottlenecked on memory bandwidth, not compute. This is exactly why GPU parallelism wins here — massively higher memory bandwidth, not just more cores.

What Changes When You Go to CUDA

When we rewrite this for the GPU (the next post), every single component has a parallel:

  • malloc → cudaMalloc (allocate GPU memory)
  • srand + rand loop → cuRAND or CPU-generated data copied with cudaMemcpy
  • for loop → parallel reduction kernel with thousands of threads
  • clock() → cudaEvent_t for GPU-side timing
  • free → cudaFree

And there's a new cost that doesn't exist in the CPU version: copying the data from host (CPU) RAM to device (GPU) VRAM over the PCIe bus. At ~16 GB/s PCIe bandwidth, transferring 4 MB takes ~0.25 ms. Whether the GPU's faster compute can overcome this transfer overhead is the core question of GPU programming — and why having an honest CPU baseline matters.

Run It Yourself: Compile Flags Matter

Try these variants to see the impact of optimization:

# No optimization (scalar, no vectorization)
gcc -O0 -o cpu_sum_O0 cpu_sum.c && ./cpu_sum_O0

# Standard optimization (auto-vectorization, common baseline)
gcc -O2 -o cpu_sum_O2 cpu_sum.c && ./cpu_sum_O2

# Aggressive + native SIMD (use CPU's best vector instructions)
gcc -O3 -march=native -o cpu_sum_O3 cpu_sum.c && ./cpu_sum_O3

You'll likely see -O3 be 2–5x faster than -O0 for this workload. This is a good reminder: when you benchmark GPU speedup, always compare against the best CPU you can run, not the worst.

Takeaways

  • Measure before you optimize. This CPU baseline is the reference point. Without it, your GPU speedup numbers are meaningless.
  • Heap allocation with malloc is mandatory for large arrays. 4 MB on the stack will crash. Always check the return value for NULL.
  • Seed your PRNG for reproducibility. Fixed seed = same data every run = comparable results across CPU and GPU implementations.
  • clock() measures CPU time, not wall time. For pure compute without I/O or sleeping, this is what you want. Cast to double before converting.
  • Compile with -O2 at minimum. The CPU will auto-vectorize the sum loop. Your baseline should be competitive, not deliberately handicapped.
  • This loop is memory-bandwidth bound, not compute-bound. The GPU wins here because of bandwidth, not just core count.
