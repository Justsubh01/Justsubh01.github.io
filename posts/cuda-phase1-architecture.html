1.2 — CPU vs GPU Architecture: Why Parallelism Changes Everything

CPUs and GPUs are built to solve completely different problems. Understanding their architectural philosophies — latency vs throughput, cache hierarchy vs massive threading, SIMD vs SIMT — is not background knowledge. It directly determines every performance decision you will make in CUDA.

Most programmers spend years writing CPU code and never question the assumption that programs execute one instruction at a time, in order, as fast as possible. Then they sit down with CUDA and discover that a "fast" GPU kernel runs slower than the CPU version, or that adding more compute instructions does not improve speed, or that a trivially written kernel hides 80% of available bandwidth. The root cause of all of this confusion is architectural. CPUs and GPUs were designed under fundamentally different constraints, for fundamentally different workloads. Once you internalize those constraints, every CUDA optimization decision becomes obvious.

1. The Fundamental Design Philosophy

The core difference between a CPU and a GPU can be stated in one sentence each:

  • CPU: Minimize the time it takes to complete one task.
  • GPU: Maximize the number of tasks completed per second.

These are different objectives, and they lead to radically different hardware designs. A CPU is latency-optimized — it will use an enormous amount of silicon and power to ensure that a single thread of execution experiences as little waiting as possible. A GPU is throughput-optimized — it will accept high latency per individual thread, as long as it can run enough threads in parallel to hide that latency and keep all its execution units busy.

The key mental model shift:
On a CPU, you think about making your thread fast. Minimize its cache misses. Minimize its branch mispredictions. Minimize its instruction count.

On a GPU, you think about occupancy — how many threads are active simultaneously on each SM. You do not optimize for one thread; you optimize for the collective throughput of thousands of threads. A single thread on a GPU is actually slower than on a CPU (lower clock, simpler execution). The GPU wins only when you give it enough parallel work to keep all its execution units occupied simultaneously.

The cache hierarchy exists entirely to bridge the gap between CPU speed (5 GHz, completing an operation every 0.2 ns) and DRAM speed (200 ns latency). An L3 miss means the CPU stalls for 40–100 cycles doing nothing. All that out-of-order execution machinery — the reorder buffer, the scheduler, the register renaming — exists to find other work to do during those stall cycles.

Branch prediction: speculative execution

Modern CPUs predict which way an if/else branch will go before evaluating the condition. They speculatively execute the predicted path. If the prediction is wrong (a misprediction), they flush the pipeline and pay a 15–20 cycle penalty. Well-predicted branches (regular patterns) are effectively free. Mispredicted branches are expensive. 

/* CPU-friendly: predictable branch (always true for sorted data) */
for (int i = 0; i < N; i++) {
    if (data[i] > 128) {   /* branch predictor learns the pattern */
        sum += data[i];
    }
}

/* CPU-unfriendly: random/unpredictable branches */
for (int i = 0; i < N; i++) {
    if (random_bits[i]) {   /* ~50% misprediction rate */
        sum += data[i];
    }
}
Why CPUs "waste" so much die area on control logic:
On a modern Intel CPU, roughly 40–50% of the die area is devoted to caches, prefetchers, branch predictors, reorder buffers, and other latency-hiding structures. Only a small fraction is actual arithmetic compute (ALUs, FPUs). This is a deliberate engineering choice: the cache and branch predictor make a single thread fast. Without them, one L3 miss would stall the CPU for 40 cycles with nothing to show for it.

The GPU makes the opposite choice: use that die area for more execution units instead. Accept that individual threads will stall (high latency), but run so many threads that there is always another warp ready to execute, hiding the latency through massive multithreading rather than hardware prediction.

3. GPU Architecture Deep Dive

The GPU is not simply "a CPU with more cores." It is an entirely different architecture built around a different execution model. The fundamental unit of GPU computation is the Streaming Multiprocessor (SM).

The Streaming Multiprocessor (SM)

An SM is the GPU's equivalent of a CPU core, but designed for throughput, not latency. Where a CPU core contains deep out-of-order execution machinery, an SM contains many simpler execution units and the ability to switch between dozens of warps (groups of 32 threads) every clock cycle.

How the GPU hides memory latency: warp switching

Global memory (HBM) latency on the A100 is approximately 600–800 cycles. A single thread waiting for a global memory load would waste 700 cycles doing nothing. The GPU's solution is zero-overhead context switching between warps: 

  Time ─────────────────────────────────────────────────────────────────►

  Warp  0: [EXEC]─────────────[MEM WAIT 700 cycles]──────────[EXEC]─────
  Warp  1:       [EXEC]──[MEM WAIT]────────────────[EXEC]───────────────
  Warp  2:             [EXEC]──────────────[MEM WAIT]──────────[EXEC]───
  ...
  Warp 63:                                   [EXEC][MEM WAIT][EXEC]──────
           │    │
           Each cycle, the warp scheduler looks at all warps.
           It picks any warp whose next instruction's operands are ready.
           If warp 0 is waiting for memory, it instantly switches to warp 1.
           No save/restore of state needed: all 64 warps' registers are
           ALWAYS live in the register file simultaneously.

  SM execution units are busy every cycle as long as enough warps are active.
  This is called OCCUPANCY: the fraction of max warps that are active.
  Higher occupancy → better latency hiding → higher throughput.
The SM register file is large for a reason:
The A100's register file per SM is 256 KB (65,536 × 32-bit registers). For comparison, a CPU core has perhaps 16–32 architectural registers (a few hundred bytes). The GPU's enormous register file exists specifically to support zero-overhead warp switching: all 64 warps' registers must be resident simultaneously, because switching between them requires no save/restore. Each warp of 32 threads gets a slice of that register file. If each thread uses 32 registers, 32 threads × 32 registers = 1,024 registers per warp, and 64 warps × 1,024 = 65,536 registers total — exactly the register file size. Using more registers per thread reduces the number of simultaneous warps the SM can support, which reduces occupancy and hurts latency hiding. This is the register pressure vs occupancy tradeoff at the heart of GPU optimization.

4. Memory Bandwidth — The Real Bottleneck

Ask most people what makes the GPU fast and they will say "lots of cores" or "high clock speed." The honest answer is memory bandwidth. The A100 has 2,039 GB/s of HBM2e bandwidth. A high-end CPU has ~200 GB/s. This 10× bandwidth advantage is the single biggest reason GPUs dominate data-parallel workloads: they can feed their many execution units with data 10× faster than a CPU can feed its few.

Why HBM is so much faster than DDR

  DDR5 (CPU):
  ─────────────────────────────────────────────────────────────
  Bus width: 64 bits × 2 channels = 128-bit effective bus
  Clock: ~4800 MT/s (DDR = Double Data Rate, so 2400 MHz)
  Bandwidth: 128 bits × 4800 MT/s / 8 = 76.8 GB/s per channel
  Dual-channel: ~153 GB/s  (quad-channel workstation: ~307 GB/s)

  Physical: separate package, connected via traces on motherboard
  Latency: ~80 ns (long trace = slow signal propagation)
  ─────────────────────────────────────────────────────────────

  HBM2e (A100):
  ─────────────────────────────────────────────────────────────
  Bus width: 1024 bits per stack × 5 stacks = 5,120-bit total bus
  Clock: ~1215 MHz (HBM = High Bandwidth Memory)
  Bandwidth: 5120 bits × 1215 MT/s / 8 ≈ 2,039 GB/s

  Physical: stacked directly on or adjacent to GPU die (interposer)
  Latency: ~60 ns (shorter path, but still slow vs on-chip)
  ─────────────────────────────────────────────────────────────

  Result: 5120-bit bus vs 128-bit bus = 40× more bandwidth lanes
          Plus higher clock on HBM = ~10-13× total bandwidth advantage

Most CUDA kernels are memory-bound, not compute-bound. This surprises people.

Your GPU has 19.5 TFLOPS of FP32 compute and only 2 TB/s of memory bandwidth. The ridge point is ~9.6 FLOPs/byte. Any kernel doing fewer FLOPs per byte than this is bottlenecked by memory bandwidth, not compute. That includes vector addition, element-wise operations, most reductions, embedding lookups, and many attention operations (without tiling).

Implication: adding more arithmetic instructions to a memory-bound kernel makes it slower, not faster. The fix is to reduce bytes read (use smaller dtypes, fuse operations to reuse data in registers), not to reduce FLOPs. Profile with ncu --set full and look at Memory Throughput % vs Compute Throughput % to determine which you are hitting.

5. SIMD vs SIMT

Both CPUs and GPUs exploit data parallelism. But they do it in fundamentally different ways that have important implications for how you write code.

SIMD: Single Instruction, Multiple Data (CPU)

CPU SIMD (SSE, AVX, AVX-512) extends the ALU with wide registers. One instruction operates on 4, 8, or 16 data elements simultaneously, packed into a single register. All elements must perform exactly the same operation — there is no masking per-element within a single SIMD instruction (though AVX-512 adds per-element masking as an exception). 

#include <immintrin.h>   /* AVX2 intrinsics */

/* Scalar: adds 8 floats one at a time */
void scalar_add(float *a, float *b, float *c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];    /* 1 float per operation */
    }
}

/* AVX2 SIMD: adds 8 floats with ONE instruction */
void simd_add(float *a, float *b, float *c, int n) {
    for (int i = 0; i < n; i += 8) {
        __m256 va = _mm256_loadu_ps(&a[i]);    /* load 8 floats into 256-bit register */
        __m256 vb = _mm256_loadu_ps(&b[i]);    /* load 8 floats */
        __m256 vc = _mm256_add_ps(va, vb);     /* add 8 floats: ONE instruction */
        _mm256_storeu_ps(&c[i], vc);           /* store 8 floats */
    }
}
/* AVX-512: 512 bits / 32 bits per float = 16 floats per instruction */

/* The programmer explicitly manages the vector width.
   The CPU executes ALL 8 (or 16) lanes simultaneously.
   Branches within a SIMD vector are handled via masking (AVX-512)
   or by computing both paths and blending results (AVX2). */

SIMT: Single Instruction, Multiple Threads (GPU)

GPU SIMT hides the vector nature from the programmer entirely. You write scalar code — each thread processes one element. The hardware groups 32 threads into a warp and executes them in lockstep: all 32 threads execute the same instruction in the same clock cycle, each on their own register values. 

/* CUDA SIMT: programmer writes scalar code for ONE thread */
__global__ void cuda_add(const float * __restrict__ a,
                         const float * __restrict__ b,
                               float * __restrict__ c,
                         int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        c[i] = a[i] + b[i];    /* scalar code: one float */
    }
}
/* The hardware runs 32 threads simultaneously.
   Threads 0-31 all execute "c[i] = a[i] + b[i]" at the same time,
   each with their own i (0, 1, 2, ..., 31), their own a[i], b[i], c[i].
   This is EXACTLY what AVX-512's 16-wide add does, but:
   - Programmer writes scalar code (no intrinsics)
   - Vector width is 32 (not 8 or 16)
   - The abstraction is threads, not vector lanes */

The key difference: divergence handling

  SIMD (CPU AVX2):
  ─────────────────────────────────────────────────────────────────────
  Branches within a vector handled by:
  1. Compute both sides of the branch for all 8 lanes
  2. Use a mask register to select correct output per lane
  3. Both sides of the branch ALWAYS execute (no real divergence)

  Example (compiled by auto-vectorizer):
  if (a[i] > 0) c[i] = a[i]; else c[i] = 0.0f;
  →
  vc_pos = va;                       /* compute if-true result */
  vc_neg = _mm256_setzero_ps();      /* compute if-false result */
  mask   = _mm256_cmp_ps(va, zero);  /* which lanes are > 0? */
  vc     = _mm256_blendv_ps(vc_neg, vc_pos, mask);  /* select */

  ─────────────────────────────────────────────────────────────────────

  SIMT (GPU):
  ─────────────────────────────────────────────────────────────────────
  A warp of 32 threads hits an if/else branch.
  Some threads (e.g., threads 0-15) take the if-path.
  Other threads (e.g., threads 16-31) take the else-path.
  This is called WARP DIVERGENCE.

  The hardware handles it by SERIALIZING:
  Step 1: Run the if-path for threads 0-15, mask out threads 16-31.
  Step 2: Run the else-path for threads 16-31, mask out threads 0-15.
  Total work = 2 passes. In the worst case (50/50 split), throughput halves.

  Key insight: SIMT ALLOWS divergence (threads can take different paths)
  but PENALIZES it (divergent paths execute serially within a warp).
  SIMD cannot diverge at all; SIMT can, but at a cost.

  __global__ void branchy_kernel(float *data, int n) {
      int i = blockIdx.x * blockDim.x + threadIdx.x;
      if (i < n) {
          if (data[i] > 0.0f) {       /* some threads go if, some go else */
              data[i] = sqrtf(data[i]);   /* expensive! serialized for half warp */
          } else {
              data[i] = 0.0f;
          }
      }
  }
Warp divergence in practice:
Within a 32-thread warp, all threads taking the same branch is the ideal case. Full divergence (each thread taking a unique path) means 32 serial passes, not 1. In practice, most GPU code has structured divergence (e.g., boundary checks like if (i < n)) that causes little real overhead because it only affects the last warp in the grid. Deep divergence inside per-element logic (random branching per element) is where performance suffers.

Profile warp divergence with ncu --metrics sm__sass_branch_targets_threads_divergent. A high count indicates your kernel has significant warp divergence that may be reducible by restructuring the algorithm.

6. Amdahl's Law — When NOT to Use a GPU

Amdahl's Law is the most important theoretical result in parallel computing. It sets a hard upper bound on the speedup achievable by parallelizing any program, given the fraction of that program that must remain serial.

The formula

  Amdahl's Law:

  S(n) = 1 / (1 - p + p/n)

  Where:
    S(n) = theoretical speedup with n processors
    p    = fraction of the program that CAN be parallelized
    1-p  = fraction that MUST run serially (irreducible)
    n    = number of parallel processors (or "GPU cores")

  Examples (as n → ∞, speedup approaches 1/(1-p)):

  Serial fraction (1-p)  │  Max possible speedup (n = ∞)
  ────────────────────────┼────────────────────────────────
          1%  (p = 0.99) │  100×
          5%  (p = 0.95) │   20×
         10%  (p = 0.90) │   10×
         20%  (p = 0.80) │    5×
         25%  (p = 0.75) │    4×
         50%  (p = 0.50) │    2×
         75%  (p = 0.25) │    1.33×

  Real example: n = 1000 GPUs
  With 10% serial fraction: S(1000) = 1 / (0.1 + 0.9/1000) = 1/0.1009 ≈ 9.9×
  Even with 1000 GPUs, you can't break 10× if 10% is serial!

PCIe transfer overhead: the hidden serial fraction

One of the most common mistakes in GPU programming is ignoring the time spent transferring data between host and device. PCIe is dramatically slower than GPU memory, and every transfer is effectively serial work that Amdahl's law caps you on. 

  PCIe bandwidth:
  ────────────────────────────────────────────────────────────────
  PCIe 3.0 ×16:  ~12 GB/s unidirectional   (64-bit address, 8 GTs)
  PCIe 4.0 ×16:  ~24 GB/s unidirectional
  PCIe 5.0 ×16:  ~48 GB/s unidirectional

  For comparison:
  GPU internal (HBM2e): 2,039 GB/s   ← 85× faster than PCIe 4.0
  PCIe effective rate:    ~24 GB/s
  PCIe latency:          ~10 microseconds per transfer (fixed overhead)

  Example: 1 GB of data
  cudaMemcpy host→device: 1 GB / 24 GB/s = ~42 ms (+ 10 µs setup)
  GPU kernel processing:  ~2 ms (if kernel is fast)
  cudaMemcpy device→host: ~42 ms

  Total pipeline: 84 ms transfer + 2 ms compute
  GPU "speedup" if CPU took 20 ms: GPU is SLOWER (86 ms vs 20 ms)
  ────────────────────────────────────────────────────────────────

  Rule of thumb: GPU helps when compute time >> 2 × transfer time
  Equivalently: AI of your full pipeline must be high enough to justify PCIe cost

When GPU helps vs when GPU hurts

  USE GPU when:                          AVOID GPU when:
  ─────────────────────────────────────  ────────────────────────────────────
  N is large (millions of elements)      N is small (thousands or fewer)
  Same operation on every element        Complex control flow per element
  Regular memory access (coalesced)      Irregular/pointer-chasing access
  High arithmetic intensity              Low AI (memory-bound, PCIe-limited)
  Data stays on GPU across many ops      Frequent host↔device transfers needed
  Batch processing (large batches)       Real-time, single-sample inference
  Training (data-parallel SGD)           Compilation, OS scheduling, parsing
  Image/video processing                 Graph traversal (BFS/DFS)
  Matrix operations (large matrices)     Very small matrices (<128×128)
  Sorting large arrays (radix sort)      Sorting tiny arrays (quicksort better)
The GPU is a coprocessor. You still need the CPU.
The CPU runs your OS, your Python interpreter, your data loading pipeline, your preprocessing, your postprocessing, and the CUDA driver itself. Even in a GPU-intensive training loop, the CPU is:
  • Reading data from disk and decoding it (DataLoader workers)
  • Calling PyTorch/CUDA APIs to launch kernels
  • Managing memory allocations and stream synchronization
  • Running the optimizer (sometimes, if not fused on GPU)
  • Logging, checkpointing, LR scheduling
If any of these CPU steps takes longer than your GPU kernels, your GPU is sitting idle waiting for the CPU. The GPU utilization metric in nvidia-smi tells you the truth: if it is below 90%, something on the CPU is your real bottleneck, not the kernels.

Practical Amdahl's Law example: ML training

  Typical training step breakdown (medium model):
  ────────────────────────────────────────────────────────────────
  Data loading + augmentation (CPU):     8 ms   ← this is your "serial" fraction
  H2D transfer (cudaMemcpy):             2 ms   ← part of transfer overhead
  Forward pass (GPU kernels):           15 ms
  Backward pass (GPU kernels):          28 ms
  Optimizer step (GPU or CPU):           5 ms
  D2H transfer (loss, metrics):          0.1 ms
  ────────────────────────────────────────────────────────────────
  Total step time:                      58 ms
  Pure GPU time:                        48 ms
  Serial/overhead time:                 10 ms   → 1/(10/58) = max 5.8× speedup
                                                  from any GPU improvement

  If you 10× your GPU speed (kernels take 4.8 ms instead of 48 ms):
  New total: 10 ms + 4.8 ms = 14.8 ms
  Speedup:   58 / 14.8 = 3.9× despite 10× faster GPU kernels!

  Fix: optimize data loading (prefetch, more workers, faster augmentation)
       not the GPU kernels.

GPU speedup claims in papers often ignore transfer time.

A common practice in GPU computing papers is to benchmark "GPU kernel time" without including the cudaMemcpy overhead. This makes speedups look dramatically better than they are in end-to-end deployment. When someone claims "100× GPU speedup," ask these questions:

1. Does that include data transfer time? (Often: no)
2. What is the CPU baseline? (Often: single-threaded, no SIMD)
3. Is the problem size realistic? (Often: chosen to maximize GPU advantage)
4. Is the data already resident on GPU? (Common in ML, but not in general computing)

In production ML systems where data stays on GPU across many operations (training loop), the transfer overhead is amortized and GPU speedups are real. In batch-processing pipelines with frequent host/device round-trips, measure end-to-end throughput, not kernel time.

The Mental Model You Must Have Before Proceeding

Before you write a single CUDA kernel, these facts must be internalized:

  • The GPU is a throughput machine. It is fast in aggregate, not per-thread. A single GPU thread is slower than a single CPU thread. You need thousands of parallel threads to win.
  • Memory bandwidth is almost always your bottleneck. If your kernel does few FLOPs per byte read, adding more FLOPs won't help. Optimize memory access patterns first. This is the subject of the entire Phase 3 of this series.
  • PCIe is an order of magnitude slower than GPU memory. Every cudaMemcpy is a performance tax. Design your pipeline to minimize it.
  • Amdahl's Law always applies. If 10% of your workload cannot be parallelized (or is CPU-bound), your maximum GPU speedup is 10×, regardless of how many GPU cores you have or how well you optimize the GPU portion.
  • Warp divergence kills throughput. Write kernels where all 32 threads in a warp take the same code path. Boundary checks at the kernel level are fine; per-element random branching is not.
  • Measure everything. Intuitions about performance fail constantly in GPU programming. Profile with ncu (kernel-level) and nsys (system-level) before optimizing anything.

Takeaways

  • CPUs are latency-optimized (fast for one thread); GPUs are throughput-optimized (fast for many threads simultaneously).
  • CPU uses large caches, branch prediction, and out-of-order execution to hide latency. GPU uses massive multithreading — when one warp stalls, the SM switches to another.
  • The SM is the fundamental GPU compute unit. An A100 has 108 SMs, each with 64 CUDA cores, 4 warp schedulers, 65,536 registers, and up to 164 KB shared memory.
  • HBM2e gives A100 2,039 GB/s of memory bandwidth — ~10× DDR5 — because of the 5,120-bit bus width from stacked HBM dies.
  • Arithmetic Intensity (FLOPs/byte) determines whether you are memory-bound or compute-bound. Most simple kernels are memory-bound. MatMul at large N is compute-bound.
  • SIMT hides vector width from the programmer but penalizes warp divergence. Write kernels where all threads in a warp take the same branch.
  • Amdahl's Law caps speedup at 1/(serial fraction). A 10% serial fraction limits you to 10× speedup no matter how big your GPU is.
  • PCIe bandwidth (~24 GB/s) is 85× slower than GPU memory. Include transfer time in all benchmarks. GPU wins only when compute time dominates transfer time.
