CUDA Interview Preparation — 40 Questions and Complete Answers

These 40 questions cover everything interviewers ask about GPU programming: thread hierarchy, memory hierarchy, coalescing, warps, reduction, streams, profiling, and architecture. For each question I give the answer I would give in an interview — direct, technical, and with numbers.

Thread Model and Execution (Q1–10)

Q1. What is a warp, and why is it 32?
A warp is a group of 32 threads that are scheduled and executed together on an SM. The hardware executes all 32 threads in lockstep — same instruction, different data (SIMT). The size 32 is a hardware design choice that balances SIMT width against die area; it has been 32 since CUDA 1.0 and is unlikely to change. Key implications: (1) block sizes should be multiples of 32 to avoid partial warps wasting capacity, (2) threads within a warp can use __shfl_sync() for zero-cost communication, (3) divergent branches within a warp are serialized.
Q2. What is the difference between a thread block and a warp?
A thread block is a programmer-defined unit of up to 1024 threads that share an SM and can communicate via shared memory + __syncthreads(). A warp is a hardware-defined unit of exactly 32 threads that execute in lockstep. A block of 256 threads contains 8 warps. Blocks are software abstractions; warps are hardware execution units. You define blocks; the hardware creates warps from them automatically.
Q3. What is SIMT and how does it differ from SIMD?
SIMT (Single Instruction Multiple Threads) executes the same instruction on 32 threads simultaneously. Unlike SIMD (e.g., x86 AVX), SIMT allows threads to diverge — take different code paths through if/else — with a serialization penalty. SIMD requires all lanes to do exactly the same operation. SIMT threads can also be at different memory addresses independently, whereas SIMD typically requires contiguous data. The tradeoff: SIMT is easier to program (write scalar code) but divergence costs throughput.
Q4. What is occupancy and why does it matter?
Occupancy = (active warps on SM) / (maximum warps on SM). For an A100: max = 64 warps. It matters because the GPU hides memory latency by switching between ready warps. More active warps = more latency hiding = better throughput for memory-bound kernels. However, occupancy is not linearly related to performance — going from 25% to 50% occupancy often has minimal impact once the SM has enough warps to hide L2 latency (~8 warps). Resource limits: registers (65,536/SM ÷ regs/thread), shared memory (192KB/SM ÷ smem/block), and thread count (2048/SM).
Q5. What is warp divergence and how do you fix it?
Warp divergence occurs when threads within the same warp take different code paths (if/else, switch). The hardware executes both paths sequentially with inactive threads masked off, reducing effective throughput. For a 50/50 branch, throughput is halved. Fixes: (1) sort data so threads in the same warp always take the same branch, (2) use arithmetic instead of branches (e.g., fmaxf(val, 0) instead of if(val<0) val=0), (3) restructure into separate kernels for different cases, (4) use warp voting (__any_sync, __all_sync) to detect uniform branches. Note: bounds-check divergence (if i >= N return) only affects the last warp and is negligible.
Q6. How does the GPU hide memory latency?
The SM has multiple warp schedulers (4 on A100). When a warp issues a global memory load (800-cycle latency), the scheduler immediately switches to another ready warp. If the SM has enough active warps, computation and memory latency overlap completely. This is called latency hiding through multi-threading — fundamentally different from a CPU's approach (out-of-order execution, cache prefetch). This is why occupancy matters: more active warps = more opportunities to hide latency. A100 can execute 64 warps × 4 schedulers = up to 256 instructions per cycle if no stalls.
Q7. Why should block sizes be multiples of 32?
Because the hardware schedules warps of 32. A block of 33 threads requires 2 warps — but the second warp has only 1 active thread and 31 idle threads. Those 31 idle threads still consume SM resources (register file slots, shared memory allocation) while doing no work. The wasted resources reduce occupancy for other blocks. Common good block sizes: 128, 256, 512. Avoid odd numbers like 100, 200, or 384 unless there's a specific reason.
Q8. What is the maximum number of threads per block?
1024 threads per block (3D: x×y×z ≤ 1024). This limit exists because all threads in a block must share an SM, and the SM has limited resources: 65,536 registers, 192 KB shared memory, 2048 maximum resident threads. A block of 1024 threads occupies at least 1024 register slots per register used. In practice, blocks of 256 or 512 are more common because they leave room for multiple concurrent blocks per SM.
Q9. What does __syncthreads() do?
__syncthreads() is a block-wide barrier: it stalls each thread until ALL threads in the block have reached the barrier, ensuring all shared memory writes before the barrier are visible to all threads after it. It is essential when threads share data through shared memory: write phase → __syncthreads() → read phase. Never put it inside a conditional that only some threads enter (deadlock). It does NOT sync across blocks (there is no cross-block sync in CUDA; use separate kernel launches or cooperative groups).
Q10. What is the difference between cudaDeviceSynchronize() and cudaStreamSynchronize()?
cudaDeviceSynchronize() blocks the CPU until ALL pending CUDA operations on ALL streams complete. cudaStreamSynchronize(stream) blocks the CPU until all operations in that specific stream complete — other streams continue running. Use cudaStreamSynchronize() in production to avoid unnecessarily blocking independent work. cudaDeviceSynchronize() is useful for debugging (ensures all errors are surfaced) and for benchmarking (ensures timing is accurate).

Memory Hierarchy (Q11–20)

Q11. What are the main memory types in CUDA and their access latencies?
Registers: ~1 cycle (on-chip, per-thread). Shared memory: ~5 cycles (on-chip, per-block). L1 cache: ~30 cycles (on-chip, per-SM). L2 cache: ~200 cycles (on-chip, all SMs). Global memory: ~800 cycles (off-chip HBM). Constant memory: ~5 cycles cached, ~800 uncached. Local memory: ~800 cycles (global DRAM, used for register spill). The 800× gap between registers and global memory is the central challenge of CUDA performance engineering.
Q12. What is coalesced memory access?
When 32 consecutive threads in a warp access 32 consecutive memory addresses, the hardware combines them into a single 128-byte L2 transaction. This is called coalesced access and achieves peak bandwidth. If threads access scattered addresses, each thread may need its own transaction — up to 32 transactions for a single warp, reducing effective bandwidth by 32×. Diagnosis: ncu sectors/request ratio should be 1.0 for coalesced, 32.0 for worst case. Fix: ensure thread i accesses data[i] (not data[i*stride]).
Q13. What are shared memory bank conflicts?
Shared memory is divided into 32 banks (4 bytes each). All 32 banks can be accessed simultaneously. A bank conflict occurs when two or more threads in the same warp access different addresses in the SAME bank — the hardware serializes these into multiple transactions. A 32-way conflict (all threads in the same bank) is 32× slower than conflict-free access. Classic cause: accessing a column of a 32-wide 2D shared array (stride=32 → all threads map to bank 0). Fix: add +1 padding to the array width.
Q14. What is the purpose of constant memory?
Constant memory is a 64 KB read-only memory space with a dedicated cache that can broadcast a single value to all 32 threads in a warp in one transaction (vs. 32 separate transactions from global memory). It is ideal for data that all threads read simultaneously: kernel configuration parameters, convolution filters, camera matrices. Declared with __constant__ at file scope, populated from CPU with cudaMemcpyToSymbol(). Performance degrades to global memory when different threads read different addresses in constant memory.
Q15. What is local memory in CUDA?
Despite the name, local memory is NOT fast. It is physically located in global DRAM with ~800-cycle latency. It is private to each thread. The compiler uses it automatically for register spill (when a kernel needs more registers than available) and for arrays indexed with non-constant indices at runtime. Detect with nvcc -Xptxas -v (look for non-zero "lmem") or ncu (l1tex__data_pipe_lsu_wavefronts_mem_local). Fix: reduce register usage, use __launch_bounds__, or restructure to use shared memory.
Q16. When would you use texture memory?
Texture memory is useful when: (1) access has 2D spatial locality (image processing, stencils) — the texture cache pre-fetches spatially nearby elements; (2) you need hardware bilinear/trilinear interpolation for free; (3) you need automatic boundary handling (clamp, mirror, wrap) at no cost; (4) the data is read-only. Modern CUDA recommendation: use __ldg() for simple read-only caching, use texture objects for 2D data needing interpolation or boundary modes.
Q17. What is Unified Memory and when should you use it?
Unified Memory (cudaMallocManaged) creates a single virtual address space accessible from both CPU and GPU. The driver migrates pages via PCIe on demand using page faults. Use it for: prototyping (less code), complex pointer-based data structures (trees, graphs), iterative algorithms where CPU and GPU alternate. Avoid in production when: performance is critical (explicit cudaMemcpy is more predictable), large streaming datasets (migration overhead per fault), or tight kernel loops. Always prefetch with cudaMemPrefetchAsync to avoid page fault stalls during kernel execution.
Q18. What is register spilling and how do you fix it?
Register spilling occurs when a kernel needs more registers than available on the SM (65,536 on A100 / threads_per_SM). Excess register state is written to local memory (global DRAM), causing hidden bandwidth overhead. Detect with nvcc -Xptxas -v (non-zero lmem). Fix: (1) reduce register usage by simplifying the kernel or breaking it into multiple passes, (2) add __launch_bounds__(maxThreadsPerBlock, minBlocksPerSM) to give the compiler occupancy hints, (3) use nvcc --maxrregcount=N to hard-cap registers (may cause explicit spill), (4) restructure to move data through shared memory instead of keeping it in registers.
Q19. What is the L2 cache size on an A100 and how can you control it?
The A100 has 40 MB of L2 cache shared across all 108 SMs. This is large enough to hold many practical working sets. CUDA 11.0+ allows L2 persistence: cudaStreamSetAttribute with cudaStreamAttributeAccessPolicyWindow lets you pin a data region in L2 (hitRatio controls what fraction of L2 is reserved, hitProp=cudaAccessPropertyPersisting). This is useful when a kernel repeatedly accesses a hot dataset that fits in L2 but would otherwise be evicted by other traffic.
Q20. Explain AOS vs SOA and when each is appropriate.
AOS (Array of Structures): struct Point{float x,y,z;} pts[N]. SOA (Structure of Arrays): float px[N], py[N], pz[N]. AOS causes stride-3 access when reading only one field (e.g., all x values) — cache lines load 3 fields but only 1 is used, wasting 67% bandwidth. SOA gives coalesced access: all px[i] are adjacent, one cache line load serves all threads. Use SOA when: kernels access one or two fields at a time (most GPU kernels). Use AOS when: kernels always access all fields of each element together (rare on GPU).

Optimization Patterns (Q21–30)

Q21. Explain the tiled matrix multiplication optimization.
Naive matmul has each thread load an entire row and column from global memory (O(N) loads per thread, O(N^3) total). Tiled matmul divides the matrices into tiles of size T×T. All T^2 threads in a block cooperatively load one tile of A and one tile of B into shared memory (~5 cycle access), compute the partial dot product, then load the next tile. Each global memory value is loaded once per tile sweep but used T times from shared memory. This reduces global traffic by T× (e.g., T=32 → 32× fewer global loads) and achieves near-peak bandwidth utilization.
Q22. What is the warp shuffle and when do you use it?
Warp shuffle intrinsics (__shfl_sync, __shfl_down_sync, etc.) allow threads within a warp to directly read another thread's register without going through shared memory. This is faster than shared memory (no load/store, no bank conflict possible, no __syncthreads needed). Use for: warp-level reduction (6 __shfl_down_sync iterations reduces 32 values to 1), prefix scan within a warp, broadcasting a single value to all lanes, butterfly patterns. Requires sm_30+ and a synchronization mask (usually 0xFFFFFFFF for all lanes).
Q23. Describe the 6 reduction optimizations from Mark Harris's paper.
(1) Interleaved addressing: works but causes divergence (50% threads idle each step). (2) Sequential addressing: reverse stride order, eliminates divergence. (3) First-add-on-load: each thread loads 2 elements and adds them immediately, halving idle threads. (4) Unroll last warp: last 6 reduction steps don't need __syncthreads (warp already in lockstep), save 5 barriers. (5) Complete unroll with templates: compile-time unrolling eliminates all loop overhead. (6) Warp shuffle: replace last 5 shared memory steps with __shfl_down_sync — no shared memory needed. Cumulative speedup: ~15-30× from step 1 to step 6. CUB::DeviceReduce adds another 2× with hardware-specific tuning.
Q24. What are CUDA streams and what can they overlap?
A CUDA stream is a sequence of operations that execute in order. Multiple streams execute concurrently. An A100 has separate hardware engines for: H→D PCIe copy, D→H PCIe copy, and kernel execution. By putting these in different streams, all three can run simultaneously. Classic use: chunk large data, pipeline H→D copy of chunk N+1 with kernel execution of chunk N and D→H copy of chunk N-1. Requires pinned (page-locked) host memory for async copies to work. Default stream (stream 0) is a global sync point — use non-default streams for overlap.
Q25. How do CUDA atomic operations work and what is their performance cost?
Atomic operations perform read-modify-write as an indivisible hardware transaction, preventing race conditions. The hardware serializes competing atomics to the same address. Performance depends entirely on contention: with 1 unique address and 2048 threads/SM, atomics serialize into 2048 sequential operations (devastating). With 1024+ unique addresses, atomics approach non-atomic performance. Optimization: use shared memory atomics to reduce to one atomic per block per bin (shared atomics are fast), then one global atomic per block. For global counters, use warp aggregation to reduce 32 atomics to 1.
Q26. What is the Roofline model?
The Roofline model bounds achievable performance as min(peak_FLOPS, bandwidth × AI). Arithmetic Intensity (AI) = FLOPs / bytes. For an A100: peak = 19.5 TFLOPS, bandwidth = 2 TB/s, ridge point = 9.75 FLOPs/byte. Kernels below the ridge point are memory-bandwidth-limited; above it, compute-limited. Vector add (AI=0.08) can achieve at most 0.08 × 2000 = 160 GFLOPS, not 19,500. This tells you: to make vector add faster, increase AI through data reuse — not by making the adds faster.
Q27. What is dynamic parallelism and when should you use it?
Dynamic parallelism allows GPU kernels to launch child kernels without CPU involvement. Use it when: the amount of work is data-dependent and only known after some GPU processing (adaptive mesh refinement, BFS with variable frontier size, recursive algorithms). Don't use it when: fixed-size sub-problems exist (just launch a larger grid from CPU), or when the overhead is prohibitive (child kernel launch costs ~50μs vs ~5μs for CPU launch). Requires -rdc=true compilation and -lcudadevrt linking. Alternatives: persistent kernels with work queues, compaction + second launch from CPU.
Q28. How do you time CUDA kernels accurately?
CPU timers (gettimeofday, std::chrono) are wrong for GPU kernels because kernel launches are asynchronous — the CPU timer captures launch time, not execution time. Use CUDA events: cudaEventRecord(start) before the kernel, cudaEventRecord(stop) after, cudaEventSynchronize(stop) to block until stop is recorded, then cudaEventElapsedTime(&ms, start, stop). This measures actual GPU execution time. For multiple kernel runs, warm up with 1-2 runs first (JIT compilation, cache warming) then average 10-100 runs for stable timing.
Q29. What is __launch_bounds__ and when do you use it?
__launch_bounds__(maxThreadsPerBlock, minBlocksPerSM) is a kernel qualifier that gives the compiler occupancy hints. maxThreadsPerBlock tells the compiler the maximum block size you will use — it can then spill registers to stay within the register budget for that occupancy level. minBlocksPerSM tells the compiler the minimum concurrent blocks you want — it limits register allocation to ensure this occupancy is achievable. Use it when: (1) your kernel uses many registers and has low occupancy, (2) you want to prevent the compiler from over-allocating registers, (3) you need reproducible register counts across compiler versions.
Q30. What are grid-stride loops and why are they preferred?
A grid-stride loop processes multiple elements per thread: for (int i = blockIdx.x * blockDim.x + threadIdx.x; i < N; i += gridDim.x * blockDim.x). Each thread processes elements at stride = total thread count. Benefits: (1) works correctly for any N, even N much smaller than total threads; (2) works correctly for any grid size; (3) easy to scale — increase grid size and each thread does less work; (4) enables persistent kernels where N is unbounded. The alternative (one thread per element, return if out of bounds) requires precise grid size calculation and breaks if grid size is wrong.

Profiling and Architecture (Q31–40)

Q31. What is the difference between nsys and ncu?
nsys (Nsight Systems) profiles at system level: shows timeline of all CPU threads, GPU kernels, memory copies, NVTX annotations. Use it first to find which kernel is hottest. 5-10% overhead. ncu (Nsight Compute) profiles individual kernels with hardware performance counters: memory throughput, occupancy, stall reasons, instruction mix, cache efficiency. 5-20× overhead. Workflow: nsys to find bottleneck → ncu on that kernel → optimize → repeat.
Q32. How many SMs does an A100 have and what are its key specs?
A100: 108 SMs, 6,912 CUDA cores (64 FP32 per SM), 432 Tensor Cores (4 per SM), 40 MB L2, 80 GB HBM2e, ~2 TB/s memory bandwidth, 19.5 TFLOPS FP32, 312 TFLOPS TF32 Tensor, 624 TFLOPS FP16 Tensor. Each SM: 65,536 registers (256 KB), 192 KB shared/L1, 4 warp schedulers, max 1024 threads / 32 warps / 2048 threads. NVLink 3.0: 600 GB/s bidirectional.
Q33. What is compute capability and why does it matter?
Compute capability (e.g., sm_80 for A100) identifies the GPU architecture and feature set. Higher compute capability = newer features: sm_30 added dynamic parallelism, sm_60 added FP64 atomic add, sm_70 added Tensor Cores and independent thread scheduling (SIMT divergence model change), sm_80 added TF32 and BF16. Specify with nvcc -arch=sm_80. Using -arch=sm_75 on an A100 compiles slower code and misses A100-specific instructions. Use -arch=native to compile for the current GPU automatically.
Q34. What is compute-sanitizer and what does it detect?
compute-sanitizer is NVIDIA's GPU memory debugger (successor to cuda-memcheck). It has four modes: memcheck (out-of-bounds accesses, use-after-free, uninitialized reads), racecheck (shared memory races, reads before __syncthreads), synccheck (missing or mismatched __syncthreads), initcheck (uninitialized global memory reads). Run with: compute-sanitizer ./myprogram. Overhead: 5-10× slower. Use during development; not production. Catches ~80% of CUDA memory bugs that cause silent corruption.
Q35. What is arithmetic intensity and how do you calculate it?
Arithmetic Intensity = (number of floating-point operations) / (bytes transferred from/to global memory). Example for SAXPY (y = a*x + y, N elements): FLOPs = 2N (1 multiply + 1 add per element), Bytes = 3N*4 (read x, read y, write y = 3 arrays of N floats). AI = 2N / 12N = 0.17 FLOPs/byte. For matrix multiply (N×N): FLOPs = 2N^3, Bytes = 3N^2*4 (read A, read B, write C). AI = 2N^3 / 12N^2 = N/6. For N=1024: AI = 171, well above A100 ridge point (9.75) → compute-bound.
Q36. Why is pinned memory required for async memory copies?
The GPU's DMA engine accesses host memory directly via PCIe. It requires the host pages to be pinned (page-locked) — guaranteed to remain at fixed physical addresses. Regular malloc memory is pageable: the OS can swap it to disk or move it in physical memory at any time. If the GPU DMA engine tries to access a swapped-out page, the result is undefined. cudaMallocHost (alias cudaHostAlloc) pins pages via OS mlock. Async copies with pageable memory silently fall back to synchronous copies. Also: pinned memory achieves ~4× higher PCIe bandwidth than pageable memory.
Q37. What is the HBM2e memory on an A100 and how does it differ from GDDR6?
HBM2e (High Bandwidth Memory 2e) is stacked DRAM connected to the GPU die via silicon interposer with extremely wide buses (5,120-bit on A100 vs 384-bit for GDDR6X). This enables A100's 2 TB/s bandwidth vs ~1 TB/s for GDDR6X GPUs (e.g., RTX 3090). Trade-off: HBM is more expensive and physically limited in capacity (80 GB on A100 vs 24 GB on RTX 3090), but bandwidth scales better with width. For bandwidth-limited workloads (most ML training, scientific computing), the 2× bandwidth advantage is decisive.
Q38. What are Tensor Cores and when can you use them?
Tensor Cores are special matrix-multiply-accumulate units. An A100 Tensor Core executes a 16×16 matrix multiply per clock (D = A×B + C) in one operation. They support FP16, BF16, TF32, INT8, FP64 data types. Tensor Core throughput: 312 TFLOPS (TF32) vs 19.5 TFLOPS (FP32) — 16× faster. Access via: cuBLAS (automatic for gemm), cuDNN (automatic for convolutions), WMMA API (manual: wmma::load_matrix_sync, wmma::mma_sync), or inline PTX. Require: matrix dimensions as multiples of 16 (or 8 for some types), appropriate data types, and sm_70+.
Q39. Explain NVLink and when it matters.
NVLink is NVIDIA's high-bandwidth interconnect between GPUs (and between GPU and CPU for Grace Hopper). NVLink 3.0 (A100): 600 GB/s bidirectional vs PCIe 4.0's 64 GB/s — a 9× bandwidth advantage. NVLink matters for: (1) multi-GPU training where gradients must be exchanged (all-reduce operations), (2) peer-to-peer access (one GPU reads another's memory), (3) Unified Memory between GPUs (pages migrated over NVLink instead of PCIe). Without NVLink, multi-GPU communication is PCIe-bottlenecked.
Q40. What is cooperative groups and how does it extend the CUDA thread model?
Cooperative Groups (CUDA 9+) generalizes the thread synchronization model beyond fixed-size thread blocks. Instead of __syncthreads() (entire block), you can sync at warp level (cooperative_groups::coalesced_threads()), or even grid level (requires cooperative launch: cudaLaunchCooperativeKernel). This enables: flexible warp-level synchronization, grid-wide barriers (for iterative algorithms that previously needed separate kernel launches), and cleaner code for algorithms with irregular thread groupings. The grid_group type allows synchronizing the entire grid in a single kernel, enabling in-place reductions without multiple kernel launches.

Coding Challenges

These are the most common CUDA coding questions asked in interviews. Practice writing them from memory.

1. Implement vectorAdd

__global__ void vectorAdd(const float* A, const float* B, float* C, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) C[i] = A[i] + B[i];
}

2. Implement parallel reduction sum

__device__ float warpReduceSum(float val) {
    for (int offset = 16; offset > 0; offset /= 2)
        val += __shfl_down_sync(0xFFFFFFFF, val, offset);
    return val;
}
__global__ void blockReduceSum(float* in, float* out, int n) {
    __shared__ float warpSums[32];
    int tid = threadIdx.x, lane = tid % 32, warpId = tid / 32;
    int i = blockIdx.x * blockDim.x + tid;
    float val = (i < n) ? in[i] : 0.0f;
    val = warpReduceSum(val);
    if (lane == 0) warpSums[warpId] = val;
    __syncthreads();
    val = (tid < blockDim.x/32) ? warpSums[tid] : 0.0f;
    if (warpId == 0) val = warpReduceSum(val);
    if (tid == 0) out[blockIdx.x] = val;
}

3. Implement matrix transpose using shared memory

#define TILE 32
__global__ void transpose(const float* in, float* out, int W, int H) {
    __shared__ float tile[TILE][TILE + 1];  // +1 avoids bank conflicts
    int x = blockIdx.x * TILE + threadIdx.x;
    int y = blockIdx.y * TILE + threadIdx.y;
    if (x < W && y < H) tile[threadIdx.y][threadIdx.x] = in[y * W + x];
    __syncthreads();
    int tx = blockIdx.y * TILE + threadIdx.x;
    int ty = blockIdx.x * TILE + threadIdx.y;
    if (tx < H && ty < W) out[ty * H + tx] = tile[threadIdx.x][threadIdx.y];
}
