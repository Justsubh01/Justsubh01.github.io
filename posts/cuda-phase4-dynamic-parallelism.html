Dynamic Parallelism — Kernels Launching Kernels

Dynamic Parallelism lets GPU kernels launch child kernels without returning to the CPU. This enables adaptive algorithms where the GPU itself decides how much work to spawn based on data seen at runtime — recursive refinement, tree traversal, and variable-depth BFS all become natural to express.

What Is Dynamic Parallelism

Before Dynamic Parallelism (pre-Kepler), every kernel launch had to come from the CPU. If your algorithm needed to decide how much work to do based on GPU-computed results, you had to: run kernel → copy result to CPU → decide → launch next kernel. The CPU round-trip added ∼10-50 µs per decision point.

With Dynamic Parallelism (Kepler sm_35+), a kernel can launch child kernels directly from the GPU. The parent kernel continues running while child kernels execute. No CPU involvement required. 

// Without Dynamic Parallelism: CPU must coordinate
void cpuCoordinated(float* data, int n) {
    // Pass 1
    kernel1<<<grid, block>>>(data, n);
    cudaDeviceSynchronize();  // back to CPU

    // CPU reads result, decides how much work for pass 2
    int* h_count; cudaMallocHost(&h_count, sizeof(int));
    cudaMemcpy(h_count, d_count, sizeof(int), cudaMemcpyDeviceToHost);
    int newGrid = (*h_count + 255) / 256;

    // Pass 2
    kernel2<<<newGrid, 256>>>(data, *h_count);
    cudaFreeHost(h_count);
}

// With Dynamic Parallelism: GPU decides
__global__ void parentKernel(float* data, int n) {
    // Phase 1 work done by this thread...
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        // GPU launches child kernel with dynamically computed grid
        int newGrid = (d_count + 255) / 256;
        childKernel<<<newGrid, 256>>>(data, d_count);
        // Parent can continue doing other work while child runs
    }
}

Compiling for Dynamic Parallelism

# Requires separate compilation (-rdc=true) and linking with cudadevrt
nvcc -arch=sm_70 -rdc=true mykernel.cu -o myprogram -lcudadevrt

# Or with CMake:
set_property(TARGET mytarget PROPERTY CUDA_SEPARABLE_COMPILATION ON)
target_link_libraries(mytarget cudadevrt)
-rdc=true has a compile-time cost. Relocatable device code disables some compiler optimizations (function inlining across TUs, some PTX-level optimizations). For kernels that don't use dynamic parallelism, prefer the default -rdc=false.

Basic Usage: Child Kernel Launch

__global__ void childKernel(float* data, int n, int depth) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        data[i] = data[i] * depth;
    }
}

__global__ void parentKernel(float* data, int n) {
    // Each block independently launches a child kernel for its portion
    int blockStart = blockIdx.x * blockDim.x;
    int blockN     = min(blockDim.x, n - blockStart);

    if (threadIdx.x == 0) {  // only one thread per block launches child
        // Child kernel: separate grid, runs asynchronously
        childKernel<<<1, blockN>>>(data + blockStart, blockN, blockIdx.x);
        // Parent continues immediately (child runs in background)
    }
}

// Host:
parentKernel<<<numBlocks, blockSize>>>(d_data, N);
cudaDeviceSynchronize();  // waits for all children too
Memory visibility: Child kernels see all global memory writes made by the parent block before the child launch. But two sibling child kernels (launched by different parent threads) don't have guaranteed ordering unless explicitly synchronized.

Synchronizing Child Kernels

__global__ void parentWithSync(float* data, int n) {
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        // Launch child
        childKernel<<<16, 256>>>(data, n);

        // Wait for child to complete before reading its results
        cudaDeviceSynchronize();  // from device: waits for all children from this thread

        // Now safe to read data modified by childKernel
        float result = data[0];
        printf("Child result: %f\n", result);
    }
}

// Stream-based child synchronization:
__global__ void parentStreams(float* A, float* B, float* C, int n) {
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        cudaStream_t s1, s2;
        cudaStreamCreateWithFlags(&s1, cudaStreamNonBlocking);
        cudaStreamCreateWithFlags(&s2, cudaStreamNonBlocking);

        childA<<<8, 256, 0, s1>>>(A, n);  // runs in s1
        childB<<<8, 256, 0, s2>>>(B, n);  // runs concurrently in s2

        cudaDeviceSynchronize();  // wait for both

        mergeKernel<<<8, 256>>>(A, B, C, n);
        cudaDeviceSynchronize();

        cudaStreamDestroy(s1);
        cudaStreamDestroy(s2);
    }
}

Recursive Kernels

// Recursive quicksort: partition, then recursively sort each half
__global__ void quicksort(int* data, int lo, int hi) {
    if (lo >= hi) return;

    // Partition (simplified: use last element as pivot, one thread)
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        int pivot = data[hi];
        int i = lo - 1;
        for (int j = lo; j < hi; j++) {
            if (data[j] <= pivot) {
                i++;
                int tmp = data[i]; data[i] = data[j]; data[j] = tmp;
            }
        }
        i++;
        int tmp = data[i]; data[i] = data[hi]; data[hi] = tmp;
        int pivotIdx = i;

        // Recursively sort left and right partitions
        if (pivotIdx - 1 > lo)
            quicksort<<<1, 1>>>(data, lo, pivotIdx - 1);
        if (pivotIdx + 1 < hi)
            quicksort<<<1, 1>>>(data, pivotIdx + 1, hi);
    }
}

// Host call:
quicksort<<<1, 1>>>(d_data, 0, N - 1);
cudaDeviceSynchronize();
Recursive dynamic parallelism is limited in depth. Maximum nesting depth is 24 levels. Each level of recursion uses device memory for the launch configuration queue. Deep recursion on large datasets can exhaust the device synchronization queue (default: 2048 pending launches).

When to Use Dynamic Parallelism

Good Use CaseWhy
Adaptive mesh refinementGPU decides which cells need subdivision
BFS on irregular graphsFrontier size unknown until runtime
N-body with variable accuracyClose particles need more sub-steps
Recursive tree traversalDepth unknown at launch time
Iterative linear solversAdaptive stopping when residual < eps
Bad Use CaseWhy / Better Alternative
Fixed-size sub-problemsJust launch a larger grid from CPU
Regular reductionsUse CUB or two-pass reduction
Sorting small arraysThrust or CUB handle this better
Simple conditional workUse warp voting + compaction instead

Overhead and Limitations

Child kernel launch overhead on A100:

# Measured overhead per child kernel launch (A100): Parent kernel launch (from CPU): ~5 us Child kernel launch (from GPU): ~50 us (10x slower than CPU launch) Child kernel launch in tight loop: ~10 us (amortized after warmup) # Launch limits: Max pending launches per device: 2048 (can set higher with cudaDeviceSetLimit) Max recursion depth: 24 Max child grid dimensions: Same as normal grid

Child kernel launches have significant overhead compared to parent launches. Launch thousands of tiny child kernels (1 thread each) and you'll spend more time on launch overhead than computation. Batch work into fewer, larger child kernels.

Alternatives to Dynamic Parallelism

Before resorting to dynamic parallelism, consider these alternatives which often have lower overhead:

  • Cooperative Groups: Flexible synchronization without nested launches. Sync at thread, warp, block, or grid level.
  • Work queues: One persistent kernel + global work queue. Threads pull work items atomically. Avoids launch overhead entirely.
  • CUB DeviceScan + Compaction: Remove inactive elements before processing, eliminating the need for adaptive grid sizing.
  • CPU dispatch loop: For coarse-grained decisions, a CPU loop with cudaDeviceSynchronize() has predictable latency and is often fast enough.
// Persistent kernel with work queue (often better than dynamic parallelism)
__global__ void persistentKernel(WorkItem* queue, int* tail, int maxWork) {
    while (true) {
        int idx = atomicAdd(tail, 1);  // claim next work item
        if (idx >= maxWork) break;

        WorkItem item = queue[idx];
        processItem(item);

        // Enqueue new work dynamically into the same queue
        if (item.needsSubdivision) {
            int newIdx = atomicAdd(tail, 2);  // claim 2 new slots
            if (newIdx + 1 < maxWork) {
                queue[newIdx]   = item.left;
                queue[newIdx+1] = item.right;
            }
        }
    }
}

// Launch once with enough threads to keep SM busy, runs until queue empty
persistentKernel<<<numSMs * 4, 256>>>(d_queue, d_tail, maxWork);
