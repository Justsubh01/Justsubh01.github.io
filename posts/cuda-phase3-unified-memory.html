Unified Memory — One Pointer, CPU and GPU

Unified Memory lets a single pointer be accessed from both CPU and GPU without explicit cudaMemcpy calls. The driver migrates pages automatically. Understanding when to use it (and when not to) can save you hundreds of lines of boilerplate — or cause subtle performance cliffs if misused.

The explicit CUDA memory management pattern has 7 steps: malloc, cudaMalloc, cudaMemcpy H→D, kernel, cudaMemcpy D→H, free, cudaFree. For complex data structures like linked lists, trees, or graphs where pointers live inside device data, this is nearly impossible to manage correctly. Unified Memory solves this.

What Is Unified Memory

cudaMallocManaged() allocates memory in a virtual address space accessible from BOTH the CPU and GPU. Physically, the data lives somewhere — initially on the CPU, migrated to the GPU when the GPU accesses it — but you use the same pointer everywhere. 

// Without Unified Memory: 60+ lines of boilerplate
float *h_A, *h_B, *h_C;
float *d_A, *d_B, *d_C;
size_t bytes = N * sizeof(float);

h_A = (float*)malloc(bytes);
h_B = (float*)malloc(bytes);
h_C = (float*)malloc(bytes);
cudaMalloc(&d_A, bytes);
cudaMalloc(&d_B, bytes);
cudaMalloc(&d_C, bytes);
// initialize h_A, h_B...
cudaMemcpy(d_A, h_A, bytes, cudaMemcpyHostToDevice);
cudaMemcpy(d_B, h_B, bytes, cudaMemcpyHostToDevice);
vectorAdd<<<grid, block>>>(d_A, d_B, d_C, N);
cudaMemcpy(h_C, d_C, bytes, cudaMemcpyDeviceToHost);
// use h_C...
free(h_A); free(h_B); free(h_C);
cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);

// With Unified Memory: 30 lines
float *A, *B, *C;
size_t bytes = N * sizeof(float);
cudaMallocManaged(&A, bytes);
cudaMallocManaged(&B, bytes);
cudaMallocManaged(&C, bytes);
// initialize A, B directly (CPU access)
for (int i = 0; i < N; i++) { A[i] = i; B[i] = N - i; }
vectorAdd<<<grid, block>>>(A, B, C, N);
cudaDeviceSynchronize();   // wait for GPU
// use C directly (CPU access)
printf("C[0] = %f\n", C[0]);
cudaFree(A); cudaFree(B); cudaFree(C);
Unified Memory truly shines for complex data structures. A tree, linked list, or graph where nodes contain pointers to other nodes is extremely difficult to copy to the GPU explicitly — every pointer must be manually updated to point to the device copy. With UM, the structure works on both CPU and GPU without any pointer fixup.

Page Fault Mechanism

When you call cudaMallocManaged(), the memory starts as unmapped. On the first access (CPU or GPU), the system takes a page fault:

  1. Fault trapped by GPU driver / OS
  2. Page(s) migrated to the accessing device via PCIe (∼12 GB/s)
  3. TLB updated on source and destination
  4. Execution resumes

Page size is typically 64 KB on discrete GPUs. A single page fault migrates 64 KB. If your kernel accesses a 1 MB array non-sequentially, it triggers 16 page faults, each with significant overhead.

Page faults during kernel execution stall the kernel. A kernel waiting on page migration is effectively paused while PCIe transfers complete. For production workloads, you almost always want to prefetch data before the kernel runs rather than relying on demand paging.
# Profile page faults with nvprof / nsys: nsys profile --trace=cuda,osrt ./myprogram # Look for "CPU Page Faults" and "GPU Page Faults" in the timeline # With on-demand paging (no prefetch): GPU Page Faults: 487 faults (each fault = ~5-50 us overhead) Total fault time: 24.3 ms Kernel time: 28.1 ms <-- 87% of kernel time was page faults! # With prefetching: GPU Page Faults: 0 Kernel time: 3.9 ms <-- 7x faster

Prefetching — Eliminating Faults

cudaMemPrefetchAsync() tells the driver to migrate pages to a device before they are needed, overlapping the migration with other work: 

float *A, *B, *C;
cudaMallocManaged(&A, N * sizeof(float));
cudaMallocManaged(&B, N * sizeof(float));
cudaMallocManaged(&C, N * sizeof(float));

// Initialize on CPU
for (int i = 0; i < N; i++) { A[i] = i; B[i] = N - i; }

int deviceId;
cudaGetDevice(&deviceId);

// Prefetch A and B to GPU before kernel runs
// This starts the migration NOW, asynchronously
cudaMemPrefetchAsync(A, N * sizeof(float), deviceId);
cudaMemPrefetchAsync(B, N * sizeof(float), deviceId);
cudaMemPrefetchAsync(C, N * sizeof(float), deviceId);

// Launch kernel - data is already on GPU (no page faults during kernel)
vectorAdd<<<grid, block>>>(A, B, C, N);

// Prefetch result back to CPU for processing
cudaMemPrefetchAsync(C, N * sizeof(float), cudaCpuDeviceId);
cudaDeviceSynchronize();

// Access C on CPU - data already migrated, no fault
for (int i = 0; i < N; i++) printf("%f\n", C[i]);
Prefetching best practice: Always prefetch Unified Memory before a kernel that uses it. The prefetch is async — you can overlap it with CPU work or other GPU work. Think of it as "explicit copy but with one pointer" — you get the simplicity of UM and the performance of explicit copies.

Memory Advice Hints

cudaMemAdvise() provides hints to the driver about expected access patterns. Unlike prefetching (which moves data), advice influences how the driver manages pages: 

// Hint 1: data is read-mostly (create read-only copies on both CPU and GPU)
// Good for: lookup tables, constants shared between CPU and GPU
cudaMemAdvise(lookupTable, size, cudaMemAdviseSetReadMostly, deviceId);

// Hint 2: preferred location (keep data here by default)
// Reduces migration for data that mostly lives on GPU
cudaMemAdvise(d_data, size, cudaMemAdviseSetPreferredLocation, deviceId);

// Hint 3: this device will access the data (map direct instead of migrate if supported)
// Good for: NVLink-connected multi-GPU, or CPU that needs rare access
cudaMemAdvise(d_data, size, cudaMemAdviseSetAccessedBy, cudaCpuDeviceId);

// Unset advice:
cudaMemAdvise(d_data, size, cudaMemAdviseUnsetReadMostly, deviceId);
AdviceEffectBest for
SetReadMostlyDriver creates per-device copiesLookup tables, shared constants
SetPreferredLocationDriver biases toward this deviceData that 90%+ of accesses are from one device
SetAccessedByEnables direct mapping if topology supportsNVLink GPU–GPU, or rare CPU access

When to Use Unified Memory

Use Unified Memory when:

  • Prototyping — remove boilerplate, focus on algorithm
  • Data structures with internal pointers (trees, graphs, linked lists)
  • CPU and GPU take turns accessing the same data (iterative algorithms)
  • Memory size approaches GPU limit — UM can overflow to CPU memory

Do NOT use Unified Memory when:

  • Production performance is critical — explicit copies are more predictable
  • Large single-direction transfer (use cudaMemcpy for one bulk copy)
  • Tight kernel loops where the kernel runs hundreds of times on the same data
  • Debugging memory bugs — page fault stalls make profiling harder to read

Multi-GPU Unified Memory

With NVLink-connected GPUs (e.g., A100 DGX systems), Unified Memory can provide direct GPU-to-GPU memory access without going through the CPU. The driver automatically routes accesses over NVLink at ~600 GB/s — much faster than PCIe migration. 

// Multi-GPU UM: each GPU works on its portion, UM handles cross-GPU access
int nDevices;
cudaGetDeviceCount(&nDevices);

float *data;
cudaMallocManaged(&data, N * sizeof(float));

// Pin data to GPU 0 (reduce migration)
cudaMemAdvise(data, N * sizeof(float), cudaMemAdviseSetPreferredLocation, 0);

// Tell GPU 1 it will also access this data (map via NVLink if available)
cudaMemAdvise(data, N * sizeof(float), cudaMemAdviseSetAccessedBy, 1);

// Each GPU processes its half
cudaSetDevice(0);
processKernel<<<grid, block>>>(data, 0, N/2);
cudaSetDevice(1);
processKernel<<<grid, block>>>(data, N/2, N);
cudaDeviceSynchronize();
