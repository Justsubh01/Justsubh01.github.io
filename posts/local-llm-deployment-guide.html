Local LLM Deployment: Ollama vs vLLM vs llama.cpp — The Definitive Deep Dive

A complete engineering guide to running large language models on your own hardware — from first principles of transformer inference to production-grade serving, covering KV-cache mechanics, quantization internals, continuous batching, Flash Attention, speculative decoding, and every optimization technique that actually moves the needle.

Running LLMs locally has crossed a threshold. With a single RTX 3090 (24 GB VRAM) you can serve Llama 3.1 8B at full fp16 precision, hit sub-100ms time-to-first-token, and field hundreds of requests per hour — all without paying an API bill or sending your data to a third-party server. But there is an enormous gap between "it runs" and "it runs well," and closing that gap requires understanding what is actually happening inside these systems — not just memorizing commands.

This post goes deep on three tools that cover the full spectrum of local deployment: Ollama (the developer-friendly wrapper that gets you running in five minutes), vLLM (the production-grade throughput server built on PagedAttention), and llama.cpp (the bare-metal C++ engine that gives you maximum control with minimum overhead). I will explain not just how to use each one, but why they work the way they do — the memory math, the scheduler internals, the quantization tradeoffs, and the mental models that let you debug problems when they inevitably appear.

Hardware context: everything in this post is measured and tuned on an NVIDIA RTX 3090 (24 GB VRAM, 936 GB/s memory bandwidth), AMD Ryzen 7 3700X (16 threads), 32 GB RAM, CUDA 13.0. Where benchmarks appear, they are from this specific machine.

Part 0: The Mental Models You Need Before Touching Any Tool

Most deployment guides skip the fundamentals and jump straight to install commands. That works until something breaks — and then you have no idea why. The next two sections give you the mental models that make everything else click.

0.1 The VRAM Math — Know Before You Run

Every deployment decision ultimately comes down to a single question: does everything fit in VRAM? Not just the model weights, but the KV cache that grows with every token in the conversation. Getting this wrong means either an out-of-memory crash mid-request or silently falling back to CPU, which tanks your throughput by 10-50x.

The formula is simpler than most people expect. Model weights stored in fp16 (the default for GPU inference) consume exactly 2 bytes per parameter. So an 8-billion parameter model needs 8,000,000,000 × 2 = 16 GB just for weights — before you generate a single token. Quantizing to 4-bit halves that to 4 GB, which is why quantization is so powerful: it is the most direct lever you have on VRAM. 

# The two things competing for your VRAM
#
# 1. Model weights (fixed, determined at load time)
model_vram_bytes = num_parameters * bytes_per_param

# bytes_per_param by precision:
#   fp32  = 4 bytes  (training, not used for inference)
#   fp16 / bf16 = 2 bytes  (standard GPU inference)
#   int8  = 1 byte   (8-bit quantization)
#   int4  = 0.5 bytes (4-bit GPTQ / AWQ / GGUF Q4)

# 2. KV cache (grows with context length and batch size)
# For each layer, we store one K matrix and one V matrix per token
kv_bytes_per_token = 2 * num_layers * num_kv_heads * head_dim * bytes_per_param

# Llama 3.1 8B specifics:
#   num_layers = 32
#   num_kv_heads = 8  (GQA: grouped query attention)
#   head_dim = 128
#   fp16: 2 * 32 * 8 * 128 * 2 = 131,072 bytes per token = 128 KB/token

# Total VRAM at runtime:
total_vram = model_vram + (context_len * batch_size * kv_bytes_per_token)

# Worked example: Llama 3.1 8B, fp16, ctx=8192, batch=4
model  = 8e9 * 2          # 16.0 GB
kvcache = 8192 * 4 * 0.000128  # 4.19 GB
total  = 20.19 GB         # fits in 24 GB with 3.8 GB headroom

That headroom number matters. vLLM and llama.cpp both need some VRAM for activation tensors, CUDA context overhead, and kernel workspace. In practice, leave 1-2 GB of buffer. If your math says 22 GB, you're in dangerous territory on a 24 GB card.

Notice the Llama 3.1 architecture uses Grouped Query Attention (GQA) with only 8 KV heads instead of 32. This is a deliberate design choice — Google Research showed that you can use far fewer K and V heads than Q heads with minimal quality loss, but the KV cache footprint shrinks dramatically. Older models like LLaMA-1 and GPT-style architectures used full Multi-Head Attention (MHA), where num_kv_heads = num_heads, making their KV cache 4x larger for the same context length. This is one reason Llama 3.x models are so much more deployment-friendly than their predecessors.

0.2 Prefill vs. Decode — Two Completely Different Problems

This is the most important conceptual distinction in LLM inference, and almost no introductory guide explains it clearly. LLM generation is not a single uniform computation — it has two distinct phases with completely different compute profiles, different bottlenecks, and different optimization strategies.

The prefill phase happens once, at the start of a request. The model processes all your input tokens simultaneously, building the attention pattern between every token and every other token. If your prompt is 2000 tokens long, the model runs a 2000 × 2000 attention computation (actually 2000 × 2000 × 32 heads). This is massively parallel and compute-bound — the GPU's 35.6 TFLOPS of FP16 compute is the limiting factor. Prefill is fast in terms of tokens processed: a 1000-token prompt typically takes 400-800ms on a 3090. It also populates the KV cache with a tensor for every input token, which is then stored in VRAM.

The decode phase is everything after that. The model generates tokens one at a time, autoregressively — each new token depends on all previous tokens, so you cannot parallelize across output positions. At each step, the new token attends to everything in the KV cache (all previously processed tokens). This is memory-bandwidth-bound, not compute-bound. For each decode step, the GPU reads the entire KV cache from VRAM into its compute units, does a relatively small computation (one row of attention), and writes the result. On a 3090, if your KV cache is 8 GB, you're reading 8 GB per token generated. At 936 GB/s bandwidth, that's roughly 8.5ms per token — about 117 tokens/sec theoretical maximum, and in practice 50-80 tokens/sec due to overhead.

This bandwidth bottleneck explains almost everything about LLM optimization:

  • Why quantization to int4 roughly doubles decode speed — the model is half the size, so you read half as much data per step.
  • Why batching multiple requests together improves throughput but not per-request latency — you amortize the weight reads across multiple sequences, getting more tokens per memory bandwidth dollar.
  • Why longer contexts slow down decode — more KV cache to read per step, even though the model weights haven't changed.
  • Why Flash Attention primarily helps with prefill speed and memory, not decode speed.
  • Why speculative decoding can help — it tries to generate multiple decode tokens per bandwidth read instead of one.
Mental model: Prefill is like reading a book — you process the whole thing in parallel. Decode is like typing a reply word by word, needing to re-read your entire draft before choosing each new word. The bottleneck shifts from reading speed (prefill) to how fast you can re-read the growing draft (decode bandwidth).

Part 1: Ollama — The Fastest Path to a Working Model

1.1 What Ollama Actually Is (and Is Not)

Ollama is not a new inference engine. It is a Go daemon that wraps llama.cpp under the hood, bolts on a model registry inspired by Docker Hub, and exposes an OpenAI-compatible REST API on port 11434. When you run ollama run llama3.1, the daemon fetches a pre-quantized GGUF file from Ollama's servers, verifies the SHA256 hash, loads it into llama.cpp with GPU offloading configured automatically, and starts a conversation loop. The whole model lifecycle — download, version management, VRAM fitting, layer offloading — is handled without you writing a single line of configuration.

That is Ollama's genuine value proposition. It removes every friction point between you and a running model. The tradeoff is that you get limited visibility and control over what's happening underneath. Ollama makes decisions about quantization, context length, and GPU offload automatically, and those decisions are often conservative. For serious throughput work, you'll outgrow Ollama quickly — but as a development tool, a quick demo server, or a personal assistant on your workstation, it's genuinely excellent.

One architectural detail worth knowing: Ollama keeps models loaded in VRAM even after a request finishes, for a configurable timeout (default 5 minutes). This means the second request to the same model is instant — no reload latency. But if you run two different large models, you can run into VRAM pressure. Ollama handles this by evicting the least-recently-used model, but that eviction shows up as a 10-30 second stall on the next request to that model. Use OLLAMA_MAX_LOADED_MODELS to control how many models stay resident.

1.2 Installation and Setup

Ollama ships as a single binary with systemd integration on Linux. The install script detects your CUDA version, places the binary, and registers a system service that starts on boot. You typically never need to manually start it again after install. 

# Install — detects CUDA automatically
curl -fsSL https://ollama.com/install.sh | sh

# The installer registers a systemd service, so it starts on boot.
# Verify it's running:
systemctl status ollama

# Check which port it's listening on and whether GPU was detected:
ollama serve &
curl http://localhost:11434/api/tags

# IMPORTANT: always verify GPU is being used, not CPU
# GPU process should appear here:
nvidia-smi

# If you see 0% GPU utilization during inference, something is wrong.
# Common cause: CUDA version mismatch between Ollama binary and your driver.

After install, Ollama creates a model store at ~/.ollama/models/. Each model is stored as a content-addressed blob (like Git objects), so the same weights shared between two quantizations are only stored once. The Modelfile manifest format specifies layers, templates, and parameters — if you've used Docker, this will feel familiar.

1.3 Choosing and Pulling Models

The Ollama library follows a naming scheme of model:variant. The default tag when you don't specify is the "best available" quantization that Ollama's maintainers chose — usually Q4_K_M for 7-8B models, which is a reasonable tradeoff. But you should be intentional about this choice rather than accepting the default blindly. Here's the practical tradeoff for the RTX 3090:

Q4_K_M is the workhorse. It uses 4-bit weights with a K-quant scheme that applies slightly higher precision to weights the model is most sensitive to. For Llama 3.1 8B, this produces a 4.8 GB file with quality benchmarks showing 97-98% of fp16 on most tasks. The decode speed is roughly 78-85 tokens/sec on a 3090, leaving almost 19 GB of free VRAM for KV cache. You can serve 8192-token contexts with batch=8 comfortably.

Q5_K_M is what you reach for when quality matters more than throughput or multi-user serving. At 5.5 GB and ~99% of fp16 quality, the quality loss becomes nearly imperceptible in blind tests. Decode speed drops to ~62 tokens/sec, but most users won't notice the difference between 62 and 80 tokens/sec — both feel fast.

fp16 (no quantization) makes sense when you need the model's full capability — long-form reasoning, complex code generation, or when you're building an evaluation pipeline and need reproducible baselines. The model takes the full 16 GB, decode speed is ~48 tokens/sec, and you have about 6 GB left for KV cache. 

# Pull specific quantizations — be deliberate about which one
ollama pull llama3.1:8b                    # default (Q4_K_M, 4.8 GB)
ollama pull llama3.1:8b-instruct-q5_K_M   # higher quality (5.5 GB)
ollama pull llama3.1:8b-instruct-fp16     # maximum quality (16 GB)

# Other strong models for a 3090:
ollama pull mistral:7b-instruct            # fast, great for code
ollama pull qwen2.5-coder:7b               # best open-source code model
ollama pull gemma2:9b                      # Google's best small model
ollama pull nomic-embed-text               # embeddings (274 MB, negligible)

# List what's loaded and how much VRAM each uses:
ollama ps

# Inspect a model's config, template, and parameters:
ollama show llama3.1:8b --modelfile

1.4 The Two APIs — Native vs. OpenAI-Compatible

Ollama exposes two distinct REST APIs on the same port. The native Ollama API at /api/ gives you access to Ollama-specific features: raw completions, embeddings, multi-modal inputs with base64-encoded images, and precise control over llama.cpp parameters like num_gpu (number of layers on GPU), num_thread (CPU threads), and mirostat sampling. Use this when you're building an Ollama-specific application.

The OpenAI-compatible API at /v1/ mirrors the OpenAI Chat Completions spec exactly. This is powerful because any library or tool built for the OpenAI API — LangChain, LlamaIndex, DSPy, Guidance, outlines — works against Ollama with a single base URL change. The model name is just the Ollama model identifier. This API is slightly more limited (no raw completions mode, no image support yet in this endpoint), but for text-only chat applications it's the standard choice. 

# Native Ollama API — more control, Ollama-specific features
curl http://localhost:11434/api/generate \
  -d '{
    "model": "llama3.1:8b",
    "prompt": "Explain transformer self-attention from first principles:",
    "stream": false,
    "options": {
      "num_predict": 1024,
      "temperature": 0.7,
      "top_p": 0.9,
      "top_k": 40,
      "num_ctx": 8192,
      "num_gpu": 99,
      "repeat_penalty": 1.1,
      "seed": 42
    }
  }'

# OpenAI-compatible API — works with any OpenAI-compatible tool
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama3.1:8b",
    "messages": [
      {"role": "system", "content": "You are an expert ML engineer."},
      {"role": "user", "content": "Explain KV cache in depth."}
    ],
    "temperature": 0.7,
    "max_tokens": 1024,
    "stream": true
  }'

The stream: true flag is worth highlighting. When streaming is enabled, the server sends Server-Sent Events (SSE) — a line per token as it's generated. This is what makes chat UIs feel responsive: the user sees the first token within ~100ms of sending the request, even if the full response takes 15 seconds. Always enable streaming in user-facing applications. Only disable it for batch processing or evaluation pipelines where you just need the final text.

1.5 Using Python with Ollama

The ollama Python package is a thin wrapper around the REST API. Alternatively, since Ollama's /v1/ endpoint is OpenAI-compatible, you can use the openai Python package directly with a custom base_url. The second approach is better for code that you might later want to run against the real OpenAI API or a vLLM server — you change one line and everything else works. 

from ollama import Client

client = Client(host='http://localhost:11434')

# Synchronous chat — blocks until complete
response = client.chat(
    model='llama3.1:8b',
    messages=[
        {'role': 'system', 'content': 'Answer with specific technical depth.'},
        {'role': 'user', 'content': 'What is the difference between GPTQ and AWQ quantization?'}
    ]
)
print(response['message']['content'])

# The response dict also contains timing metadata:
# response['eval_count']    = tokens generated
# response['eval_duration'] = time in nanoseconds
# response['prompt_eval_count'] = prompt tokens
tokens_per_sec = response['eval_count'] / (response['eval_duration'] / 1e9)
print(f"Speed: {tokens_per_sec:.1f} tokens/sec")

# Streaming — yields one chunk per token as it's generated
# This is how you build a responsive chat UI
import sys

for chunk in client.chat(
    model='llama3.1:8b',
    messages=[{'role': 'user', 'content': 'Explain Flash Attention step by step.'}],
    stream=True
):
    # Each chunk has 'message' with 'content' (might be empty on the last chunk)
    content = chunk['message']['content']
    if content:
        sys.stdout.write(content)
        sys.stdout.flush()
print()  # newline at the end

# Using the OpenAI SDK — the portable approach
# This code works unchanged against vLLM, OpenAI, or any compatible server.
from openai import OpenAI

client = OpenAI(
    base_url='http://localhost:11434/v1',
    api_key='ollama'   # Ollama ignores the key; must be a non-empty string
)

response = client.chat.completions.create(
    model='llama3.1:8b',
    messages=[
        {'role': 'system', 'content': 'You are an expert in GPU systems.'},
        {'role': 'user', 'content': 'Why is the decode phase memory-bandwidth-bound?'}
    ],
    max_tokens=512,
    temperature=0.5
)

print(response.choices[0].message.content)
print(f"Usage: {response.usage.prompt_tokens} in, {response.usage.completion_tokens} out")

1.6 Modelfiles — Baking in Persona and Defaults

A Modelfile is Ollama's version of a Dockerfile for LLM configurations. It lets you define a named model variant with a specific base model, system prompt, and default sampling parameters baked in. This is more than just convenience: when the system prompt is part of the Modelfile, Ollama can treat it as a persistent prefix and cache the KV computation for it across requests. A long system prompt (500+ tokens) that you repeat on every request would normally cost ~50ms of prefill time each time — baking it into a Modelfile eliminates that cost after the first call.

The sampling parameters in a Modelfile also serve as sane defaults that prevent callers from accidentally using bad settings. Setting temperature 0.1 in a code-generation Modelfile means the model defaults to deterministic output unless the caller explicitly overrides it. This is good API design — make the defaults right for the use case. 

# Create a specialized Modelfile for code review tasks
cat > Modelfile.code-reviewer << 'EOF'
FROM llama3.1:8b

# System prompt is cached across requests after the first call
SYSTEM """
You are a senior software engineer conducting thorough code reviews.
For every piece of code you review:
1. Identify correctness issues (bugs, edge cases, race conditions)
2. Note performance problems with specific explanations
3. Flag security vulnerabilities (SQL injection, XSS, auth bypass, etc.)
4. Suggest idiomatic improvements for the language in use
Always cite the line number and explain the reasoning behind each suggestion.
Be direct and specific — avoid vague feedback.
"""

# Conservative temperature for code tasks — more deterministic
PARAMETER temperature 0.1
PARAMETER top_p 0.9
# 8192 context = can handle large files
PARAMETER num_ctx 8192
# 4096 output = enough for thorough reviews
PARAMETER num_predict 4096
# Penalize repetition — models sometimes loop on code
PARAMETER repeat_penalty 1.15
EOF

# Build the custom model
ollama create code-reviewer -f Modelfile.code-reviewer

# Use it just like any model
ollama run code-reviewer "Review this Python function:
def find_user(db, user_id):
    query = f'SELECT * FROM users WHERE id = {user_id}'
    return db.execute(query).fetchone()"

# The model is available via API too
curl http://localhost:11434/v1/chat/completions \
  -d '{"model": "code-reviewer", "messages": [{"role": "user", "content": "def f(x): return x*2"}]}'

1.7 Configuring Ollama for Concurrent Use

Out of the box, Ollama processes requests sequentially — one request at a time per model. For a personal workstation this is fine. If you're serving multiple users or applications, you need to configure parallel processing. This is done via environment variables in the systemd service override, not through a config file.

The key variable is OLLAMA_NUM_PARALLEL, which sets how many requests the model processes simultaneously. Each parallel slot needs its own KV cache, so increasing this number directly increases VRAM usage. With Llama 3.1 8B Q4_K_M (4.8 GB model, 19 GB free on a 3090), each parallel slot at 4096 context costs about 2 GB of KV cache. You can safely run 4-6 parallel slots before hitting VRAM limits. Beyond that, you need to either reduce context length or switch to a smaller model. 

# Create systemd override directory
sudo mkdir -p /etc/systemd/system/ollama.service.d/

# Create the override file
sudo tee /etc/systemd/system/ollama.service.d/override.conf > /dev/null << 'EOF'
[Service]
# How many requests to process simultaneously (each needs its own KV cache)
Environment="OLLAMA_NUM_PARALLEL=4"

# How many different models to keep loaded in VRAM simultaneously
# If you have two 8B models loaded at once, you need ~10 GB just for weights
Environment="OLLAMA_MAX_LOADED_MODELS=2"

# Flash Attention — always enable; it reduces memory usage and speeds up long prompts
Environment="OLLAMA_FLASH_ATTENTION=1"

# Which GPU to use (0-indexed). Omit to use all GPUs.
Environment="CUDA_VISIBLE_DEVICES=0"

# How long to keep an idle model in VRAM (default: 5m)
# Set to 0 to unload immediately, -1 to never unload
Environment="OLLAMA_KEEP_ALIVE=10m"
EOF

# Apply changes
sudo systemctl daemon-reload
sudo systemctl restart ollama

# Verify settings took effect
systemctl show ollama | grep Environment

Part 2: vLLM — Production-Grade Serving Built on PagedAttention

2.1 The Problem vLLM Was Designed to Solve

To understand vLLM, you first need to understand the problem that existed before it. In 2022 and early 2023, if you wanted to serve an LLM to multiple concurrent users with frameworks like HuggingFace Transformers or FasterTransformer, you had a choice: either process one request at a time (simple but wasteful) or batch multiple requests together (faster but tricky). The tricky part was that requests have different lengths — both inputs and outputs. A request asking for a haiku finishes in 15 tokens. A request asking for a detailed essay might generate 2000 tokens. If you batch these together, the haiku finishes early and its GPU resources sit idle for 100x longer while the essay finishes. Worse, KV cache had to be pre-allocated for the maximum possible sequence length, wasting potentially gigabytes of VRAM on unused slots.

The Berkeley researchers who built vLLM solved this with PagedAttention, a system borrowed directly from how operating systems manage physical RAM with virtual memory. The insight is elegant: instead of allocating a contiguous block of VRAM for each sequence's KV cache upfront, store KV data in small fixed-size blocks (16 tokens per block by default) and maintain a mapping table from logical sequence positions to physical block locations. This is almost exactly how an OS maps virtual memory pages to physical RAM pages.

The benefits cascade from this one design decision. Because blocks are a fixed small size, there is no internal fragmentation — you never waste VRAM. Because blocks are non-contiguous, you can allocate and free them one at a time as sequences grow and finish. And because two sequences can map to the same physical block (read-only), sequences that share a common prefix (like the same system prompt) can literally share their KV cache data in VRAM. One copy of the system prompt's KV data serves all requests.

vLLM's original paper showed 24x higher throughput than naive HuggingFace generation on a single A100. On an RTX 3090 with realistic workloads (mixed prompt lengths, concurrent users), the improvement is typically 8-15x compared to sequential serving — more than enough to justify the additional setup complexity.

2.2 Continuous Batching — Never Waiting for the Slowest Request

PagedAttention solves the memory problem. Continuous batching solves the scheduling problem. In traditional batched serving, you collect a fixed batch of requests, process them all together until every sequence finishes, then start the next batch. The problem: if one request generates 2000 tokens and the other 31 in your batch finish at 100 tokens, those 31 requests' GPU slots are idle for 1900 decode steps. You're paying for 32 GPUs and using 1 of them for most of the time.

Continuous batching (also called iteration-level scheduling) solves this by making the batch composition dynamic. After every single decode step — every single forward pass — vLLM's scheduler checks which sequences have finished (hit an EOS token or max length). Finished sequences are immediately evicted from the batch and their KV cache blocks are freed. Waiting prefill requests immediately fill those slots. The GPU is always processing the maximum number of sequences it can handle, every single step.

This is why vLLM's throughput scales so well with concurrency. At batch=1, you get ~48 tokens/sec for Llama 3.1 8B fp16 on a 3090. At batch=8 with continuous batching, you get ~320 tokens/sec — eight requests each progressing at about 40 tokens/sec simultaneously. The total throughput scales nearly linearly with batch size up to the point where VRAM for KV cache or the compute budget runs out.

2.4 Starting the Server — The Right Way

vLLM's server is an asyncio FastAPI application. The key configuration decisions happen at startup — unlike Ollama, you can't hot-change most settings without restarting the process. Think through your configuration before starting.

The two most important flags are --gpu-memory-utilization and --max-model-len. The utilization flag tells vLLM what fraction of GPU VRAM to use for KV cache after loading model weights. It defaults to 0.90, meaning vLLM will use 90% of available VRAM for KV cache blocks. Setting it too high risks CUDA OOM on large batches; setting it too low wastes throughput potential. For a 3090 running Llama 3.1 8B fp16 (16 GB weights), the remaining ~8 GB at 0.90 utilization gives you roughly 8 GB / (128 KB/token) = ~62,000 token-slots across all concurrent requests. With max_seq_len=8192, that's about 7-8 concurrent sequences before vLLM starts preempting (swapping to CPU RAM). 

# Start vLLM server — production-ready configuration
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-8B-Instruct \
  --dtype float16 \
  --gpu-memory-utilization 0.90 \
  --max-model-len 8192 \
  --max-num-seqs 256 \
  --enable-prefix-caching \
  --port 8000 \
  --host 0.0.0.0

# Explanation of each flag:
#
# --dtype float16
#   Use half-precision weights. bf16 is better for A100/H100 (more numerically stable),
#   but RTX 3090 has good fp16 support. Never use fp32 for inference — no benefit, 2x memory.
#
# --gpu-memory-utilization 0.90
#   After loading weights, reserve 90% of remaining VRAM for KV cache blocks.
#   Leave 10% as buffer for activations and CUDA overhead.
#   Start at 0.85 if you see OOM errors, bump up to 0.92 for maximum throughput.
#
# --max-model-len 8192
#   Maximum sequence length (input + output tokens combined).
#   Longer = more VRAM for KV cache per sequence. Start conservative.
#   For Llama 3.1 which supports 128K natively, you need much more VRAM to use it.
#
# --max-num-seqs 256
#   Maximum number of sequences that can be in flight simultaneously.
#   vLLM's scheduler queues beyond this limit. 256 is a safe default;
#   increase for high-concurrency deployments.
#
# --enable-prefix-caching
#   Cache KV data for common prefixes (system prompts).
#   FREE speedup if you use the same system prompt across requests.
#   Has essentially zero downside — always enable it.

# With AWQ quantized model (fits larger models in VRAM)
python -m vllm.entrypoints.openai.api_server \
  --model casperhansen/llama-3-8b-instruct-awq \
  --quantization awq \
  --dtype half \
  --gpu-memory-utilization 0.90 \
  --max-model-len 16384 \
  --enable-prefix-caching \
  --port 8000

2.5 Offline Batch Inference vs. Online Serving

vLLM supports two distinct usage patterns. The online server is what we've been discussing — a long-running HTTP server that accepts requests over the network. Offline inference loads the model directly in Python, feeds it a list of prompts, and gets back results — no HTTP overhead, no network, maximum throughput for batch workloads. If you're running evaluations, generating synthetic training data, or processing a dataset of 10,000 documents, offline inference is what you want.

The key advantage of offline inference is that vLLM can see the entire batch of requests upfront and make optimal scheduling decisions — it knows which prompts are long and which are short, can group them intelligently, and can tune the compute graph accordingly. With online serving, the scheduler only knows about requests that have already arrived. 

from vllm import LLM, SamplingParams
import time

# Load model once — expensive, takes 20-40 seconds
llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",
    dtype="float16",
    gpu_memory_utilization=0.90,

2.6 Prefix Caching — The Free Throughput Win

Prefix caching is one of those optimizations that delivers significant wins with zero quality impact and almost no engineering work. Here's the core idea: if you're building a chatbot or an API service, every single request starts with the same system prompt — maybe 500-1000 tokens explaining who the model should be, what it should and shouldn't say, and how to format responses. Without prefix caching, every request has to process all those tokens through prefill. With prefix caching, vLLM computes a hash of the input prefix, stores the resulting KV cache blocks, and reuses them on cache hits.

The time saved scales directly with system prompt length. A 1000-token system prompt on a 3090 takes about 100ms to prefill. With prefix caching, after the first request, subsequent requests skip that 100ms entirely. At 100 requests/minute, you're saving 10 seconds of compute per minute — 16% of your total compute budget — just by caching a prompt you were going to repeat anyway. 

from vllm import LLM, SamplingParams
import time

# Enable prefix caching at construction time
llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",
    dtype="float16",
    enable_prefix_caching=True,
)

# Simulate a realistic system prompt — long enough to matter
SYSTEM_PROMPT = """

- Always explain the underlying mechanism, not just the surface behavior
- Give concrete performance numbers when relevant (latency, throughput, memory)
- Identify common pitfalls and how to avoid them
- Prefer simple solutions over clever ones
- Point out when a question has hidden complexity
""" * 8  # Make it long to see the caching effect clearly

def make_prompt(question: str) -> str:
    return f"<|system|>{SYSTEM_PROMPT}<|user|>{question}<|assistant|>"

questions = [
    "Why is KV cache the bottleneck in LLM serving?",

Part 3: llama.cpp — Bare Metal, Maximum Flexibility

3.1 Why llama.cpp Still Matters

When Georgi Gerganov published llama.cpp in March 2023, it was remarkable for one reason: it ran a 7B parameter LLaMA model on a MacBook CPU, in pure C with no Python, no CUDA, no GPU at all. The trick was aggressive quantization (4-bit integer), hand-written SIMD kernels for Apple Silicon and x86, and meticulous memory management. It was slow by today's standards, but it worked, and it worked anywhere.

Three years later, llama.cpp has grown into something more sophisticated: a full inference library with CUDA, Metal (Apple), Vulkan, ROCm (AMD), and SYCL backends, custom kernels for every quantization format, Flash Attention support, speculative decoding, and an HTTP server mode. But it retains its original character: it is a C/C++ library with no runtime dependencies, no Python overhead, and no framework abstractions between you and the hardware.

This makes llama.cpp the right choice in several situations: you need to embed inference in a C++ application; you need to run a model that's too large for your VRAM and want transparent CPU offloading; you're targeting edge devices, ARM servers, or environments without CUDA; you need the most memory-efficient inference possible; or you want to understand inference at the lowest practical level. For multi-user production serving, vLLM is usually better — but llama.cpp's single-user performance is excellent and its flexibility is unmatched.

3.2 The GGUF Format — One File, Everything Included

llama.cpp introduced GGUF (GGML Unified Format) to replace the fragmented collection of files that HuggingFace models use. A GGUF file is a single binary container that includes model weights in the chosen quantization format, the tokenizer (vocabulary, merge rules, special tokens), model architecture metadata (number of layers, heads, dimensions), and any other parameters needed to load and run the model.

This self-contained design is a significant operational advantage. Deploying a GGUF model means copying one file. There are no tokenizer configs to keep in sync with weight files, no config.json that might not match the weights, no special_tokens_map.json that might be missing. The file either works or it doesn't. For reproducibility and distribution, GGUF is much better than the HuggingFace multi-file format.

The naming convention for GGUF quantizations encodes the precision: Q4_K_M means 4-bit K-quant, medium variant. K-quants use 256-element super-blocks where most weights are quantized to 4-bit, but a small fraction (the most important ones, determined by their magnitude) get 6-bit precision. The S/M/L suffix controls what fraction gets the higher precision treatment — M (medium) is the standard choice offering the best quality-per-byte tradeoff.

Part 4: Quantization — The Deep Dive

4.1 Why Quantization Works (and When It Doesn't)

The intuition behind neural network quantization is that neural network weights are highly redundant. A trained model has billions of weights, but the information those weights collectively encode is much less than their raw bit count suggests. Many weights cluster tightly in small ranges, many are near zero and unimportant, and the model can tolerate small errors in individual weights because the errors cancel statistically across large dot products. This redundancy is what quantization exploits.

The key question is how much information you can strip out before the model's behavior noticeably degrades. fp16→int8 (halving bit depth) is almost universally safe — benchmarks consistently show less than 1% quality loss on standard tasks. fp16→int4 (quartering bit depth) starts to matter: you see 2-5% degradation on knowledge-intensive tasks, more on math and code. Going below 4-bit (int3, int2) causes noticeable quality degradation in most models — useful only for extreme memory constraints.

There is also a crucial architectural consideration: not all layers are equally sensitive to quantization. The first and last transformer layers (the embedding and unembedding layers) are particularly sensitive — quantizing them causes disproportionate quality loss. Good quantization methods (K-quants, AWQ) identify and protect these sensitive layers. Naive uniform quantization does not, which is one reason naive int4 quantization is worse than AWQ or GPTQ at the same bit-width.

4.2 GPTQ — Post-Training Quantization with Hessian-Based Calibration

GPTQ (from the 2022 paper "GPTQ: Accurate Post-Training Quantization for Generative Pre-trained Transformers") was the first method to make 4-bit quantization of large language models practical without significant quality loss. The core innovation is using second-order information (the Hessian of the layer's output with respect to its weights) to determine which weights are most important and compensate for quantization errors.

Concretely: GPTQ processes one linear layer at a time, using a small calibration dataset (typically 128 random text samples from Wikipedia or code). For each layer, it quantizes weights in groups of 128, and after quantizing each group, it adjusts the remaining unquantized weights to compensate for the error introduced so far. This sequential error-correction is computationally expensive (takes 1-4 hours for a 7B model on a GPU) but produces much better results than naive rounding. 

from transformers import AutoTokenizer
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from datasets import load_dataset

model_name = "meta-llama/Llama-3.1-8B-Instruct"

# Configure the quantization

    bits=4,               # 4-bit target precision

                          # smaller group = better quality, more overhead

    damp_percent=0.1,     # damping factor for Hessian computation stability

4.3 AWQ — Activation-Aware Weight Quantization

AWQ (Lin et al., 2023) addresses a key weakness of GPTQ: it doesn't distinguish between weights that matter a lot and weights that don't. The AWQ paper made an important observation: in LLMs, roughly 1% of weights are responsible for a disproportionate fraction of the model's output quality. These "salient" weights correspond to channels where the input activations have high magnitude — in other words, they're the channels the model actually relies on heavily during inference.

AWQ's approach: instead of compensating for quantization errors after the fact (like GPTQ), pre-scale the weight channels before quantization to give the important ones more effective dynamic range. Since scaling is equivalent to transforming the activation distribution, this is mathematically lossless — you scale the weights up and the activations down by the same factor, so the output is identical. After scaling, quantizing with the same number of bits gives the important channels better precision at the cost of minor precision loss on the channels that matter less.

The result: AWQ consistently outperforms GPTQ at the same bit-width, especially on reasoning and math tasks where precision matters more. It's also faster to run — the GEMM-style AWQ kernels are highly optimized and often 10-20% faster than GPTQ kernels at the same quantization level. 

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

Part 5: Flash Attention — The Algorithmic Breakthrough

5.1 The Problem with Standard Attention

The attention mechanism in transformers computes, for each query token, how much to "attend to" each key token by computing dot products, applying softmax, and using the resulting weights to aggregate value vectors. In mathematical notation: Attention(Q, K, V) = softmax(QK^T / √d_k) · V. This formula hides a severe memory problem: the matrix QK^T is size N×N where N is the sequence length. For a 32K-token sequence with 32 attention heads, each of size N×N in fp16, that intermediate matrix is 32,768 × 32,768 × 32 × 2 bytes = 64 GB. You can't even fit a single attention computation in VRAM, let alone process multiple sequences.

The standard "fix" before Flash Attention was to limit context length. Want 128K context? Standard attention needs 128K × 128K = 16 billion elements just for the attention matrix. At fp16, that's 32 GB for one layer's attention matrix. With 32 layers, it's 1 TB. It's simply impossible.

5.2 Flash Attention — Tiling the Computation

Tri Dao's Flash Attention (2022) solves this by reorganizing the computation so the full N×N matrix is never materialized in GPU HBM (high-bandwidth memory). Instead, the algorithm divides the Q, K, and V matrices into tiles small enough to fit in SRAM (the GPU's on-chip memory, which is 10-30x faster than HBM but only 48-96 KB per streaming multiprocessor). It then processes tiles in sequence, maintaining running statistics for the softmax normalization (specifically: the running maximum value and the running sum, using the log-sum-exp trick to stay numerically stable without seeing the entire row).

The result is mathematically identical to standard attention — not an approximation — but the memory complexity drops from O(N²) to O(N). The entire attention computation for any sequence length can be performed using only a fixed amount of SRAM, regardless of how long the sequence is. HBM reads are reduced from O(N²) to O(N), which is the source of the speedup — the computation itself is the same, but you spend far less time waiting for data to move between HBM and compute units.

Flash Attention 2 (2023) improved on this by better parallelizing across attention heads and reducing non-matmul FLOPs. For a 3090 running Llama 3.1 8B:

  • Prefill with a 2K-token prompt: standard attention ~300ms → Flash Attention ~120ms (2.5x faster)
  • Prefill with a 8K-token prompt: standard attention ~4.8s → Flash Attention ~480ms (10x faster, because N² vs N scaling)
  • Decode with 8K context: minimal change — decode is bandwidth-bound, not compute-bound
  • Peak VRAM reduction: standard attention needs ~8 GB for 8K context attention matrix → Flash Attention needs ~0

Part 6: Speculative Decoding

6.1 The Bottleneck It Solves

Speculative decoding targets the fundamental inefficiency of autoregressive decode: because each token depends on all previous tokens, you cannot generate token 5 until token 4 is done. You're stuck generating one token per forward pass of the full model. On a 3090, one forward pass of Llama 3.1 8B takes about 20ms for a single sequence. You get ~50 tokens/sec — and there's no obvious way to parallelize further without changing the algorithm.

Speculative decoding changes the algorithm. The key observation: verifying a sequence of tokens is much cheaper than generating them, because verification is parallelizable. If I give you five candidate tokens, you can check all five in a single forward pass of the full model — the model scores them all simultaneously, like a parallel scoring step. This is the core idea: use a cheap small model to propose a sequence of tokens, then use one forward pass of the expensive large model to accept or reject them.

The mathematical guarantee (from the original DeepMind paper) is that the token distribution produced by speculative decoding is identical to the distribution of the target model. This is not an approximation — it's an exact equivalence, achieved through rejection sampling. When the large model disagrees with a draft token, it rejects it and samples a corrected token instead. So you get the exact same output distribution as running the large model alone, but potentially 2-5x faster.

The speedup depends on the acceptance rate — how often the large model agrees with the draft. For predictable text (code with consistent patterns, formal writing, repetitive outputs), acceptance rates hit 80-90% and you get nearly num_speculative_tokens tokens per step instead of one. For highly creative or unpredictable outputs, acceptance rates drop to 40-50% and the speedup shrinks or disappears. There's also overhead from running the draft model, so if acceptance rates are too low, speculative decoding can actually be slower than standard decode. 

from vllm import LLM, SamplingParams

Part 7: Challenges and How to Actually Debug Them

7.1 VRAM Out-of-Memory Errors

OOM is the most common error when deploying LLMs. It usually happens in one of three patterns: the model doesn't fit at all (wrong size estimate), the model fits but the KV cache overflows (too many concurrent requests or too long context), or everything fits at startup but a large request triggers an OOM later (the KV cache grew to fill VRAM and a large batch pushed it over). Each pattern requires a different fix.

The diagnostic step is always the same: check nvidia-smi and compare actual VRAM usage against your theoretical estimate. If the model itself is unexpectedly large, check whether it loaded in fp32 instead of fp16 (this doubles memory usage and is a common mistake). If the model size is correct but OOM still happens, the KV cache is the culprit — you need to reduce context length, batch size, or switch to KV cache quantization.

7.2 Slow Time-to-First-Token

TTFT (time to first token) is what users feel as "the pause before the model starts answering." It's almost entirely determined by prefill time, which scales with the number of input tokens. A system prompt + a question that totals 3000 tokens will feel noticeably slower to start than one that totals 300 tokens, even if the final answer is the same length. High TTFT is particularly punishing for user-facing applications because users stare at a blank screen wondering if anything is happening.

The fixes fall into two categories: structural (change what you send to the model) and technical (change how the model processes it). Prefix caching is the most impactful technical fix — if your system prompt is long and consistent, caching its KV data reduces TTFT from "full prefill time" to essentially zero for that portion. Flash Attention is the second lever — it reduces prefill time roughly 2-5x for medium contexts and up to 10x for long ones. If neither helps enough, you may need to compress or shorten the input using techniques like LLMLingua before sending it to the model.

7.3 Low Tokens-Per-Second During Generation

If decode speed is lower than expected, the diagnostic hierarchy is: first check whether inference is actually happening on GPU (not CPU), then check whether the context length or batch size is causing excessive KV cache reads, then check whether quantization could help, and finally consider whether you're running a model that's too large for the memory bandwidth to sustain the desired speed.

A common mistake is running with GPU offload partially active without realizing it. If a layer is on CPU, the data has to transfer over PCIe between each layer's forward pass — and PCIe bandwidth is about 16 GB/s, compared to VRAM's 936 GB/s. Even one layer on CPU can drag performance down significantly. Always verify with nvidia-smi that GPU utilization is high during generation.

7.4 Hallucinations and Quality Problems After Quantization

Quantization-induced quality degradation is real but often overstated. The key is measuring it empirically rather than assuming it's acceptable or catastrophic. Perplexity on a held-out dataset correlates with output quality — a 1-2% perplexity increase is generally invisible to users, while a 5%+ increase starts to produce noticeably worse outputs on reasoning and knowledge tasks.

Sampling parameters also matter enormously for perceived quality. A model at temperature 1.0 will produce noticeably more incoherent outputs than the same model at temperature 0.3, regardless of quantization. Most people's "this model seems bad" experience is actually a misconfigured temperature. For factual question answering and code generation, start at temperature 0.0-0.2. Increase only for creative tasks. 

# Measure quality degradation from quantization
from llama_cpp import Llama
import math

def perplexity(llm, text):
    """Compute perplexity of text under the model."""
    tokens = llm.tokenize(text.encode())
    log_probs = []
    for i in range(1, len(tokens)):
        # Get log probability of each token given its context
        context = tokens[:i]
        logits = llm.eval(context)
        token_logit = logits[tokens[i]]
        # Convert logit to log-probability (softmax + log)
        # Simplified: use llm.scores for exact log probs
        log_probs.append(token_logit)
    return math.exp(-sum(log_probs) / len(log_probs))

#
# Temperature: controls randomness. Lower = more deterministic.
#   - 0.0: always pick the highest-probability token (greedy)
#   - 0.1-0.3: very focused, good for factual tasks and code
#   - 0.7: balanced, good default for general chat
#   - 1.0+: more creative, but more likely to produce nonsense
#
# Top-p (nucleus sampling): at each step, only consider tokens
# whose cumulative probability exceeds p.
#   - 0.9: standard — ignores the long tail of low-probability tokens
#   - 0.95: slightly more creative
#   - 1.0: no filtering (all tokens considered)
#
# Top-k: at each step, only consider the k highest-probability tokens.
#   - 40: standard default — ignores most of the vocabulary
#   - 0: disabled (don't use alone without top-p)
#
# min-p: a newer alternative to top-k that scales with the top token's probability.
#   - 0.05: ignore tokens with probability < 5% of the top token's probability
#   - Often produces better results than top-k at the same apparent temperature

Part 8: Advanced Topics

8.1 LoRA — Fine-Tuning Without Touching Base Weights

LoRA (Low-Rank Adaptation) solves a genuinely hard problem: you have a 16 GB model that you want to fine-tune for a specific task, but full fine-tuning requires gradients and optimizer states that would consume 3-5x the model size in GPU memory — roughly 60-80 GB for an 8B parameter model. That doesn't fit anywhere.

The LoRA insight: for fine-tuning on a specific task, the change in weights from the base model to the fine-tuned model lies in a low-rank subspace. Instead of updating the full weight matrix W (dimension d×d), you add a low-rank update: W_new = W + BA, where B is d×r and A is r×d, with r much smaller than d (typically 4, 8, or 16). For d=4096 and r=16, the update has 4096×16 + 16×4096 = 131,072 parameters instead of 4096×4096 = 16.7 million. That's 128x fewer trainable parameters.

The practical implications are significant. You can fine-tune an 8B model on a single 3090 in fp16 with LoRA (about 800 MB of adapter parameters instead of 16 GB). Multiple adapters can be stored cheaply and swapped at serving time. vLLM's LoRA serving lets you host one base model and serve dozens of task-specific adapters, swapping them per-request with no reload latency.

8.2 Structured Output — Guaranteeing Valid JSON

One of the most reliable ways to make LLMs work in production pipelines is to force structured output. Rather than hoping the model outputs valid JSON and then parsing it with error handling and retry logic, constrained decoding ensures the output always matches a schema — mathematically. The model's output distribution is modified at each decode step to zero out the probability of tokens that would violate the schema, then renormalize. It's not post-processing; it's in-the-loop constraint enforcement.

vLLM supports guided decoding natively, and outlines provides a higher-level interface that works with multiple backends. The key insight is that this comes at essentially zero quality cost — you're not restricting what the model thinks, just which syntactic forms it can use to express it. A JSON-constrained model will still choose incorrect field values if it doesn't know the right answer; it just won't produce malformed JSON.

8.3 QLoRA — Fine-Tuning 70B Models on a Single GPU

Standard LoRA requires the base model to be in fp16 (or bf16) during training. For Llama 3.1 8B, that's 16 GB just for the frozen base weights, plus gradients for the adapter, plus optimizer state. The total comes to roughly 20-22 GB — which fits on a 3090, but barely, with no headroom for a reasonable batch size. For a 13B or 70B model, it's simply impossible on a single consumer GPU.

QLoRA (Dettmers et al., 2023) solves this by combining two ideas: load the base model in 4-bit NF4 (Normal Float 4) quantization, and apply LoRA adapters on top. The base weights consume only a quarter of their fp16 footprint, but training still works because all computation happens in fp16 — QLoRA introduces double quantization (quantizing the quantization constants themselves) and paged optimizers (offloading optimizer states to CPU RAM when VRAM is tight) to keep gradient computation numerically stable even though the base weights are 4-bit.

The practical consequence is dramatic: QLoRA enables fine-tuning a 65B parameter model on a single 48 GB GPU, or a 13B model on a single 24 GB RTX 3090. For our purposes — 8B model on a 3090 — QLoRA is overkill (regular LoRA fits fine), but for anyone trying to fine-tune a 13B or 70B model without a multi-GPU setup, QLoRA is the only viable path.

The key architectural concept behind NF4 is worth understanding. Most quantization schemes (int4, int8) divide the weight range into equal intervals. NF4 instead uses intervals derived from the quantiles of a standard normal distribution — the assumption being that neural network weights approximately follow a bell curve centered at zero. By placing quantization levels where weight values are most dense (near zero), NF4 minimizes quantization error for the majority of weights. The result is that NF4 quality matches int8 in practice, despite using only 4 bits.

8.4 Multi-GPU Inference — Tensor, Pipeline, and Data Parallelism

When a model doesn't fit on a single GPU, or when you need more throughput than one GPU can provide, you need to distribute computation across multiple devices. There are three fundamentally different ways to do this, and they have radically different performance characteristics, bandwidth requirements, and use cases. Choosing the wrong strategy is a common and expensive mistake.

Tensor Parallelism (TP) splits each individual weight matrix across multiple GPUs. For an attention layer with weight matrices of size [d_model × d_model], tensor parallelism with TP=4 gives each GPU a [d_model × d_model/4] shard. All GPUs process every token, but each only computes a fraction of each layer. After each layer, they exchange partial results via an AllReduce operation. This requires very fast interconnect between GPUs — NVLink (400-600 GB/s per direction on A100/H100) handles this well; PCIe (64 GB/s) introduces a bottleneck that can negate the parallelism benefit. On a multi-3090 system connected via PCIe, tensor parallelism TP=2 is often the practical limit before interconnect becomes the bottleneck.

Pipeline Parallelism (PP) assigns different layers to different GPUs. With PP=4 and a 32-layer model, GPU 0 runs layers 0-7, GPU 1 runs layers 8-15, GPU 2 runs layers 16-23, and GPU 3 runs layers 24-31. Data flows through the GPUs like an assembly line. This requires much less inter-GPU bandwidth than tensor parallelism — only the activations between stage boundaries need to transfer, not all-reduce results. The downside is the "pipeline bubble": at any given moment, most GPUs are idle waiting for the previous stage to finish. Microbatching (splitting one batch into many small micro-batches) fills this bubble, but adds latency. Pipeline parallelism is best for extremely large models (100B+) where the pipeline bubble is acceptable, or for PCIe-connected GPU configurations.

Data Parallelism (DP) runs a complete copy of the model on each GPU and distributes different requests across them. GPU 0 handles user A, GPU 1 handles user B. No inter-GPU communication during inference (except optionally for weight synchronization if fine-tuning). This requires each GPU to have enough VRAM to hold the full model, so it doesn't help with VRAM constraints — only with throughput. For serving multiple concurrent users with a model that fits on one GPU, this is the right choice: run separate vLLM instances on each GPU and load-balance at the nginx layer.

8.5 RoPE Scaling — Extending Context Beyond Training Length

Llama 3.1 was trained with a 128K context window. But many models, including fine-tuned variants and older Llama-family models, were trained on shorter contexts (4096 or 8192 tokens). What happens if you try to use them with longer inputs? The answer involves understanding Rotary Position Embeddings (RoPE), the position encoding scheme used by most modern LLMs.

RoPE encodes position by rotating the query and key vectors in attention by an angle that depends on the token's position. The rotation frequencies are fixed at training time: high-frequency rotations for short-range dependencies, low-frequency rotations for long-range ones. When you feed a model a sequence longer than it was trained on, the low-frequency components rotate further than the model has ever seen — the position encoding lands in an "extrapolation" regime where the model has no training signal. Attention scores become unreliable, and the model produces garbage.

Several techniques address this without retraining the model from scratch:

Linear RoPE scaling (Position Interpolation): instead of letting positions go from 0 to N where N > training_max, scale them down by a factor of N/training_max. A token at position 8000 in a model trained to 4096 tokens is mapped to position 4000. This keeps all positions within the trained range. Quality degrades for tasks that require understanding very long-range dependencies, but the model doesn't catastrophically fail. This is what --rope-scaling-factor does in llama.cpp.

YaRN (Yet another RoPE extensioN): a more sophisticated approach that interpolates only the frequency components that need extending, leaving short-range (high-frequency) components unchanged. Short-range attention continues to work exactly as trained. Long-range attention is interpolated to cover the extended range. YaRN also applies a temperature correction to the attention scores to compensate for the change in softmax behavior. The result: much better quality at long contexts than simple linear scaling, at the cost of slightly more complex implementation.

8.6 RAG — Retrieval-Augmented Generation with Local LLMs

Retrieval-Augmented Generation (RAG) is the practical solution to LLMs' two biggest weaknesses: knowledge cutoffs (the model doesn't know anything after its training date) and hallucination on domain-specific facts (the model confabulates when it lacks confidence). The concept is simple: before calling the LLM, retrieve relevant documents from a vector database, inject them into the context window, and instruct the model to answer using only the provided context. If the context doesn't contain the answer, the model should say so rather than guess.

Building RAG on local LLMs has a significant advantage over cloud-based RAG: your documents never leave your machine. This is critical for medical records, financial data, legal documents, or proprietary code. The local embedding model (which converts your documents to vectors) and the local LLM (which generates answers) both run entirely on your hardware.

The RAG pipeline has two distinct phases. The indexing phase (offline, run once per document corpus): chunk your documents, embed each chunk with an embedding model, and store the (chunk, embedding) pairs in a vector database. The retrieval phase (online, per-request): embed the user's query, find the k most similar chunks using cosine similarity, and include them in the LLM prompt. The quality of RAG depends heavily on the quality of chunking (too small = no context, too large = irrelevant noise) and the embedding model (must capture semantic similarity, not just keyword overlap).

def chunk_document(text: str, chunk_size: int = 512, overlap: int = 64) -> list[str]:
    """
    Split document into overlapping chunks.

    Why overlap? When you split "...sentence A. Sentence B..." at a boundary,
    the chunk containing sentence A loses its connection to sentence B.
    Overlap ensures no sentence is orphaned — its neighbors appear in
    multiple chunks, preserving continuity across boundaries.

    chunk_size=512 tokens (roughly 400 words) is a sweet spot:
    - Smaller chunks (128-256): more precise retrieval, but each chunk lacks context
    - Larger chunks (1024+): more context, but vector embeds the "average" meaning,
      diluting specific facts — a 2000-word chunk about Python might match a query
      about JavaScript just because both discuss programming.
    """
    words = text.split()
    chunks = []
    step = chunk_size - overlap
    for i in range(0, len(words), step):
        chunk = " ".join(words[i:i + chunk_size])
        if len(chunk.split()) >= 50:    # skip tiny fragments
            chunks.append(chunk)
    return chunks

def index_documents(document_paths: list[str]):
    """Index a list of document files into ChromaDB."""
    all_chunks = []
    all_ids = []
    all_metadata = []

    for doc_path in document_paths:
        text = Path(doc_path).read_text(encoding="utf-8", errors="ignore")
        chunks = chunk_document(text)

        for i, chunk in enumerate(chunks):
            all_chunks.append(chunk)
            all_ids.append(f"{doc_path}::{i}")
            all_metadata.append({"source": doc_path, "chunk_index": i})

    # Batch embed all chunks — much faster than one at a time
    # nomic-embed-text requires a task prefix for the type of text
    prefixed = [f"search_document: {chunk}" for chunk in all_chunks]
    embeddings = embedding_model.encode(
        prefixed,
        batch_size=64,          # adjust based on GPU VRAM
        show_progress_bar=True,
        normalize_embeddings=True,
    )

    # Store in ChromaDB — this is persistent, survives restarts
    collection.add(
        documents=all_chunks,
        embeddings=embeddings.tolist(),
        ids=all_ids,
        metadatas=all_metadata,
    )
    print(f"Indexed {len(all_chunks)} chunks from {len(document_paths)} documents")

# ---- PHASE 2: RETRIEVAL AND GENERATION ----
# Run this for every user query.

def retrieve(query: str, top_k: int = 5) -> list[dict]:
    """
    Find the most relevant document chunks for a query.

    Uses cosine similarity between the query embedding and all stored chunk embeddings.
    ChromaDB's HNSW index makes this O(log n) even with millions of chunks.
    """
    # Prefix with "search_query" for query embeddings (asymmetric retrieval)
    query_embedding = embedding_model.encode(
        f"search_query: {query}",
        normalize_embeddings=True
    )

    results = collection.query(
        query_embeddings=[query_embedding.tolist()],
        n_results=top_k,
        include=["documents", "metadatas", "distances"]
    )

    chunks = []
    for doc, meta, dist in zip(
        results["documents"][0],
        results["metadatas"][0],
        results["distances"][0]
    ):
        chunks.append({
            "text": doc,
            "source": meta["source"],
            "similarity": 1 - dist   # ChromaDB returns distance; convert to similarity
        })

    return chunks

def rag_query(question: str, min_similarity: float = 0.3) -> str:
    """
    Full RAG pipeline: retrieve relevant chunks, then generate an answer.

    min_similarity threshold filters out chunks that aren't relevant enough —
    better to tell the user "I don't know" than to hallucinate from irrelevant context.
    """
    # Retrieve top-5 chunks
    chunks = retrieve(question, top_k=5)

    # Filter by relevance threshold
    relevant_chunks = [c for c in chunks if c["similarity"] >= min_similarity]

    if not relevant_chunks:
        return "I could not find relevant information in the knowledge base to answer this question."

    # Build the context block for the prompt
    # Include source attribution so users can verify
    context_parts = []
    for i, chunk in enumerate(relevant_chunks, 1):
        context_parts.append(
            f"[Source {i}: {chunk['source']} (relevance: {chunk['similarity']:.2f})]\n{chunk['text']}"
        )
    context = "\n\n---\n\n".join(context_parts)

    # Prompt engineering is critical for RAG quality.
    # Key instructions:
    # 1. Explicitly tell the model to use ONLY the provided context
    # 2. Tell it to say "I don't know" rather than hallucinate
    # 3. Ask it to cite sources — this forces it to ground answers in the retrieved text
    prompt = f"""You are a precise assistant that answers questions using only the provided context.
If the context does not contain enough information to answer the question, say so explicitly.
Do not use any knowledge outside of the provided context.

CONTEXT:
{context}

QUESTION: {question}

ANSWER (cite specific sources from the context):"""

    # Call local LLM via OpenAI-compatible API (Ollama or vLLM)
    response = requests.post(
        "http://localhost:11434/v1/chat/completions",
        json={
            "model": "llama3.1:8b",
            "messages": [{"role": "user", "content": prompt}],
            "temperature": 0.1,      # low temperature = more faithful to context
            "max_tokens": 1024,
        }
    )
    return response.json()["choices"][0]["message"]["content"]

There are several RAG failure modes worth knowing. Semantic drift: the embedding model embeds the query in a different semantic space than the documents (more common when mixing domains). Fix: use a domain-specific embedding model or fine-tune on your data. Chunk boundary hallucination: the answer to the question spans two adjacent chunks, neither of which alone contains the full answer. Fix: use larger chunks or hierarchical retrieval (retrieve at multiple granularities). Retrieval-faithfulness gap: you retrieved the right chunks but the LLM still ignores them and answers from training. Fix: stronger prompt instructions, lower temperature, or use a model that's been specifically fine-tuned for RAG (citation-following).

8.7 Embeddings — The Foundation of Semantic Search

Embeddings are dense vector representations of text where semantic similarity corresponds to geometric proximity. A sentence about "GPU memory bandwidth" and a sentence about "VRAM throughput" will produce embedding vectors that are close in high-dimensional space, even though they share no keywords. This is fundamentally different from keyword search (BM25, TF-IDF), which misses synonyms and paraphrases.

For local deployment, the embedding model runs separately from the LLM — it's a much smaller encoder-only model (typically 0.1-0.5 GB) that doesn't generate text, just transforms text into vectors. You can run the embedding model on CPU if needed, or on GPU for high-throughput batch indexing.

def embed_for_retrieval(texts: List[str], is_query: bool = False) -> np.ndarray:
    """
    Generate embeddings with the correct task prefix for asymmetric retrieval.

    Why asymmetric? Queries and documents have different characteristics:
    - Queries are short, terse, often missing context: "llm inference speed"
    - Documents are long, detailed, explicit: "Inference speed for LLMs depends on..."
    Training with separate prefixes teaches the model to bridge this gap —
    a short query maps to the same region as its matching long document.
    """
    prefix = "search_query: " if is_query else "search_document: "
    prefixed_texts = [prefix + t for t in texts]

    embeddings = model.encode(
        prefixed_texts,
        batch_size=128,                 # adjust for GPU VRAM (128 typically fine)
        show_progress_bar=len(texts) > 100,
        normalize_embeddings=True,      # normalize to unit sphere for cosine similarity
        convert_to_numpy=True,
    )
    return embeddings

def batch_embed_documents(document_paths: List[str]) -> dict:
    """High-throughput batch embedding for large document collections."""
    from tqdm import tqdm

    all_chunks = {}
    for path in document_paths:
        text = open(path).read()
        chunks = chunk_document(text)   # from previous section
        all_chunks[path] = chunks

    flat_chunks = [(path, chunk) for path, chunks in all_chunks.items()
                   for chunk in chunks]

    texts = [chunk for _, chunk in flat_chunks]
    embeddings = embed_for_retrieval(texts, is_query=False)

    return {
        "embeddings": embeddings,
        "metadata": [{"source": path, "text": chunk} for path, chunk in flat_chunks]
    }

8.8 Production Monitoring — Know What's Failing Before Users Do

vLLM exposes a Prometheus metrics endpoint at /metrics that gives you real-time visibility into what's actually happening inside the server. Most teams running local LLMs either don't monitor at all (until something breaks) or only check nvidia-smi occasionally. Neither is good enough for anything serving external users. Here's what each metric actually means and what threshold should trigger an alert.

The most important metric is vllm:gpu_cache_usage_perc — the fraction of KV cache blocks currently in use. When this approaches 1.0, vLLM starts preempting: it pauses a running sequence, evicts its KV cache to CPU RAM, and continues other requests. When the preempted request is unpaused, it re-runs prefill from scratch to rebuild the KV cache. Preemption is invisible to the user (they just experience higher latency) but shows up as a sudden spike in TTFT for the affected request. If this happens frequently, you're overloaded — increase --gpu-memory-utilization, reduce --max-model-len, or add more GPU capacity.

Perplexity (PPL) measures how surprised the model is by held-out text. Lower is better. It's the standard quantization quality metric and correlates well with general language quality. But perplexity is not a good proxy for task performance — a model with PPL=5.9 vs PPL=5.85 might perform identically on your actual use case, or might differ significantly. The difference depends on whether the quantization happens to affect the specific capabilities you care about.

The standard benchmarks are designed to isolate specific capabilities. MMLU (Massive Multitask Language Understanding) covers 57 subjects from STEM to law — it measures broad world knowledge. HumanEval tests Python code generation (164 problems). GSM8K tests grade-school math word problems (8500 problems). HellaSwag tests commonsense reasoning. For deployment, pick the benchmark that aligns with your actual use case and run it before and after any model change.

Part 9: Optimization — The Full Checklist

9.1 The Throughput vs. Latency Tradeoff

Before optimizing, get clear on what you're optimizing for. Throughput and latency are fundamentally opposed in LLM serving, and the right optimization depends on your use case.

Latency optimization (minimize time for a single request to complete): keep batch size low (ideally 1), use the smallest model that meets quality requirements, quantize aggressively (int4 halves decode time per request), use speculative decoding for predictable outputs. Accept that you're leaving GPU compute on the table — at batch=1, the GPU is only 5-10% utilized during decode.

Throughput optimization (maximize tokens generated per second across all users): use vLLM with continuous batching, maximize batch size, use prefix caching for shared system prompts, enable CUDA graph capture (pre-compiles common batch sizes, eliminating kernel launch overhead). Accept that individual request latency will be higher — a request that takes 3 seconds at batch=1 might take 5 seconds at batch=16, but you're serving 16x more users.

9.2 CUDA Graphs — Eliminating Kernel Launch Overhead

Each CUDA kernel launch has an overhead of roughly 5-20 microseconds. For the decode phase, where you're doing hundreds of kernel calls per token, this adds up. CUDA graphs let you record a sequence of kernel launches once, then replay the entire graph with a single API call. For common batch sizes in vLLM, this can reduce per-step latency by 10-30%. vLLM enables this by default (enforce_eager=False) — it captures graphs for batch sizes 1, 2, 4, 8, etc., at startup.

Part 10: Tool Selection — The Decision Framework

After going through all three tools in depth, the selection becomes clearer. This is not a case where one tool is objectively better — each occupies a different point in the tradeoff space, and the right choice depends entirely on your requirements.

Choose Ollama when you're a single developer using a model as a tool — a local code assistant, a personal writing helper, a quick API for a prototype. Ollama's value is zero friction: one command and you're running. Its model registry, automatic GPU detection, and Modelfile system mean you never touch a config file unless you want to. The limitation is that Ollama's scheduler is sequential by default, and even with OLLAMA_NUM_PARALLEL, it's not competitive with vLLM for serious multi-user throughput. Also, since it wraps llama.cpp, you're limited to GGUF-format models — if you have a HuggingFace-format fine-tune, you need to convert it first.

Choose vLLM when you're building a service with multiple concurrent users, an API backend, or any production workload where throughput matters. The PagedAttention scheduler, continuous batching, prefix caching, and native LoRA serving make it the right engineering choice for non-trivial serving scenarios. It also has the richest ecosystem: native AWQ/GPTQ/FP8 quantization, structured output, speculative decoding, multi-GPU support, and deep HuggingFace integration. The cost is complexity — vLLM has more moving parts and requires more configuration thought than Ollama.

Choose llama.cpp when you need fine-grained control over the inference stack, are embedding inference in a C++ application, need to run a model larger than your VRAM with CPU offloading, or are targeting non-CUDA environments (ARM, AMD, Apple Silicon). llama.cpp's single-user decode performance is competitive with vLLM, and its GGUF ecosystem is excellent. The weakness is multi-user throughput — llama-server's scheduler is not as sophisticated as vLLM's, and high-concurrency workloads will show the difference.

Final Thoughts

The field has moved extraordinarily fast. The same RTX 3090 that struggled to run a 7B model in 2023 now runs 8B models at full precision with room to spare for a KV cache that supports eight simultaneous conversations. The software has evolved even faster than the hardware — vLLM's PagedAttention, Flash Attention 2, AWQ quantization, and speculative decoding didn't exist in mature form 18 months ago.

What hasn't changed is the fundamentals: VRAM is finite, bandwidth is the decode bottleneck, and every optimization technique is ultimately either reducing how much data you read (quantization), amortizing that data reading across more work (batching), or reducing the number of reads needed per output (speculative decoding, prefix caching). When you understand those first principles, every new tool and every new paper slot into place immediately — you can evaluate a new technique by asking "which bottleneck does this address and by how much?" instead of treating each advance as magic.

Start with Ollama to get familiar with the models. Move to vLLM when you need to serve more than yourself. Use llama.cpp when you need to go places that Python frameworks can't follow. And always measure — the theoretical speedup from any optimization is only meaningful if you verify it empirically on your actual workload.
