Multi-GPU Kubernetes: How We Stopped Wasting 40% of Our Compute

GPU utilization monitoring, resource quotas, node affinity rules, and the specific NVIDIA device plugin configs that took us from 58% to 94% average GPU utilization across the cluster.

When we first looked at our GPU utilization dashboards six months ago, the number was 58%. On a cluster that costs a significant amount of money per month. That's 42% of our compute budget going to heat. This is the story of how we fixed it.

Why GPU Clusters Waste Compute

The culprits are almost always the same: over-provisioned resource requests, poor scheduling bin-packing, idle pods holding GPU reservations, and workload bursts that spike then crater. Kubernetes doesn't know what a GPU actually is — it just sees an opaque resource count. Getting it to schedule efficiently requires deliberate configuration.

The NVIDIA Device Plugin Setup That Actually Works

The NVIDIA device plugin is the bridge between Kubernetes and your GPUs. The default installation works, but the default config doesn't optimize for sharing. We use time-slicing for our preprocessing workloads (which use <10% GPU each) and dedicated GPU allocation for inference: 

# nvidia-device-plugin ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: nvidia-device-plugin
  namespace: kube-system
data:
  config.yaml: |
    version: v1
    flags:
      migStrategy: none
    sharing:
      timeSlicing:
        resources:
          - name: nvidia.com/gpu
            replicas: 4  # 4 logical GPUs per physical GPU
            # use for preprocessing pods only

Node Affinity: Putting the Right Workload on the Right Node

We have two node pools: A100s for inference, T4s for preprocessing and training. Without affinity rules, the scheduler happily puts inference pods on T4s and preprocessing on A100s. Affinity rules fix this: 

# For inference pods
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: nvidia.com/gpu.product
          operator: In
          values:
          - A100-SXM4-80GB

# Resource request that reserves a whole GPU
resources:
  limits:
    nvidia.com/gpu: 1

Right-Sizing Resource Requests

The single highest-impact change we made: auditing and correcting our resource requests. Most pods had been configured with "just in case" headroom — often requesting 2x the actual peak usage. We profiled every workload under production load for a week, then set requests at P95 actual usage + 15% buffer.

Over-requesting GPU memory is a silent killer. A pod requesting 40GB but using 12GB blocks 28GB from every other workload, indefinitely.

Monitoring That Actually Helps

We run DCGM exporter on every node and scrape into Prometheus. The dashboards that matter most for utilization optimization:

  • Per-pod GPU utilization — identify under-utilizing pods
  • GPU memory allocation vs usage — catch over-provisioning
  • GPU idle time per node — scheduling gaps and bin-packing issues
  • Pending pod queue depth — indicates resource pressure

The Results

After three months of systematic fixes — right-sizing requests, node affinity, time-slicing for low-utilization workloads, and fixing a scheduling bug where pods were landing on the wrong node pool — average cluster utilization went from 58% to 94%. That's a 62% improvement in compute efficiency with zero additional hardware spend.
