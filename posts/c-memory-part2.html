C Memory Internals #2: Realloc, Linked Lists & Struct Padding

Part 2 of the C memory series. We go deeper into heap mechanics — resizing memory with realloc, building linked lists that scatter nodes across the heap, and the invisible padding bytes the compiler hides inside your structs. Each concept ships with a live C program and an animated diagram.

Part 1 covered the fundamentals: stack, heap, pointer arithmetic, memory segments, and dynamic 2D arrays. Now we go deeper into how the heap actually behaves when you resize allocations, chain nodes together, and pack data into structs. These are the concepts that explain subtle bugs — the kind where your program "mostly works" until it mysteriously corrupts data or wastes 40% of its memory to padding.

Program 5 — realloc(): Resizing Heap Memory

malloc gives you a fixed-size block. But what if you need more space? realloc(ptr, new_size) resizes an existing allocation. The tricky part: it might move your data to a completely different address if there isn't enough free space after the current block.

Step 1: initial malloc(4 ints). Step 2: realloc grows in place — address unchanged, new slots added. Step 3: no space after the block — realloc allocates a new larger block, copies old data, frees the old block. Step 4: shrinking trims the tail.

#include <stdio.h>
#include <stdlib.h>

int main() {
    // Start with 4 ints
    int *arr = (int *)malloc(4 * sizeof(int));
    arr[0] = 10; arr[1] = 20; arr[2] = 30; arr[3] = 40;
    printf("Before realloc: arr = %p\n", (void*)arr);

    // Grow to 8 ints
    // DANGER: do NOT do:  arr = realloc(arr, 8 * sizeof(int));
    // If realloc fails and returns NULL, you lose the original pointer!
    int *tmp = (int *)realloc(arr, 8 * sizeof(int));
    if (!tmp) { free(arr); return 1; }  // safe pattern
    arr = tmp;
    arr[4]=50; arr[5]=60; arr[6]=70; arr[7]=80;
    printf("After  realloc: arr = %p\n", (void*)arr);
    // Address may be the same (grew in place) or different (moved)

    // Shrink back to 3 ints
    tmp = (int *)realloc(arr, 3 * sizeof(int));
    if (tmp) arr = tmp;  // arr[3]..arr[7] are now gone

    free(arr);
    return 0;
}
Common bug: arr = realloc(arr, new_size) is dangerous. If realloc fails it returns NULL, and you've just overwritten your only pointer to the old memory — it's leaked forever. Always use a temporary: tmp = realloc(arr, ...); if (tmp) arr = tmp;

What realloc does under the hood:

  1. Check if there's free space immediately after the current block
  2. If yes — extend in place, same address, no copy needed (fast)
  3. If no — allocate a new larger block, memcpy old data into it, free the old block (slow)
Real-world use: Dynamic arrays (like C++'s std::vector) use realloc internally, doubling size on each growth to amortise the cost. In CUDA, you can't realloc device memory — you must cudaMalloc a new buffer, cudaMemcpy the data, then cudaFree the old one. Same idea, more ceremony.

Program 6 — Linked List: Nodes Scattered Across the Heap

A linked list is a chain of nodes. Each node has two fields: a data value and a next pointer to the next node. Unlike arrays, nodes are not contiguous — each one is a separate malloc call, and they can end up anywhere in the heap.

Each node is a separate malloc call. The arrows follow the next pointer from node to node. Notice the addresses — they jump around the heap. This is why linked lists have poor cache performance: the CPU has to chase pointers across non-contiguous memory.

#include <stdio.h>
#include <stdlib.h>

typedef struct Node {
    int         data;
    struct Node *next;   // pointer to the next node (or NULL if last)
} Node;

Node *new_node(int value) {
    Node *n = (Node *)malloc(sizeof(Node));  // one heap allocation per node
    n->data = value;
    n->next = NULL;
    return n;
}

int main() {
    // Build: 10 -> 20 -> 30 -> 40 -> NULL
    Node *head = new_node(10);
    head->next = new_node(20);
    head->next->next = new_node(30);
    head->next->next->next = new_node(40);

    // Walk the chain and print addresses
    Node *cur = head;
    while (cur) {
        printf("node(%d) at %p  next-> %p\n",
               cur->data, (void*)cur, (void*)cur->next);
        cur = cur->next;
    }
    // Addresses will NOT be contiguous — they scatter across the heap

    // Free: walk and free each node individually
    cur = head;
    while (cur) {
        Node *tmp = cur->next;  // save next BEFORE freeing cur
        free(cur);
        cur = tmp;
    }
    return 0;
}

The node struct uses n->field syntax (arrow operator) instead of (*n).field. They're identical — the arrow is just shorthand for "dereference the pointer, then access the field".

  • O(1) insert at head — just point new node's next at old head
  • O(n) access by index — must walk from head every time
  • Poor cache performance — nodes land at random heap addresses, CPU cache misses on every step
  • Must free one node at a time — you cannot bulk-free a linked list
This is why GPU algorithms avoid pointer-chasing data structures. A GPU has thousands of threads, each needing to access memory in parallel — random pointer hops destroy memory throughput.

Program 7 — Struct Padding & Memory Alignment

This one surprised me the most. sizeof(struct) is often larger than the sum of its member sizes. The compiler inserts invisible padding bytes to satisfy the CPU's alignment requirements.

The CPU wants to read an int (4 bytes) from an address divisible by 4. It wants a double (8 bytes) at an address divisible by 8. If the data isn't aligned, the CPU needs two memory reads instead of one. So the compiler adds padding bytes before each field to push it to the right alignment boundary.

The 32 bytes of struct Bad are revealed one by one. Red slots are padding — completely wasted. The struct has only 15 bytes of actual data but takes 32 bytes because of poor member ordering.

#include <stdio.h>
#include <stddef.h>   // offsetof()

// BAD ordering — lots of padding
typedef struct {
    char   a;    //  1 byte  @ offset 0
                 //  3 bytes PADDING (to align b to a multiple of 4)
    int    b;    //  4 bytes @ offset 4
    char   c;    //  1 byte  @ offset 8
                 //  7 bytes PADDING (to align d to a multiple of 8)
    double d;    //  8 bytes @ offset 16
    char   e;    //  1 byte  @ offset 24
                 //  7 bytes PADDING (struct size must be multiple of largest member=8)
} Bad;           // Total: 32 bytes.  But members only sum to 15!

// GOOD ordering — largest members first
typedef struct {
    double d;    //  8 bytes @ offset 0
    int    b;    //  4 bytes @ offset 8
    char   a;    //  1 byte  @ offset 12
    char   c;    //  1 byte  @ offset 13
    char   e;    //  1 byte  @ offset 14
                 //  1 byte  tail pad
} Good;          // Total: 16 bytes.  Same data, half the size!

int main() {
    printf("sizeof(Bad)  = %zu\n", sizeof(Bad));   // 32
    printf("sizeof(Good) = %zu\n", sizeof(Good));  // 16
    // Same 5 members, different order — 16 bytes saved per struct instance!

    printf("\nBad offsets:\n");
    printf("  a @ %zu\n", offsetof(Bad, a));   // 0
    printf("  b @ %zu\n", offsetof(Bad, b));   // 4  (3 bytes wasted)
    printf("  c @ %zu\n", offsetof(Bad, c));   // 8
    printf("  d @ %zu\n", offsetof(Bad, d));   // 16 (7 bytes wasted)
    printf("  e @ %zu\n", offsetof(Bad, e));   // 24
    return 0;
}
The rule: Order struct members from largest to smallest type: double (8) → int (4) → short (2) → char (1). This eliminates most internal padding. If you store millions of structs in an array, this saves significant memory — and better cache utilisation means better performance.

__attribute__((packed)) removes all padding entirely but is dangerous on some CPUs (causes unaligned access faults on ARM) and is always slower on x86. Use it only for network packet headers or binary file formats where the exact byte layout is mandated by an external protocol.

In CUDA, struct layout is even more critical. GPU memory transactions happen in 128-byte cache lines. If your struct has 17 bytes of padding, you're wasting that bandwidth on every single thread.
