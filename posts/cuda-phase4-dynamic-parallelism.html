Dynamic Parallelism — Kernels Launching Kernels

Dynamic Parallelism lets GPU kernels launch child kernels without returning to the CPU. This enables adaptive algorithms where the GPU itself decides how much work to spawn based on data seen at runtime — recursive refinement, tree traversal, and variable-depth BFS all become natural to express. But it comes with real overhead and is often the wrong tool. This post explains both when it helps and when to avoid it.

Before Dynamic Parallelism existed, every kernel launch required a CPU round-trip. Your algorithm had to finish a GPU pass, return to the CPU, make a decision, then launch the next GPU pass. That round-trip adds 10–50 microseconds per decision — tolerable for occasional decisions, catastrophic for recursive algorithms that make thousands of decisions per second.

What Is Dynamic Parallelism

With Dynamic Parallelism (Kepler sm_35+), a kernel thread can launch a new kernel directly from the GPU. The syntax is the same as CPU-side kernel launches: childKernel<<<grid, block>>>(args), but called from inside a __global__ function instead of from host code. 

// Without Dynamic Parallelism: CPU must coordinate every decision
void cpuCoordinated(float* data, int n) {
    // Pass 1: compute partial results
    kernel1<<<grid, block>>>(data, d_count, n);
    cudaDeviceSynchronize();  // GPU → CPU: ~10-50us latency

    // CPU reads result, decides how much work for pass 2
    int h_count;
    cudaMemcpy(&h_count, d_count, sizeof(int), cudaMemcpyDeviceToHost);
    // More latency: PCIe transfer + CPU decision + kernel launch

    // Pass 2: work decided by CPU based on GPU-computed count
    int newGrid = (h_count + 255) / 256;
    kernel2<<<newGrid, 256>>>(data, h_count);
    // Total added latency: ~50-100us per CPU round-trip
}

// With Dynamic Parallelism: GPU decides and launches immediately
__global__ void gpuCoordinated(float* data, int* d_count, int n) {
    // ... parent thread does phase 1 work ...

    // Only ONE thread needs to launch the child (see explanation below)
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        // GPU computes d_count, then launches phase 2 directly -- no CPU involved
        int newGrid = (*d_count + 255) / 256;
        childKernel2<<<newGrid, 256>>>(data, *d_count);
        // ~50us launch overhead, but no PCIe round-trip
    }
}

Compiling for Dynamic Parallelism

# Requires -rdc=true (relocatable device code) and linking with cudadevrt
nvcc -arch=sm_70 -rdc=true mykernel.cu -o myprogram -lcudadevrt

# Or with CMake:
set_property(TARGET mytarget PROPERTY CUDA_SEPARABLE_COMPILATION ON)
target_link_libraries(mytarget cudadevrt)
What is -rdc=true and why does it have a cost?

Normal CUDA compilation (-rdc=false) compiles each .cu file into a self-contained binary. The compiler can inline functions across the whole file, optimize aggressively, and produce compact PTX.

Relocatable device code (-rdc=true) compiles each file into a relocatable object, similar to how CPU linkers work. Device functions are linked at binary link time, not compile time. This is required for: (a) device-side kernel launches, (b) calls to device functions in other .cu files.

The cost: the compiler cannot inline across translation unit boundaries at compile time. Some PTX-level optimizations are disabled. Kernels that do NOT use dynamic parallelism should use -rdc=false where possible.

Basic Usage: Child Kernel Launch

__global__ void childKernel(float* data, int n, int depth) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        data[i] = data[i] * depth;
    }
}

__global__ void parentKernel(float* data, int n) {
    int blockStart = blockIdx.x * blockDim.x;
    int blockN     = min(blockDim.x, n - blockStart);

    // IMPORTANT: gate child launch on threadIdx.x == 0
    // (full explanation below)
    if (threadIdx.x == 0) {
        childKernel<<<1, blockN>>>(data + blockStart, blockN, blockIdx.x);
        // The child kernel runs asynchronously -- parent continues executing
        // while child is queued and (eventually) runs
    }

    // Parent can do other work here while child is pending...
    // But: global memory writes by the parent after this point
    // are NOT guaranteed visible to the child without explicit sync
}

// Host:
parentKernel<<<numBlocks, blockSize>>>(d_data, N);
cudaDeviceSynchronize();  // waits for parent AND all children to complete
Why gate child launches on threadIdx.x == 0?

Each thread in a block has its own execution context. If 256 threads all call childKernel<<<...>>>(), the GPU queues 256 separate child grids. That is almost never what you want — it means 256 copies of the same computation, consuming the pending launch queue extremely fast and adding 256x the overhead.

The convention is: exactly one thread per block (or per grid) is responsible for launching the child. Use if (threadIdx.x == 0) to select one thread per block, or if (threadIdx.x == 0 && blockIdx.x == 0) for one launch per grid.

If different blocks need different child grid sizes (adaptive refinement), you would use if (threadIdx.x == 0) to let each block launch its own appropriately-sized child. This is one of the primary use cases for dynamic parallelism.
Memory visibility rules for parent and child:

(1) The parent's global memory writes made before the child launch are visible to the child kernel — the launch acts as a memory fence for the launching thread's block.
(2) Sibling child kernels (launched by different parent threads or blocks) have no guaranteed ordering relative to each other. Treat them as independent unless you add explicit synchronization.
(3) Child kernel results are visible to the parent only after cudaDeviceSynchronize() (from device code) returns.

Synchronizing Child Kernels

__global__ void parentWithSync(float* data, int n) {
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        // Launch child to fill data[]
        childKernel<<<16, 256>>>(data, n);

        // cudaDeviceSynchronize() from DEVICE code:
        // waits for all children launched by any thread in THIS grid
        // (not the entire GPU -- just children of this kernel invocation)
        cudaDeviceSynchronize();

        // NOW safe to read data[] modified by childKernel
        printf("data[0] = %f\n", data[0]);

        // Launch a second child that processes the results
        processKernel<<<16, 256>>>(data, n);
        cudaDeviceSynchronize();
    }
}

// Using streams for concurrent child kernels:
__global__ void parentConcurrent(float* A, float* B, float* C, int n) {
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        // Create streams (device-side streams are for concurrent child launches)
        cudaStream_t s1, s2;
        cudaStreamCreateWithFlags(&s1, cudaStreamNonBlocking);
        cudaStreamCreateWithFlags(&s2, cudaStreamNonBlocking);

        // Launch two independent children concurrently
        childA<<<8, 256, 0, s1>>>(A, n);  // fills A[]
        childB<<<8, 256, 0, s2>>>(B, n);  // fills B[] concurrently

        cudaDeviceSynchronize();  // wait for both A and B to finish

        // Now merge: A[] and B[] are both ready
        mergeKernel<<<8, 256>>>(A, B, C, n);
        cudaDeviceSynchronize();

        cudaStreamDestroy(s1);
        cudaStreamDestroy(s2);
    }
}

Recursive Kernels

The most iconic dynamic parallelism use case: a kernel that calls itself recursively. Each recursive level subdivides the problem and launches smaller sub-problems. 

// Recursive quicksort: each call partitions a range, then recursively sorts each half
// This is a simplified demo -- a production quicksort would use multiple threads for partitioning
__global__ void quicksort(int* data, int lo, int hi) {
    if (lo >= hi) return;  // base case: 0 or 1 element -- already sorted

    // One thread partitions the range (simplified Lomuto partition)
    if (threadIdx.x == 0 && blockIdx.x == 0) {
        // Choose last element as pivot
        int pivot = data[hi];
        int i = lo - 1;

        // Move all elements <= pivot to the left of i
        for (int j = lo; j < hi; j++) {
            if (data[j] <= pivot) {
                i++;
                int tmp = data[i]; data[i] = data[j]; data[j] = tmp;
            }
        }
        // Place pivot at its final position
        i++;
        int tmp = data[i]; data[i] = data[hi]; data[hi] = tmp;
        int pivotIdx = i;

        // Recursively sort left partition (elements < pivot)
        if (pivotIdx - 1 > lo)
            quicksort<<<1, 1>>>(data, lo, pivotIdx - 1);

        // Recursively sort right partition (elements > pivot)
        if (pivotIdx + 1 < hi)
            quicksort<<<1, 1>>>(data, pivotIdx + 1, hi);
        // Both child launches run concurrently with each other
    }
}

// Initial call from host:
quicksort<<<1, 1>>>(d_data, 0, N - 1);
cudaDeviceSynchronize();
This quicksort is a demonstration, not production code. The single-thread partition is O(N) serial work per level — no parallelism within a partition step. A real GPU quicksort would use a parallel partition (thrust::partition or a warp-parallel Hoare scheme) within each kernel. The recursion structure shown here is correct; the inner partition needs more work.
Recursive dynamic parallelism has hard limits:
Maximum nesting depth: 24 levels
Default pending launch queue: 2048 launches
Each recursive level uses device memory for launch state

For N=1000 elements, quicksort has O(log N) ≈ 10 levels on average — well within limits. For N=1M, worst-case O(N) levels would overflow the queue. Always add a minimum problem size threshold to avoid deep recursion on tiny subproblems.

When to Use Dynamic Parallelism

Good Use CaseWhy DP helps
Adaptive mesh refinementGPU inspects each cell and decides whether to subdivide it; decision and launch happen without CPU involvement
Irregular graph traversal (BFS)Frontier size is unknown until each BFS level completes; DP avoids overprovisioning or under-using the GPU
Variable-depth recursionTree structures where depth is data-dependent; each node decides independently
Iterative solvers with adaptive convergenceGPU checks residual and launches next iteration only if needed
N-body with variable accuracyClose particles need more sub-steps; GPU decides per particle pair
Bad Use CaseBetter Alternative
Fixed, known problem sizeLaunch the right grid size from CPU; no overhead
Regular tree reductionTwo-pass reduction (post 4.2) or CUB -- far less overhead
Conditional work selectionWarp voting + compaction; keeps all threads busy
SortingThrust or CUB sort -- GPU-optimized, handles all sizes
Launching one kernel from inside another "just because you can"Just restructure the kernel; the overhead is rarely worth it

Overhead and Limitations

Dynamic parallelism is not free. Each child kernel launch from the GPU goes through a device-side command queue (the "pending launch queue") before reaching the GPU scheduler.

# Kernel launch overhead comparison (A100): CPU-side kernel launch: ~5 us (microseconds) GPU-side (dynamic) kernel launch: ~50 us (10x more expensive than CPU launch!) GPU-side launch (warm, amortized): ~10 us (after queue warmup on repeated calls) # Pending launch queue defaults: Max pending launches per device: 2048 (configurable with cudaDeviceSetLimit) Max recursion depth: 24 levels Max child grid dimensions: Same as normal grid (no additional constraints) # cudaDeviceSetLimit example (increase queue size for deep recursion): cudaDeviceSetLimit(cudaLimitDevRuntimePendingLaunchCount, 8192);
Understanding the pending launch queue:

Think of it as an inbox. Every child<<<...>>>() call from device code posts a message to this inbox. The GPU scheduler reads messages and launches kernels as resources become available. The inbox holds 2048 messages by default.

If your algorithm launches 3000 children before the scheduler has processed any of them, the 2049th call will either block the parent thread (stalling it) or fail with an error. For deep recursive algorithms, increase the limit with cudaDeviceSetLimit before launching the parent kernel from the CPU.

This is why child launches have ∼50 us overhead — the launch must go through this queue and the associated synchronization. A CPU launch bypasses this entirely and writes directly to the GPU command buffer.

Alternatives to Dynamic Parallelism

Before using dynamic parallelism, try these alternatives. They are usually faster with less complexity:

  • Cooperative Groups (CUDA 9+): Flexible synchronization at thread, warp, block, or multi-block level. Grid-wide sync without child launches.
  • Persistent kernels + work queues: One kernel that lives until the queue is empty. Threads atomically claim work items. Eliminates launch overhead entirely.
  • CUB DeviceSelect / DevicePartition: Compact active elements, then launch a fixed kernel on the compacted data. No adaptive grid sizing needed.
  • CPU dispatch loop with cudaStreamWaitEvent: For coarse-grained decisions (every 10ms), a CPU loop is perfectly fine and has predictable behavior.
// Persistent kernel with work queue -- often better than dynamic parallelism
// One kernel launch processes unlimited amounts of work adaptively
__global__ void persistentKernel(WorkItem* queue, int* tail, int maxWork) {
    // Each thread keeps claiming and processing items until queue is empty
    while (true) {
        // Atomically claim the next work item index
        int idx = atomicAdd(tail, 1);
        if (idx >= maxWork) break;  // queue exhausted, thread exits

        WorkItem item = queue[idx];
        processItem(item);

        // If this item generates sub-problems, enqueue them
        if (item.needsSubdivision) {
            // Claim 2 new slots at the end of the queue
            int newIdx = atomicAdd(tail, 2);
            if (newIdx + 1 < maxWork) {
                queue[newIdx]   = subdivide(item, 0);  // left sub-problem
                queue[newIdx+1] = subdivide(item, 1);  // right sub-problem
            }
        }
    }
}

// Launch once with enough threads to saturate SMs
// The kernel processes ALL work adaptively without ever returning to CPU
persistentKernel<<<numSMs * 4, 256>>>(d_queue, d_tail, maxWork);
// maxWork = initial queue size + max possible expansions

// Advantages over dynamic parallelism:
// - No per-launch overhead (50 us per child vs 0 per work item)
// - No pending queue limits
// - No recursion depth limits
// - Naturally load-balanced (fast threads claim more items)
