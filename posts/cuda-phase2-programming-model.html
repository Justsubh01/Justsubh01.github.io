CUDA Mastery Phase 2.1 вҖ” The CUDA Programming Model: Threads, Blocks, and Grids

How CUDA organizes thousands of threads into a three-level hierarchy of threads, blocks, and grids вҖ” how to compute global thread indices, how to choose block sizes, and how the hardware maps your abstract hierarchy onto physical SMs and CUDA cores.

CUDA's programming model is elegantly simple. There are three levels: a thread does one unit of work, blocks group threads that can cooperate via shared memory, and a grid is the full collection of blocks for one kernel launch. Everything else вҖ” occupancy, coalescing, divergence вҖ” is built on top of this foundation.

1. The Thread Hierarchy

When you launch a CUDA kernel, you create a grid of blocks. Each block contains a fixed number of threads. Every thread runs the same kernel function, but identifies itself using built-in index variables.

2. Built-in Variables

Inside a kernel, four built-in structs give each thread its identity. All are of type dim3 (a struct with x, y, z fields). 

  threadIdx  вҖ” thread's index within its block  (0 to blockDim-1)
  blockIdx   вҖ” block's index within the grid    (0 to gridDim-1)
  blockDim   вҖ” dimensions of each block         (constant for a launch)
  gridDim    вҖ” dimensions of the grid           (constant for a launch)

  Example for a 1D launch with blockDim.x=256:

  Thread in block 3, position 7:
    threadIdx.x = 7
    blockIdx.x  = 3
    blockDim.x  = 256
    gridDim.x   = (N + 255) / 256

  Global index (unique ID for this thread):
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    //      = 3          * 256         + 7
    //      = 775

3. Computing Thread Indices вҖ” 1D, 2D, 3D

1D (most common)

__global__ void kernel_1d(float *data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n)
        data[idx] = process(data[idx]);
}

2D (images, matrices)

__global__ void kernel_2d(float *matrix, int width, int height) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;  // x = column
    int row = blockIdx.y * blockDim.y + threadIdx.y;  // y = row

    if (col < width && row < height) {
        int idx = row * width + col;   // row-major linearization
        matrix[idx] = process(matrix[idx]);
    }
}

// Launch:
dim3 blockDim(16, 16);   // 16Г—16 = 256 threads per block
dim3 gridDim((W + 15) / 16, (H + 15) / 16);
kernel_2d<<<gridDim, blockDim>>>(d_mat, W, H);

3D (volumes, batched ops)

__global__ void kernel_3d(float *volume, int W, int H, int D) {
    int x = blockIdx.x * blockDim.x + threadIdx.x;
    int y = blockIdx.y * blockDim.y + threadIdx.y;
    int z = blockIdx.z * blockDim.z + threadIdx.z;

    if (x < W && y < H && z < D) {
        int idx = z * (W * H) + y * W + x;
        volume[idx] = process(volume[idx]);
    }
}

5. Choosing Block Size вҖ” The Rules

Block size is one of the most impactful tuning parameters. Here are the rules:

  • Always use a multiple of 32 (warp size). Non-multiples waste CUDA cores in the last warp.
  • Common good choices: 128, 256, 512 threads/block.
  • Maximum: 1024 threads per block (hardware limit).
  • Minimum practical: 64 threads (fewer = poor occupancy).
  • Start with 256; profile and tune from there.
  Block size tradeoffs:

  Too small (e.g. 32):
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  + Easy to reason about
  - Need more blocks (more launch overhead)
  - Less shared memory reuse within a block
  - Low occupancy if few blocks fit per SM

  Too large (e.g. 1024):
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  + More threads can share data via __shared__
  - More register pressure (fewer blocks fit per SM)
  - Limited flexibility in grid sizing

  Sweet spot: 128вҖ“256 for most kernels.
  Use cudaOccupancyMaxPotentialBlockSize() to find the optimal value.
  // CUDA helper to find optimal block size:
  int minGridSize, blockSize;
  cudaOccupancyMaxPotentialBlockSize(
      &minGridSize,       // min grid size for full occupancy
      &blockSize,         // optimal block size
      myKernel,           // the kernel function
      0,                  // dynamic shared mem per block
      0                   // maximum block size (0 = no limit)
  );
  int gridSize = (N + blockSize - 1) / blockSize;
  myKernel<<<gridSize, blockSize>>>(d_data, N);

6. Grid Size Calculation

  Problem: launch enough threads to cover N elements.
  Each block has blockSize threads.

  Naive (WRONG вҖ” misses last partial block):
    gridSize = N / blockSize;    // integer truncation!
    // N=1000, blockSize=256 вҶ’ gridSize=3 вҶ’ only 768 threads!

  Correct (ceiling division):
    gridSize = (N + blockSize - 1) / blockSize;
    // N=1000, blockSize=256 вҶ’ (1000+255)/256 = 4 вҶ’ 1024 threads
    // elements [768..999] are in block 3, elements [1000..1023] are out-of-bounds

  Always guard in kernel with bounds check:
    if (idx < N) { ... }
    // threads 1000-1023 do nothing вҖ” wastes 24 threads but avoids memory corruption

7. Function Qualifiers

  __global__   вҖ” kernel function
                 вҖў Called from HOST, executes on DEVICE
                 вҖў Returns void (always)
                 вҖў Launch with <<<grid, block>>>

  __device__   вҖ” device function
                 вҖў Called from DEVICE, executes on DEVICE
                 вҖў Cannot be called from host
                 вҖў Gets inlined by nvcc (like inline in C)

  __host__     вҖ” host function (default if no qualifier)
                 вҖў Called from HOST, executes on HOST
                 вҖў Normal C/C++ function

  __host__ __device__  вҖ” compiled for both
                 вҖў Useful for math helper functions used on both CPU and GPU
                 вҖў e.g., clamp(), lerp(), vec3 operations

  Examples:
  __global__ void vectorAdd(float *a, float *b, float *c, int n);
  __device__ float sigmoid(float x) { return 1.0f / (1.0f + expf(-x)); }
  __host__ __device__ float clamp(float x, float lo, float hi) {
      return fmaxf(lo, fminf(x, hi));
  }

8. How Hardware Executes Your Launch

  Step 1: You call kernel<<<grid, block>>>(args)
  Step 2: CUDA driver packages this as a "task" for the GPU
  Step 3: GPU Work Distribution unit assigns blocks to SMs

  SM assignment rules:
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  - Each SM gets blocks assigned one at a time
  - A SM can hold multiple blocks simultaneously (resource permitting)
  - Resources that limit blocks per SM:
      * Registers: 65536 total per SM / (registers_per_thread Г— block_size)
      * Shared memory: 48KB total / shared_mem_per_block
      * Thread limit: 2048 max threads per SM / threads_per_block

  Step 4: Within each SM, threads are organized into warps of 32
  Step 5: Warp schedulers issue instructions to ready warps each cycle
  Step 6: When a warp stalls (memory), another warp executes (latency hiding)
  Step 7: When all threads in a block complete, SM gets a new block

10. Grid-Stride Loop Pattern

When N is larger than the grid size, or when you want to reuse threads, use a grid-stride loop. This is more flexible than one-thread-per-element: 

__global__ void process_large(float *data, int n) {
    // Each thread starts at its unique global index...
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    // ...and steps by the total grid width each iteration
    int stride = blockDim.x * gridDim.x;

    for (; idx < n; idx += stride) {
        data[idx] = sqrtf(data[idx]);
    }
}

// Can launch with any grid size (even 1 block!)
// Still processes all N elements correctly:
process_large<<<128, 256>>>(d_data, 10000000);
//  128Г—256 = 32768 threads, each handles ~305 elements

Takeaways

  • CUDA thread hierarchy: thread вҶ’ block вҶ’ grid. Block = cooperation unit; grid = full launch.
  • Global index formula: idx = blockIdx.x * blockDim.x + threadIdx.x
  • Always guard kernels with if (idx < N) вҖ” last block may be partial.
  • Use ceiling division for grid size: (N + blockSize - 1) / blockSize
  • Block size should be a multiple of 32. Start with 256; tune with occupancy API.
  • __global__ = kernel, __device__ = device helper, __host__ __device__ = both.
  • Blocks are assigned to SMs; register/shared memory limits how many fit simultaneously.
  • Grid-stride loops let one kernel handle any N, regardless of grid size.
