1.1 — C/C++ Fundamentals Every CUDA Developer Must Know

Before you write a single kernel, you need to own these C concepts cold. Pointers, heap vs stack, struct padding, storage classes, the CUDA_CHECK macro, function pointers, and the restrict keyword — all with direct CUDA connections explained. This is the foundation everything else rests on.

You can copy-paste a CUDA vector addition kernel and get it to compile on day one. But when you hit a segfault in device code, a silent memory corruption that gives wrong results with no error message, or a 10x performance drop you cannot explain — that is when shallow C knowledge kills you. Every confusing CUDA concept maps directly back to something C does first. This post is that map.

We are going through seven C/C++ topics in surgical detail. Each one has code you can run, output you should expect, and an explicit note on exactly where and how it shows up in CUDA. No filler — every section earns its place.

1. Pointers — The Foundation

A pointer is a variable that stores a memory address. Not a value, not a reference in the Java sense — a raw numeric address into your process's virtual address space. The type attached to a pointer (int*, float*, char*) tells the compiler how many bytes to read when you dereference it, and how many bytes to advance when you do pointer arithmetic. That is the entire magic of pointers. Everything else is a consequence.

Declaration, dereferencing, and address-of

#include <stdio.h>

int main(void) {
    int x = 42;

    /* &x is the "address-of" operator: gives us the address where x lives */
    int *p = &x;

    printf("value of x      : %d\n",   x);     /* 42                   */
    printf("address of x    : %p\n",  &x);     /* 0x7ffd4a2b3c10       */
    printf("value of p      : %p\n",   p);     /* same address as &x   */
    printf("dereferenced *p : %d\n",  *p);     /* 42 (read through ptr)*/

    /* Writing through the pointer changes x */
    *p = 99;
    printf("x after *p = 99 : %d\n",   x);     /* 99                   */

    return 0;
}
value of x : 42
address of x : 0x7ffd4a2b3c10
value of p : 0x7ffd4a2b3c10
dereferenced *p : 42
x after *p = 99 : 99

The address printed will differ every run because of ASLR (Address Space Layout Randomization), but the pattern is always the same: p holds the same numeric value as &x, and dereferencing reads or writes the bytes at that address.

Pointer arithmetic

When you add an integer to a pointer, the compiler scales by sizeof(pointed-to type). So p + 1 on an int* advances by 4 bytes, not 1. 

#include <stdio.h>

int main(void) {
    int arr[5] = {10, 20, 30, 40, 50};
    int *p = arr;   /* arr decays to &arr[0] */

    printf("arr[0] at %p = %d\n", p,   *p);        /* 10 */
    printf("arr[1] at %p = %d\n", p+1, *(p+1));    /* 20 */
    printf("arr[4] at %p = %d\n", p+4, *(p+4));    /* 50 */

    /* arr[i] and *(arr + i) are IDENTICAL; the compiler sees both the same way */
    printf("%d == %d\n", arr[3], *(arr + 3));       /* 40 == 40 */

    /* Legal but unhinged: index commutes (a[b] == b[a] because *(a+b) == *(b+a)) */
    printf("%d\n", 3[arr]);     /* 40 */

    return 0;
}

/* Memory layout (sizeof(int) == 4):
   Index : [0]  [1]  [2]  [3]  [4]
   Value :  10   20   30   40   50
   Offset:   0    4    8   12   16  bytes from arr[0]
*/
arr[0] at 0x7ffd... = 10
arr[1] at 0x7ffd...+4 = 20
arr[4] at 0x7ffd...+16 = 50
40 == 40

Pointer to pointer (**pp)

A pointer can point to another pointer. This is not a trick question — it is just another level of indirection. A int** stores the address of an int*. 

#include <stdio.h>

int main(void) {
    int   value = 7;
    int  *p     = &value;   /* p  holds address of value */
    int **pp    = &p;       /* pp holds address of p     */

    printf("value   = %d\n",   value);    /* 7 */
    printf("*p      = %d\n",  *p);        /* 7 (one dereference) */
    printf("**pp    = %d\n", **pp);       /* 7 (two dereferences) */

    /* Modify through double pointer */
    **pp = 42;
    printf("value after **pp = 42: %d\n", value);   /* 42 */

    printf("\nAddresses:\n");
    printf("&value = %p\n", (void*)&value);
    printf("p      = %p (same as &value)\n", (void*)p);
    printf("&p     = %p\n", (void*)&p);
    printf("pp     = %p (same as &p)\n", (void*)pp);

    return 0;
}
value = 7
*p = 7
**pp = 7
value after **pp = 42: 42

Addresses:
&value = 0x7ffd...10
p = 0x7ffd...10 (same as &value)
&p = 0x7ffd...18
pp = 0x7ffd...18 (same as &p)

void* and casting

void* is a "pointer to anything" — it holds an address but carries no type information. You cannot dereference it directly; you must cast it to a typed pointer first. malloc() returns void* precisely because it does not know what you are allocating. 

#include <stdlib.h>
#include <stdio.h>

int main(void) {
    void *raw = malloc(4);          /* void* — no type yet */
    int  *ip  = (int*)raw;          /* cast to int*        */
    *ip = 1234;
    printf("%d\n", *ip);            /* 1234 */
    free(raw);                      /* free the same pointer, either name works */

    return 0;
}
CUDA connection — cudaMalloc uses void**:
Every CUDA memory allocation gives you a pointer. cudaMalloc(void** devPtr, size_t size) takes a pointer to a pointer because it needs to write the device address into your variable. The signature is exactly the double-pointer pattern you just saw. If you do not understand pointers, you will not understand CUDA. This is not hyperbole. Every kernel argument that touches GPU memory is a device pointer. Every copy is from address to address. The entire CUDA memory model is pointer arithmetic across a separate address space. 
float *d_arr = NULL;
/* cudaMalloc writes the device address into d_arr */
cudaMalloc((void**)&d_arr, N * sizeof(float));
/* d_arr now holds an address in GPU DRAM, not CPU RAM */

2. Stack vs Heap — Where Memory Lives

Every byte of memory your program touches lives somewhere in the process's virtual address space. The two regions programmers manage directly are the stack and the heap.

The stack

The stack is automatic, LIFO (last-in first-out), and fast. Local variables, function arguments, and return addresses all live on the stack. When a function is called, the CPU pushes a new stack frame onto the stack. When the function returns, that frame is popped and the memory is instantly reclaimed — no free() required. The OS typically limits the stack to around 8 MB. Exceeding that limit is a stack overflow (segfault).

The heap

The heap is manual, large (limited by RAM + swap), and slower than the stack because allocation requires bookkeeping. malloc()/free() (C) or new/delete (C++) manage heap memory. You are responsible for freeing it. If you don't, you have a memory leak. 

#include <stdio.h>
#include <stdlib.h>

void demonstrate_stack_and_heap(void) {
    /* STACK: declared inside function, gone when function returns */
    int  stack_int   = 10;
    char stack_char  = 'A';
    int  stack_arr[4] = {1, 2, 3, 4};   /* 16 bytes on stack */

    /* HEAP: explicitly allocated, survives function return until freed */
    int  *heap_arr   = (int*)malloc(1000000 * sizeof(int));  /* ~4 MB on heap */
    if (!heap_arr) { perror("malloc"); return; }

    heap_arr[0] = 999;

    printf("stack_int   addr: %p\n", (void*)&stack_int);
    printf("stack_char  addr: %p\n", (void*)&stack_char);
    printf("stack_arr   addr: %p\n", (void*)stack_arr);
    printf("heap_arr    addr: %p  (completely different region)\n",
           (void*)heap_arr);

    /* MUST free heap memory; stack memory is reclaimed automatically */
    free(heap_arr);
}

int main(void) {
    demonstrate_stack_and_heap();
    /* stack variables are now gone; heap was freed inside the function */
    return 0;
}
stack_int addr: 0x7ffd4a2b3c14 <-- high address (stack grows down)
stack_char addr: 0x7ffd4a2b3c10
stack_arr addr: 0x7ffd4a2b3c00
heap_arr addr: 0x55f8b2a00260 <-- low address (heap grows up)

Notice the addresses: stack lives at high addresses (near the top of the virtual address space), heap lives at low addresses. The two grow toward each other. The gap between them is the safety net — modern 64-bit address spaces make this gap effectively infinite.

Heap fragmentation

When you repeatedly malloc() and free() blocks of different sizes, the heap develops "holes" — free regions too small to satisfy large requests. This is heap fragmentation. It doesn't cause crashes (the allocator handles it), but it wastes memory and can slow down allocation over time. This is one reason high-performance code uses arena allocators or memory pools instead of raw malloc().

CUDA: cudaMalloc allocates on DEVICE heap. Never pass a host stack pointer to a kernel.

cudaMalloc() allocates memory in GPU global memory (the device heap). cudaMemcpy() moves data between host heap/stack and device heap. But here is the rule you must burn into your brain: the GPU cannot dereference a host pointer. GPU threads live in a separate virtual address space. A host stack address like 0x7ffd4a2b3c14 is meaningless to a GPU thread — dereferencing it is undefined behavior and will either silently corrupt data or crash. Always allocate device memory with cudaMalloc() and copy data to it with cudaMemcpy() before the kernel runs.

float stack_val = 3.14f;
kernel<<<1,1>>>(&stack_val); // WRONG: passing host stack address to GPU

3. const, static, volatile — Storage Classes

Storage class keywords control three things: where a variable lives in memory, how long it lives (lifetime), and who can see it (scope). Getting these right matters for correctness and performance in both C and CUDA.

const — value won't change

#include <stdio.h>

int main(void) {
    const int MAX_SIZE = 1024;
    /* MAX_SIZE = 2048;  // compile error: assignment of read-only variable */

    /* Pointer to const: value read-only, pointer can move */
    const float *pf = NULL;
    float a = 1.0f, b = 2.0f;
    pf = &a;          /* OK: pf can change where it points */
    /* *pf = 9.0f; */ /* ERROR: cannot write through const pointer */
    pf = &b;          /* OK */

    /* Const pointer: pointer fixed, value writable */
    float * const cp = &a;
    *cp = 5.0f;        /* OK: value is writable */
    /* cp = &b; */    /* ERROR: cannot move the pointer */

    return 0;
}

static local — persists across calls

#include <stdio.h>

void count_calls(void) {
    /* NOT on the stack; lives in BSS/DATA segment for the lifetime of the program.
       Initialized ONCE the first time this function is called. */
    static int call_count = 0;
    call_count++;
    printf("This function has been called %d time(s)\n", call_count);
}

int main(void) {
    count_calls();   /* Called 1 time(s) */
    count_calls();   /* Called 2 time(s) */
    count_calls();   /* Called 3 time(s) */
    return 0;
}
This function has been called 1 time(s)
This function has been called 2 time(s)
This function has been called 3 time(s)

static global — file-scope linkage

/* file: module_a.c */
static int private_counter = 0;   /* ONLY visible within module_a.c */
                                   /* Other .c files cannot access this */

int internal_helper(void) {
    return ++private_counter;      /* fine: same file */
}

/* file: module_b.c */
/* extern int private_counter; */ /* would give linker error: undefined symbol */

volatile — don't cache this in a register

volatile tells the compiler: "every access to this variable must go to actual memory; do not cache it in a register or reorder reads/writes around it." Without it, the compiler's optimizer might eliminate what it sees as redundant reads of a variable that "hasn't changed" — breaking code that depends on hardware registers, signals, or shared memory written by another thread. 

#include <stdio.h>
#include <stdint.h>

/* Memory-mapped hardware register at a fixed address */
volatile uint32_t * const STATUS_REG = (volatile uint32_t *)0x40001000;

void wait_for_ready(void) {
    /* Without volatile: optimizer might hoist the read out of the loop
       (since it looks like STATUS_REG never changes from this thread's view)
       and turn this into an infinite loop or eliminate it entirely.
       With volatile: every loop iteration re-reads from address 0x40001000. */
    while ((*STATUS_REG & 0x1) == 0) {
        /* spin-wait: status bit 0 set by hardware when ready */
    }
}

extern — declaration without definition

/* file: globals.c */
int shared_counter = 0;            /* definition lives here */

/* file: worker.c */
extern int shared_counter;         /* declaration: "it exists somewhere else" */
void increment(void) {
    shared_counter++;              /* linker resolves this reference */
}
CUDA connection — how C storage classes map to GPU memory spaces:
C Storage Class CUDA Equivalent GPU Memory Space Scope / Lifetime
const __constant__ Constant memory (64KB, cached) Read-only from all threads; program lifetime
static (local) __shared__ Shared memory (on-chip, per SM) Block lifetime; all threads in block see it
global / extern __device__ Global memory (GPU DRAM) Program lifetime; all threads on device
volatile volatile __shared__ Shared memory Required for warp-level synchronization patterns
__shared__ is like static: it does not live on the per-thread stack; it lives in a shared region. __constant__ is like a const global: read-only during kernel execution, broadcast-cached across threads in a warp. volatile __shared__ is critical for warp-level reductions without __syncthreads(): without volatile, the compiler can keep values in registers and threads in the same warp may not see each other's writes.

4. Struct Layout and Padding

The compiler does not pack struct members as tightly as possible. It aligns each member to a boundary equal to that member's size (or the platform's maximum alignment, whichever is smaller). Misaligned accesses are slower (or illegal) on most hardware, so the compiler inserts padding bytes to prevent them.

Alignment rules in practice

#include <stdio.h>
#include <stddef.h>   /* for offsetof() */

struct Padded {
    char   a;    /* 1 byte  at offset  0 */
                 /* 3 bytes PADDING     -- int must start at offset divisible by 4 */
    int    b;    /* 4 bytes at offset  4 */
    char   c;    /* 1 byte  at offset  8 */
                 /* 7 bytes PADDING     -- double must start at offset divisible by 8 */
    double d;    /* 8 bytes at offset 16 */
};               /* Total: 24 bytes. Sum of members: 1+4+1+8 = 14. Wasted: 10! */

struct Packed {
    double d;    /* 8 bytes at offset  0 */
    int    b;    /* 4 bytes at offset  8 */
    char   a;    /* 1 byte  at offset 12 */
    char   c;    /* 1 byte  at offset 13 */
                 /* 2 bytes PADDING     -- struct size must be multiple of largest member (8) */
};               /* Total: 16 bytes. Same members, better order. Saved 8 bytes per instance! */

struct ForcedPacked {
    char   a;
    int    b;
    char   c;
    double d;
} __attribute__((packed));   /* GCC: disable padding entirely */
                             /* sizeof == 14 but causes unaligned accesses (SLOW or ILLEGAL) */

int main(void) {
    printf("sizeof(Padded)       = %zu\n", sizeof(struct Padded));       /* 24 */
    printf("sizeof(Packed)       = %zu\n", sizeof(struct Packed));       /* 16 */
    printf("sizeof(ForcedPacked) = %zu\n", sizeof(struct ForcedPacked)); /* 14 */

    printf("\nPadded offsets:\n");
    printf("  a at offset %zu\n", offsetof(struct Padded, a));   /* 0  */
    printf("  b at offset %zu\n", offsetof(struct Padded, b));   /* 4  */
    printf("  c at offset %zu\n", offsetof(struct Padded, c));   /* 8  */
    printf("  d at offset %zu\n", offsetof(struct Padded, d));   /* 16 */

    printf("\nPacked offsets:\n");
    printf("  d at offset %zu\n", offsetof(struct Packed, d));   /* 0  */
    printf("  b at offset %zu\n", offsetof(struct Packed, b));   /* 8  */
    printf("  a at offset %zu\n", offsetof(struct Packed, a));   /* 12 */
    printf("  c at offset %zu\n", offsetof(struct Packed, c));   /* 13 */

    return 0;
}

AOS vs SOA: how struct padding kills GPU performance

Imagine a particle simulation with 10 million particles, each having position (x, y, z) and mass. There are two ways to lay this out in memory: 

/* Array of Structures (AOS) -- common in CPU code */
struct Particle {
    float x, y, z;   /* 12 bytes */
    float mass;       /* 4 bytes  */
};                    /* 16 bytes per particle, nicely packed here */

struct Particle particles_aos[10000000];
/* Memory layout: [x0,y0,z0,m0] [x1,y1,z1,m1] [x2,y2,z2,m2] ... */


/* Structure of Arrays (SOA) -- preferred for GPU kernels */
struct ParticlesSoA {
    float *x;      /* array of 10M x-coordinates */
    float *y;      /* array of 10M y-coordinates */
    float *z;      /* array of 10M z-coordinates */
    float *mass;   /* array of 10M masses         */
};
/* Memory layout: [x0,x1,x2,...] [y0,y1,y2,...] [z0,z1,z2,...] [m0,m1,m2,...] */


/* GPU kernel that only needs x and mass: */
__global__ void scale_x_by_mass_AOS(struct Particle *p, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        /* Thread i loads p[i].x (needed) AND p[i].y, p[i].z (NOT needed)
           because the whole 16-byte struct gets pulled into cache.
           Neighboring threads load: p[i].x, p[i+1].x, p[i+2].x ...
           These are 16 bytes apart -- NOT coalesced! */
        p[i].x *= p[i].mass;
    }
}

__global__ void scale_x_by_mass_SOA(float *x, float *mass, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        /* Thread i loads x[i], thread i+1 loads x[i+1] -- consecutive addresses.
           This IS coalesced: 128-byte memory transaction serves 32 threads at once.
           Up to 4x better memory bandwidth utilization! */
        x[i] *= mass[i];
    }
}
AOS vs SOA is one of the most impactful GPU optimizations you can make.
On the GPU, 32 threads in a warp execute simultaneously. If thread 0 needs x[0] and thread 1 needs x[1], those are 4 bytes apart — perfect for a coalesced 128-byte transaction. But if they are accessing AOS structs with 16 bytes of padding between each x, each thread falls in a different 128-byte cache line and you get 32 separate memory transactions instead of 1. That is a 32x memory bandwidth penalty. In practice, converting a large simulation from AOS to SOA often yields 3–5x speedup with zero algorithmic change.

5. The CUDA_CHECK Macro Pattern

This is the section that will save you the most debugging time in your CUDA career. CUDA API calls return error codes. They do not throw exceptions. They do not print anything to the terminal. If you do not check the return value of cudaMalloc(), and it fails because you requested more memory than the GPU has, your program will continue executing with a null pointer. Every subsequent kernel launch will silently produce garbage or do nothing. You will spend hours debugging phantom bugs caused by an undetected error on line 3.

Why errors are silent by default

#include <cuda_runtime.h>
#include <stdio.h>

/* BAD: no error checking */
void bad_example(void) {
    float *d_ptr;
    /* Request 100GB -- will fail on any GPU, silently */
    cudaMalloc((void**)&d_ptr, 100ULL * 1024 * 1024 * 1024);
    /* d_ptr is still NULL or garbage. No error printed. Program continues. */
    /* Kernel launched with null pointer: silent undefined behavior */
    /* some_kernel<<<1,1>>>(d_ptr); */  /* reads/writes address 0x0 on GPU */
}

The CUDA_CHECK macro

#include <cuda_runtime.h>
#include <stdio.h>
#include <stdlib.h>

/* Full error-checking macro for CUDA API calls */
#define CUDA_CHECK(call)                                                    \
    do {                                                                    \
        cudaError_t _err = (call);                                          \
        if (_err != cudaSuccess) {                                          \
            fprintf(stderr,                                                 \
                    "CUDA error at %s:%d  code=%d (%s)\n",                  \
                    __FILE__, __LINE__,                                      \
                    (int)_err, cudaGetErrorString(_err));                   \
            exit(EXIT_FAILURE);                                             \
        }                                                                   \
    } while (0)

/* Separate macro for checking kernel launch errors.
   Kernels are ASYNCHRONOUS -- errors don't appear immediately.
   cudaGetLastError() pops the error from the internal error queue.
   cudaDeviceSynchronize() waits for all kernels to finish and
   surfaces any async errors that occurred during execution. */
#define CUDA_CHECK_KERNEL()                                                 \
    do {                                                                    \
        cudaError_t _err = cudaGetLastError();                              \
        if (_err != cudaSuccess) {                                          \
            fprintf(stderr,                                                 \
                    "Kernel launch error at %s:%d  code=%d (%s)\n",         \
                    __FILE__, __LINE__,                                      \
                    (int)_err, cudaGetErrorString(_err));                   \
            exit(EXIT_FAILURE);                                             \
        }                                                                   \
        _err = cudaDeviceSynchronize();                                     \
        if (_err != cudaSuccess) {                                          \
            fprintf(stderr,                                                 \
                    "Kernel execution error at %s:%d  code=%d (%s)\n",      \
                    __FILE__, __LINE__,                                      \
                    (int)_err, cudaGetErrorString(_err));                   \
            exit(EXIT_FAILURE);                                             \
        }                                                                   \
    } while (0)

Correct usage wrapping every CUDA API call

__global__ void vectorAdd(const float *a, const float *b,
                          float *c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) c[i] = a[i] + b[i];
}

int main(void) {
    const int N = 1 << 20;   /* 1M elements */
    size_t bytes = N * sizeof(float);

    float *h_a = (float*)malloc(bytes);
    float *h_b = (float*)malloc(bytes);
    float *h_c = (float*)malloc(bytes);

    /* Initialize host arrays ... */
    for (int i = 0; i < N; i++) { h_a[i] = 1.0f; h_b[i] = 2.0f; }

    float *d_a, *d_b, *d_c;

    /* EVERY CUDA call wrapped in CUDA_CHECK */
    CUDA_CHECK( cudaMalloc((void**)&d_a, bytes) );
    CUDA_CHECK( cudaMalloc((void**)&d_b, bytes) );
    CUDA_CHECK( cudaMalloc((void**)&d_c, bytes) );

    CUDA_CHECK( cudaMemcpy(d_a, h_a, bytes, cudaMemcpyHostToDevice) );
    CUDA_CHECK( cudaMemcpy(d_b, h_b, bytes, cudaMemcpyHostToDevice) );

    int block = 256;
    int grid  = (N + block - 1) / block;
    vectorAdd<<<grid, block>>>(d_a, d_b, d_c, N);
    CUDA_CHECK_KERNEL();   /* catches launch config errors AND execution errors */

    CUDA_CHECK( cudaMemcpy(h_c, d_c, bytes, cudaMemcpyDeviceToHost) );

    printf("h_c[0] = %.1f (expected 3.0)\n", h_c[0]);

    CUDA_CHECK( cudaFree(d_a) );
    CUDA_CHECK( cudaFree(d_b) );
    CUDA_CHECK( cudaFree(d_c) );
    free(h_a); free(h_b); free(h_c);
    return 0;
}

The do-while(0) wrapping is not optional.
Consider: if (cond) CUDA_CHECK(call); else do_something();
If the macro expands to bare if (...) { ... }, the preprocessor will bind the else to the inner if, not to your outer if (cond). The do { ... } while(0) pattern turns the macro into a single statement that requires a semicolon and never confuses the parser. Always use this pattern for multi-statement macros.

6. Function Pointers and Callbacks

A function pointer stores the address of a function in the text (code) segment. It lets you pass behavior as data — the foundation of callbacks, dispatch tables, and runtime polymorphism in C.

Syntax: declaration and use

#include <stdio.h>

/* Function pointer type: "pointer to a function taking int, returning int" */
typedef int (*transform_fn)(int);

/* Two functions matching that signature */
int double_it(int x) { return x * 2; }
int square_it(int x) { return x * x; }
int negate_it(int x) { return -x;    }

/* apply() takes a function pointer and calls it on each element */
void apply(int *arr, int n, transform_fn fn) {
    for (int i = 0; i < n; i++) {
        arr[i] = fn(arr[i]);    /* call through the pointer */
    }
}

/* Dispatch table: array of function pointers */
transform_fn ops[] = { double_it, square_it, negate_it };
const char  *names[] = { "double", "square", "negate" };

int main(void) {
    int data[5] = {1, 2, 3, 4, 5};

    apply(data, 5, double_it);
    printf("After double: %d %d %d %d %d\n",
           data[0], data[1], data[2], data[3], data[4]);
    /* 2 4 6 8 10 */

    /* Store a function pointer in a variable */
    transform_fn chosen = square_it;
    for (int i = 0; i < 5; i++) data[i] = chosen(data[i]);
    printf("After square: %d %d %d %d %d\n",
           data[0], data[1], data[2], data[3], data[4]);
    /* 4 16 36 64 100 */

    /* Dispatch table: select operation by index */
    int op_index = 2;   /* negate */
    for (int i = 0; i < 5; i++) data[i] = ops[op_index](data[i]);
    printf("After %s: %d %d %d %d %d\n", names[op_index],
           data[0], data[1], data[2], data[3], data[4]);
    /* After negate: -4 -16 -36 -64 -100 */

    return 0;
}

Device function pointers in CUDA (Compute Capability 2.0+)

CUDA supports function pointers in device code starting with Fermi (compute capability 2.0). Device function pointers point into the GPU's instruction memory and can be stored in device-side arrays for dispatch tables on the GPU. 

#include <cuda_runtime.h>
#include <stdio.h>

/* __device__ functions: run on GPU, callable from GPU only */
__device__ float add_one(float x)  { return x + 1.0f; }
__device__ float times_two(float x){ return x * 2.0f; }
__device__ float square(float x)   { return x * x;    }

/* Typedef for a device function pointer */
typedef float (*DeviceOp)(float);

/* Store device function addresses at program startup */
__device__ DeviceOp d_ops[3];

__global__ void init_ops(void) {
    /* This kernel runs on GPU and captures device function addresses */
    d_ops[0] = add_one;
    d_ops[1] = times_two;
    d_ops[2] = square;
}

__global__ void apply_op(float *data, int n, int op_index) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        /* Dispatch through device function pointer */
        data[i] = d_ops[op_index](data[i]);
    }
}

/* Note: device function pointers are rarely needed in practice.
   The compiler inlines __device__ functions by default, which is
   almost always what you want. Function pointers prevent inlining
   and add indirection overhead. Use them only when you genuinely
   need runtime dispatch on the GPU. */
The more important CUDA "function pointer" pattern: __global__ as a dispatch:
A CUDA kernel launch kernel<<<grid, block>>>(args) is conceptually similar to calling a function pointer. nvcc compiles __global__ functions to PTX/SASS and the CUDA driver dispatches them to the GPU. Template kernels let you select specializations at compile time, which is more efficient than device function pointers at runtime. The __host__ __device__ qualifier compiles a function for both CPU and GPU, useful for utility math functions shared between host setup code and device kernels. 
/* Compiled for BOTH host and device */
__host__ __device__ inline float clamp(float v, float lo, float hi) {
    return v < lo ? lo : (v > hi ? hi : v);
}

7. Memory Model: Strict Aliasing and __restrict__

The C/C++ strict aliasing rule says: the compiler is allowed to assume that pointers of different types do not point to the same memory. This lets the optimizer reorder or eliminate loads and stores that it can prove do not conflict. When the rule is violated — when two pointers of different types do alias — the result is undefined behavior and the optimizer may generate subtly wrong code.

The aliasing problem in GPU kernels

Consider a simple vectorAdd kernel. The compiler sees that you write to c[i] and then (in the next iteration) read from a[i+1] and b[i+1]. Without a guarantee that c doesn't overlap with a or b, the compiler must be conservative: it cannot reorder those operations or cache the loads in registers across iterations, because writing to c[i] might change the value of a[i+1] (if they point to the same memory). 

/* Without __restrict__: compiler must assume a, b, c could overlap */
__global__ void vectorAdd_unsafe(float *a, float *b, float *c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        /* Compiler cannot cache a[i] or b[i] in registers across statements
           because c[i] write might alias them. Conservative code generation. */
        c[i] = a[i] + b[i];
    }
}

/* With __restrict__: promise that a, b, c do NOT overlap */
__global__ void vectorAdd_restrict(const float * __restrict__ a,
                                   const float * __restrict__ b,
                                         float * __restrict__ c,
                                   int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        /* Compiler KNOWS: writing c[i] cannot affect future reads of a or b.
           It can now:
           - Pre-load a[i] and b[i] into registers before the addition
           - Reorder memory operations for better memory-level parallelism
           - Vectorize across elements if loop-unrolling is used
           - Generate LDCS (cache streaming) loads for better cache utilization */
        c[i] = a[i] + b[i];
    }
}

Practical performance difference

/* More dramatic example: reduction-like kernel */
__global__ void saxpy_no_restrict(float *y, const float *x, float a, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    /* y[i] = a * x[i] + y[i]
       Without restrict: compiler must re-read y[i] on the right-hand side
       every iteration because it might alias x. Even in this single statement,
       the compiler may not cache y[i] in a register across the load and store. */
    if (i < n) y[i] = a * x[i] + y[i];
}

__global__ void saxpy_restrict(float * __restrict__ y,
                               const float * __restrict__ x,
                               float a, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    /* Now the compiler knows x and y don't overlap.
       It can:
       - Load x[i] with LDCS (streaming, bypasses L1 since x is read-once)
       - Load y[i] with LDCA (cached, since y is read+write)
       - Generate fused multiply-add (FMA) instruction
       Measured difference on A100: ~8% better effective bandwidth */
    if (i < n) y[i] = a * x[i] + y[i];
}

Strict aliasing with different types

#include <string.h>

/* UNDEFINED BEHAVIOR: violates strict aliasing */
float f = 3.14f;
int i_alias = *(int*)&f;   /* reinterpret float bytes as int -- UB in C! */
                            /* The compiler may assume float* and int* never alias
                               and generate code that does NOT read the float bytes */

/* DEFINED BEHAVIOR: use memcpy for type-punning */
int i_safe;
memcpy(&i_safe, &f, sizeof(float));   /* always correct, compiler optimizes to register move */

/* ALSO DEFINED: char* can alias anything (special exemption in C standard) */
unsigned char *bytes = (unsigned char*)&f;   /* legal: inspect raw bytes */
for (int j = 0; j < 4; j++)
    printf("byte %d: 0x%02x\n", j, bytes[j]);
Always add __restrict__ to kernel pointer parameters when the arrays genuinely don't overlap.
The CUDA compiler (nvcc / ptxas) uses restrict information to generate better memory access instructions (LDCS vs LDCA), reorder loads for better memory-level parallelism, and sometimes vectorize. The performance gain varies from a few percent to 15%+ depending on the kernel. The cost is zero if the pointers truly don't overlap. The cost of not adding it is that you leave performance on the table on every kernel in your codebase. Make it a habit: if your kernel reads from input arrays and writes to an output array with no overlap, add __restrict__ to all of them.

restrict in a grid-stride loop (production pattern)

/* Production-quality kernel combining all best practices from this post */
__global__ void elementwise_relu(const float * __restrict__ input,
                                       float * __restrict__ output,
                                 int n) {
    /* Grid-stride loop: handles any N regardless of grid size */
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
             i < n;
             i += blockDim.x * gridDim.x) {
        /* __restrict__ allows compiler to cache input[i] without re-reading.
           const prevents accidental writes to input.
           Both are cheap to add and help the optimizer. */
        float val = input[i];
        output[i] = val > 0.0f ? val : 0.0f;
    }
}

/* Caller side -- all best practices in use */
void run_relu(float *d_in, float *d_out, int n) {
    int block = 256;
    int grid  = min((n + block - 1) / block, 65535);   /* cap at CUDA grid limit */
    elementwise_relu<<<grid, block>>>(d_in, d_out, n);
    CUDA_CHECK_KERNEL();
}

Checklist Before Writing Your First Kernel

  • Can you explain what & and * do, without looking it up?
  • Can you explain why cudaMalloc takes void**?
  • Do you know the difference between stack and heap, and which one CUDA uses?
  • Can you explain why the GPU cannot use a pointer returned by malloc()?
  • Do you know what static inside a function does to the variable's lifetime?
  • Do you know what volatile prevents the compiler from doing?
  • Can you rearrange a struct's fields to minimize its size?
  • Can you explain why AOS is usually bad for GPU kernels and SOA is usually better?
  • Can you write the CUDA_CHECK macro from memory?
  • Do you know why kernels need a separate error-check after cudaDeviceSynchronize()?
  • Can you explain what __restrict__ promises the compiler?

If you can answer all of these, you have the C foundation to write, debug, and optimize real CUDA kernels. Takeaways

  • A pointer is a numeric address. Dereferencing reads or writes the bytes at that address. Pointer arithmetic scales by sizeof(type).
  • cudaMalloc((void**)&ptr, size) uses double-pointer so it can write the device address into your variable.
  • Stack is automatic, fast, ~8 MB limit. Heap is manual, large, requires free(). CUDA device memory is a separate heap that only GPU code can address.
  • const prevents writes. static local persists across calls (lives in BSS, not stack). volatile prevents compiler from caching the value in a register. extern is a declaration without a definition.
  • Struct padding can significantly inflate struct size. Order members largest-first to minimize wasted bytes. Use offsetof() to inspect layout.
  • AOS layout hurts GPU memory coalescing; SOA layout where each field is its own array nearly always wins on GPUs.
  • Wrap every CUDA API call in CUDA_CHECK(). Check kernel errors with cudaGetLastError() + cudaDeviceSynchronize(). Silent errors are the most dangerous bugs in CUDA.
  • Add const __restrict__ to read-only kernel pointer parameters to enable better compiler optimization and memory access patterns.
