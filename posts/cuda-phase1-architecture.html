CUDA Mastery Phase 1.2 — CPU vs GPU Architecture: Why Parallelism Changes Everything

Why CPUs and GPUs are designed so differently, how memory bandwidth determines GPU performance, and the fundamental mental model shift you need to think in parallel — with real numbers and architecture diagrams.

Most programmers spend years writing CPU code and never question the assumption that programs run one instruction at a time, sequentially. Then they hit CUDA and everything breaks. The reason is architectural: CPUs and GPUs were designed to solve completely different problems. Understanding why changes how you think about every kernel you write.

1. CPU Architecture — Built for Latency

A modern CPU like an Intel Core i9 or AMD Ryzen 9 has 16–32 physical cores. Each core is a masterpiece of latency-hiding engineering:

  • Out-of-order execution: instructions execute in whatever order avoids stalls, not source order
  • Branch prediction: speculatively executes likely branches before the condition is evaluated
  • Deep cache hierarchy: L1 (32–64KB, ~4 cycles), L2 (256KB–1MB, ~12 cycles), L3 (8–64MB, ~40 cycles)
  • Superscalar execution: multiple instruction units can execute in parallel per core
  • SIMD (AVX-512): 512-bit vector registers for data parallelism
  CPU Architecture (simplified):

  +------------------------------------------------------------+
  |                     CPU Die                                |
  |                                                            |
  |  +----------+  +----------+  +----------+  +----------+   |
  |  | Core 0   |  | Core 1   |  | Core 2   |  | Core 3   |   |
  |  | [L1 32KB]|  | [L1 32KB]|  | [L1 32KB]|  | [L1 32KB]|   |
  |  | [L2 1MB] |  | [L2 1MB] |  | [L2 1MB] |  | [L2 1MB] |   |
  |  | OOO exec |  | OOO exec |  | OOO exec |  | OOO exec |   |
  |  | branch   |  | branch   |  | branch   |  | branch   |   |
  |  | predict  |  | predict  |  | predict  |  | predict  |   |
  |  +----------+  +----------+  +----------+  +----------+   |
  |                                                            |
  |  +------------------------------------------------------+  |
  |  |              Shared L3 Cache (32MB)                  |  |
  |  +------------------------------------------------------+  |
  |                                                            |
  |  +------------------------------------------------------+  |
  |  |              Memory Controller                       |  |
  |  +------------------------------------------------------+  |
  +------------------------------------------------------------+
            |
  +---------+---------+
  | DDR5 RAM (192GB/s)|
  +-------------------+

The key insight: a CPU core has enormous hardware dedicated to running a single thread as fast as possible. All that out-of-order logic, branch predictors, and large caches exist to minimize the latency any single instruction stream experiences.

2. GPU Architecture — Built for Throughput

An NVIDIA A100 GPU has 6912 CUDA cores organized into 108 Streaming Multiprocessors (SMs). Each SM is far simpler than a CPU core — it lacks the complex latency-hiding hardware. Instead, it hides latency a different way: by switching between thousands of in-flight threads. 

  GPU Architecture (simplified A100):

  +----------------------------------------------------------------+
  |                          A100 GPU                              |
  |                                                                |
  |  SM0  SM1  SM2  SM3  ...  SM107                               |
  |  +--+  +--+  +--+  +--+   +--+                                |
  |  |  |  |  |  |  |  |  |   |  |   Each SM contains:           |
  |  |  |  |  |  |  |  |  |   |  |   - 64 CUDA cores (FP32)      |
  |  +--+  +--+  +--+  +--+   +--+   - 4 Tensor Core units       |
  |                                   - 48KB L1 / shared memory   |
  |  108 SMs × 64 cores = 6912 CUDA cores total                   |
  |                                                                |
  |  +----------------------------------------------------------+  |
  |  |           L2 Cache (40MB on A100)                        |  |
  |  +----------------------------------------------------------+  |
  |                                                                |
  |  +----------------------------------------------------------+  |
  |  |           HBM2e Memory (80GB, 2 TB/s)                    |  |
  |  +----------------------------------------------------------+  |
  +----------------------------------------------------------------+
  Inside one SM (Streaming Multiprocessor):

  +------------------------------------------------------+
  |                 Streaming Multiprocessor             |
  |                                                      |
  |  Warp Scheduler  Warp Scheduler  (4 schedulers)      |
  |  [issues 1 inst/cycle to a warp of 32 threads]       |
  |                                                      |
  |  +--------+  +--------+  +--------+  +--------+      |
  |  | FP32   |  | FP32   |  | INT32  |  | Tensor |      |
  |  | 16 cores|  | 16 cores|  | 16 cores|  | Core   |  |
  |  +--------+  +--------+  +--------+  +--------+      |
  |                                                      |
  |  Register File (65536 × 32-bit registers)            |
  |  Shared Memory / L1 Cache (48KB + 16KB, or 32+32)   |
  |                                                      |
  |  Active warps: up to 64 warps (2048 threads)        |
  +------------------------------------------------------+

The GPU's secret weapon is massive multithreading. When one warp stalls waiting for global memory (~800 cycle latency), the SM instantly switches to another ready warp. With 64 active warps per SM, the SM can stay busy as long as enough warps are always ready to execute.

3. Latency vs Throughput — The Design Philosophy

This is the single most important mental model shift: 

  CPU philosophy: MINIMIZE LATENCY for one thread
  ─────────────────────────────────────────────────
  - Big caches to avoid waiting for RAM
  - Branch prediction to avoid wasted cycles
  - Out-of-order execution to reorder around stalls
  - Few, powerful cores

  Result: One task in 1 microsecond.


  GPU philosophy: MAXIMIZE THROUGHPUT across many threads
  ────────────────────────────────────────────────────────
  - Small caches (latency is EXPECTED, not avoided)
  - No branch prediction (just predicate both paths)
  - In-order execution (simplicity = more cores)
  - Many, simple cores

  Result: 10,000 tasks in 10 microseconds (1000x the work,
          only 10x the time = 100x the throughput).

4. Memory Bandwidth — The Real Bottleneck

GPU performance is almost always limited by memory bandwidth, not compute. The A100 has 2 TB/s of HBM2 bandwidth. A CPU has ~200 GB/s of DDR5 bandwidth. That 10x difference is why GPUs dominate memory-intensive workloads. 

  Memory Bandwidth Comparison:
  ─────────────────────────────────────────────────────────
  DDR5 (CPU) :  ████                          ~200 GB/s
  GDDR6X     :  ████████████                  ~600 GB/s  (RTX 3090)
  HBM2e (A100):  ████████████████████████████  ~2000 GB/s
  ─────────────────────────────────────────────────────────
  Each '█' represents ~70 GB/s

Why does HBM (High Bandwidth Memory) achieve such high bandwidth? It's stacked directly on the GPU die with a very wide bus (5120-bit on A100), whereas CPU memory uses a narrow 64-bit DDR bus. More lanes = more data per clock.

5. The Memory Wall and the Roofline Model

Any computation has two potential bottlenecks: compute (FLOPs/sec) and memory bandwidth (bytes/sec). The Roofline model captures this: 

  Performance = min(Peak FLOPS,  Bandwidth × Arithmetic_Intensity)

  Arithmetic Intensity = FLOPs performed / bytes transferred from memory


  Roofline plot:

  Performance
  (TFLOPS)
    │
 20 │                            ─ ─ ─ ─ ─ ─ ─ Peak compute (A100: 19.5T)
    │                       ╱─────────────────────────────────
    │                  ╱───╱
    │             ╱───╱
 10 │        ╱───╱
    │   ╱───╱   ← memory-bound      compute-bound →
    │───╱
    └─────────────────────────────────────────────── Arithmetic Intensity
    0   1   2   4   8  16  32  64  (FLOPs/byte)
                   ↑
             Ridge point (~10 FLOPs/byte for A100)
  • Left of the ridge point: memory-bound. More bandwidth = more performance.
  • Right of the ridge point: compute-bound. More FLOPS = more performance.
  • Vector addition: ~0.5 FLOPs/byte → deeply memory-bound.
  • Matrix multiplication: ~100+ FLOPs/byte → compute-bound.

6. SIMD vs SIMT — Two Approaches to Data Parallelism

  SIMD (CPU): Single Instruction, Multiple Data
  ───────────────────────────────────────────────
  One instruction, multiple data elements in a vector register.
  All elements MUST execute the same instruction (no masking at instruction level).

  AVX-512: 512 bits / 32 bits per float = 16 floats simultaneously
  C code: float a[16] + float b[16] → one _mm512_add_ps() intrinsic

  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
  │ f0  │ f1  │ f2  │ f3  │ f4  │ f5  │ f6  │ f7  │  ... (16 wide)
  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
       All 16 elements: same op, same cycle


  SIMT (GPU): Single Instruction, Multiple Threads
  ─────────────────────────────────────────────────
  32 threads in a warp execute the same instruction in lockstep.
  But each thread has its OWN program counter and registers.
  Divergent branches are handled by predication (inactive threads masked out).

  Warp of 32 threads:
  T0  T1  T2  T3  ... T31
  All execute: fadd r1, r2, r3  (each on their own r1, r2, r3 values)

  Key difference: SIMT hides the vector width from the programmer.
  You write scalar code; the hardware batches 32 threads.

7. Amdahl's Law — Limits of GPU Speedup

Amdahl's Law says that speedup is limited by the serial portion of your code: 

  Speedup = 1 / (serial_fraction + parallel_fraction / N)

  If 10% of your code is serial (not GPU-parallelizable):
  Maximum possible speedup = 1 / 0.10 = 10x  (regardless of GPU size!)

  serial fraction:   1%  →  max speedup: 100x
                    10%  →  max speedup:  10x
                    25%  →  max speedup:   4x
                    50%  →  max speedup:   2x

This has a practical implication: if your pipeline has a 10ms CPU pre-processing step and a 1ms GPU kernel, optimizing the GPU further won't help — the CPU is your bottleneck. Profile both CPU and GPU. Don't assume the GPU is the bottleneck.

8. PCIe Bandwidth — The CPU↔GPU Transfer Tax

CPU and GPU sit on opposite ends of a PCIe bus. Every cudaMemcpy() pays this tax: 

  PCIe Bandwidth:
  ─────────────────────────────────────────────────────
  PCIe 4.0 x16:  ~32 GB/s bidirectional
  PCIe 5.0 x16:  ~64 GB/s bidirectional
  NVLink (A100):  ~600 GB/s (GPU↔GPU, not CPU↔GPU)

  Comparison:
  GPU global memory bandwidth:  2000 GB/s
  PCIe 4.0 to CPU:                32 GB/s   ← 62x slower!
  ─────────────────────────────────────────────────────

  Implication: transferring data over PCIe is EXPENSIVE.
  Minimize host↔device transfers. Keep data on GPU as long as possible.
  Common mistake: transfer data back to CPU every iteration
  ─────────────────────────────────────────────────────────
  for each frame:
      CPU → GPU: upload frame        (PCIe, expensive)
      GPU: process frame             (fast, in GPU memory)
      GPU → CPU: download result     (PCIe, expensive!)  ← avoid if CPU doesn't need it
      CPU: upload to display         (more PCIe)

  Better: keep results on GPU, only transfer final output
  Or: use GPU-to-display paths (OpenGL/Vulkan interop)

9. When NOT to Use a GPU

GPUs are not always faster. Understanding when to use a CPU is as important as knowing how to write CUDA. 

  Use GPU when:                    Use CPU when:
  ──────────────────────────       ──────────────────────────────
  ✓ Data parallelism dominates     ✗ Task is inherently serial
  ✓ Same op on millions of items   ✗ Lots of branching/control flow
  ✓ Large data (fills GPU memory)  ✗ Small data (transfer overhead dominates)
  ✓ Regular memory access          ✗ Irregular/pointer-chasing access
  ✓ Compute-intensive math         ✗ Low arithmetic intensity
  ✓ Can tolerate PCIe transfer     ✗ Need ultra-low latency (<1ms)
  ✓ Batch processing               ✗ Online/interactive processing

Classic examples where GPU loses: graph traversal (BFS/DFS — pointer chasing, irregular access), parsing (lots of branching), small neural network inference (transfer overhead exceeds compute time), and anything where N < 1000.

10. The Mental Model You Need

Before writing any CUDA kernel, ask:

  1. Can this work be split into independent parallel tasks? (embarrassingly parallel?)
  2. Does each task do enough work to justify thread launch overhead?
  3. Is the memory access pattern friendly? (coalesced, not random)
  4. Is my data already on the GPU, or will transfer cost dominate?
  5. What fraction of total runtime is this step? (Amdahl's law check)
The GPU is a throughput machine. It's happy when you give it 10 million simple, independent, regular tasks. It struggles with serial dependencies, irregular data access, and lots of branch conditions.

Takeaways

  • CPUs optimize for latency (one thread fast); GPUs optimize for throughput (all threads eventually).
  • GPU hides memory latency by keeping thousands of warps active and switching between them.
  • Memory bandwidth (2 TB/s on A100) is typically the GPU bottleneck, not compute.
  • SIMT: 32 threads execute the same instruction in lockstep — this is the warp.
  • Amdahl's Law caps your speedup at 1/serial_fraction — eliminate serial bottlenecks first.
  • PCIe is 62x slower than GPU memory — minimize data transfers.
  • GPUs lose on irregular access, small data, serial code, and latency-sensitive tasks.
