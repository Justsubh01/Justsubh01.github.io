CUDA Mastery Phase 6 — NVIDIA Interview Prep: Questions, Coding Problems, and What to Study

The complete preparation guide for NVIDIA and GPU-computing interviews — 25+ conceptual Q&As, full solutions to 6 coding problems, system design patterns, and the study schedule that gets you ready in 4 weeks.

NVIDIA interviews for GPU software roles are deep. They expect you to reason about hardware at the warp level, write optimized kernels from scratch, and design full GPU pipelines. I've compiled the most common questions from public sources, Discord communities, and my own prep. This is the guide I wish I had.

Part 1 — Conceptual Questions

Q1: What is a warp? How many threads does it contain?

A warp is the fundamental execution unit of the GPU: a group of 32 consecutive threads from the same block that execute the same instruction in lockstep (SIMT). The warp size is always 32 on all current NVIDIA GPU architectures. All performance concepts — coalescing, divergence, occupancy — are defined at the warp level.

Q2: What is warp divergence? How do you detect and avoid it?

Warp divergence occurs when threads in the same warp take different paths through an if/else branch. The hardware executes both paths sequentially, masking out inactive threads — up to 32× slower in the worst case. Detect it with ncu --metrics smsp__sass_branch_targets_threads_divergent.sum. Avoid it by: sorting data so warps contain homogeneous values, branching on blockIdx instead of threadIdx, or restructuring algorithms to eliminate conditional branches.

Q3: Explain memory coalescing. What determines a coalesced access?

When a warp issues a global memory access, the hardware coalesces all 32 thread requests into as few 128-byte cache line transactions as possible. A perfectly coalesced access occurs when 32 consecutive threads access 32 consecutive addresses aligned to a 128-byte boundary — resulting in a single transaction. Strided or random access requires multiple transactions, wasting bandwidth. Check with smsp__sass_average_data_bytes_per_sector_mem_global_op_ld.pct in Nsight Compute.

Q4: What is the difference between shared memory and global memory?

  Property      | Shared Memory          | Global Memory
  ──────────────┼────────────────────────┼────────────────────────
  Location      | On-chip (SM)           | Off-chip (DRAM)
  Latency       | ~5 cycles              | ~800 cycles
  Bandwidth     | ~10+ TB/s              | ~2 TB/s (A100)
  Size          | 48KB per block         | GBs (80GB on A100)
  Scope         | Block only             | All threads, host
  Managed by    | Programmer             | Programmer (cudaMalloc)
  Declaration   | __shared__             | cudaMalloc / __device__
  Lifetime      | Kernel                 | Application
  ──────────────┴────────────────────────┴────────────────────────

Q5: What is occupancy? What limits it? How do you maximize it?

Occupancy is the ratio of active warps to maximum warps on an SM. Higher occupancy means more warps to hide memory latency. It's limited by: register count per thread (65536 registers / SM), shared memory per block (96KB / SM on A100), and block count per SM (32 max). Maximize it by: reducing register usage (simplify code, check ptxas output), reducing shared memory per block (use smaller tiles), choosing block sizes that fill SM capacity, and using __launch_bounds__ to give the compiler a register budget constraint.

Q6: What is a bank conflict?

Shared memory is divided into 32 banks, each 4 bytes wide. A bank conflict occurs when multiple threads in a warp access the same bank at different addresses — their accesses serialize. An N-way conflict is N× slower. Fix by padding shared memory arrays: __shared__ float data[TILE][TILE + 1] shifts column accesses to different banks.

Q7: What is the Roofline model?

The Roofline model describes the achievable performance as: Performance = min(Peak_FLOPS, Bandwidth × Arithmetic_Intensity). Arithmetic Intensity (AI) = FLOPs / bytes transferred from DRAM. If your kernel has AI less than the "ridge point" (Peak_FLOPS / Bandwidth ≈ 10 for A100), it's memory-bound; adding more compute doesn't help. If AI exceeds the ridge point, it's compute-bound; adding bandwidth doesn't help. This determines which optimizations to pursue.

Q8: When should you NOT use a GPU?

Don't use a GPU when: the workload is inherently serial (Amdahl's law limits gain), the data is too small (PCIe transfer overhead dominates computation time), the algorithm has lots of irregular branching and pointer-chasing (graphs, trees), the application requires very low latency (<1ms end-to-end), or when frequent CPU↔GPU synchronization would serialize the pipeline.

Q9: What is the difference between __syncthreads() and cudaDeviceSynchronize()?

  __syncthreads():
  • Barrier within a SINGLE BLOCK
  • All threads in the block must reach this point
  • Executes on the GPU (device side)
  • Cost: a few cycles (stalls until all threads arrive)
  • Use: after loading data into shared memory

  cudaDeviceSynchronize():
  • Waits for ALL pending GPU work (all kernels, all streams)
  • Executes on the CPU (host side — blocks CPU until GPU done)
  • Cost: hundreds of microseconds (CPU–GPU round trip)
  • Use: before reading results on CPU, for error checking

Q10: What is pinned memory and why does it matter for transfers?

Pinned (page-locked) memory is host memory that the OS guarantees will never be swapped to disk. The GPU's DMA engine can transfer directly from pinned memory without CPU involvement, enabling truly asynchronous cudaMemcpyAsync() and achieving near-PCIe-peak bandwidth (~16 GB/s vs ~8 GB/s for pageable). Allocate with cudaMallocHost(); free with cudaFreeHost(). Downside: reduces available RAM for other processes; only pin what you need.

Q11: What is arithmetic intensity? What is it for vector addition?

Arithmetic Intensity = FLOPs performed / bytes transferred from main memory. For vector addition (c = a + b): 1 FLOP (add) per element, 3 float reads/writes (12 bytes) per element. AI = 1/12 ≈ 0.083 FLOPs/byte. Vector addition is deeply memory-bound — the GPU spends most time waiting for DRAM, not computing.

Q12: How does the SM schedule warps?

Each SM has 4 warp schedulers. Each cycle, each scheduler selects one "ready" warp (not waiting for memory or other resources) and issues one instruction to its 32 threads. The SM can hold up to 64 warps simultaneously. When a warp stalls (e.g., waiting 800 cycles for global memory), the scheduler switches to another ready warp, hiding the latency. With enough active warps, the SM stays busy continuously — this is how GPUs tolerate high memory latency without large caches.

Q13: What is register pressure?

Register pressure occurs when a kernel uses more registers per thread than the hardware supports (255 on modern GPUs, but shared among all blocks on an SM). When registers overflow, the compiler spills them to "local memory" — which is actually in global DRAM — causing ~800 cycle read/write latencies for each spilled value. Check with nvcc --ptxas-options=-v; the output shows N bytes lmem if spilling occurs.

Q14: How do streams enable overlap?

CUDA streams are ordered queues of GPU operations. Operations in the same stream execute sequentially; operations in different streams can execute concurrently if hardware resources allow. By splitting data into chunks and assigning each chunk to a different stream, you can overlap H→D transfer (chunk N+1) with kernel execution (chunk N) with D→H transfer (chunk N-1) simultaneously. Requires pinned host memory for async copies.

Q15: Explain Amdahl's Law in the context of CUDA

Amdahl's Law: max speedup = 1 / (serial_fraction + parallel_fraction / N). If 10% of your code cannot be parallelized, the maximum possible GPU speedup is 10×, regardless of how many cores you throw at it. In CUDA: if your data pre-processing on CPU takes 10ms and the GPU kernel takes 1ms, optimizing the kernel further won't help — profile the whole pipeline and find the actual bottleneck.

Part 2 — Coding Problems

Problem 1: Vector Addition

// Write a kernel to compute c[i] = a[i] + b[i] for N elements.
// Handle the case where N is not a multiple of blockSize.

__global__ void vectorAdd(const float *a, const float *b, float *c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n)  // bounds check for last partial block
        c[i] = a[i] + b[i];
}

// Host launch:
void launch_vectorAdd(float *d_a, float *d_b, float *d_c, int n) {
    int blockSize = 256;
    int gridSize  = (n + blockSize - 1) / blockSize;
    vectorAdd<<<gridSize, blockSize>>>(d_a, d_b, d_c, n);
    CUDA_CHECK(cudaGetLastError());
}

Problem 2: Parallel Sum Reduction

// Reduce N floats to their sum. Use shared memory.
// This is the most commonly asked CUDA coding question.

__global__ void reduce_sum(float *data, float *partial_sums, int n) {
    extern __shared__ float sdata[];
    int tid = threadIdx.x;
    int i   = blockIdx.x * (blockDim.x * 2) + threadIdx.x;

    // Load and first add during load (handles 2 elements per thread):
    float val = 0.0f;
    if (i < n)              val  = data[i];
    if (i + blockDim.x < n) val += data[i + blockDim.x];
    sdata[tid] = val;
    __syncthreads();

    // Sequential addressing reduction:
    for (int s = blockDim.x / 2; s > 32; s >>= 1) {
        if (tid < s) sdata[tid] += sdata[tid + s];
        __syncthreads();
    }

    // Last warp: unroll with shuffle (no __syncthreads needed):
    if (tid < 32) {
        volatile float *vsdata = sdata;
        vsdata[tid] += vsdata[tid + 32];
        vsdata[tid] += vsdata[tid + 16];
        vsdata[tid] += vsdata[tid +  8];
        vsdata[tid] += vsdata[tid +  4];
        vsdata[tid] += vsdata[tid +  2];
        vsdata[tid] += vsdata[tid +  1];
    }

    if (tid == 0) partial_sums[blockIdx.x] = sdata[0];
}

// Two-pass reduction for large N:
float gpu_sum(float *d_data, int n) {
    int blockSize = 256;
    int gridSize  = (n + blockSize * 2 - 1) / (blockSize * 2);
    float *d_partial;
    cudaMalloc(&d_partial, gridSize * sizeof(float));

    reduce_sum<<<gridSize, blockSize, blockSize*sizeof(float)>>>
               (d_data, d_partial, n);

    // Reduce the partial sums (if gridSize is small enough, do on CPU):
    float *h_partial = (float*)malloc(gridSize * sizeof(float));
    cudaMemcpy(h_partial, d_partial, gridSize*sizeof(float), cudaMemcpyDeviceToHost);
    float total = 0;
    for (int i = 0; i < gridSize; i++) total += h_partial[i];
    free(h_partial);
    cudaFree(d_partial);
    return total;
}

Problem 3: Matrix Transpose (Optimized)

// Transpose an NxN matrix. Must be coalesced for both read AND write.
// Solution: use shared memory as a transposition buffer.

#define TILE 32

__global__ void transpose(float *out, const float *in, int N) {
    __shared__ float tile[TILE][TILE + 1];  // +1 to avoid bank conflicts

    // Coalesced READ: threads read a tile from input matrix (row by row)
    int x = blockIdx.x * TILE + threadIdx.x;  // input column
    int y = blockIdx.y * TILE + threadIdx.y;  // input row
    if (x < N && y < N)
        tile[threadIdx.y][threadIdx.x] = in[y * N + x];
    __syncthreads();

    // Coalesced WRITE: write the transposed tile (column → row)
    // Swap block indices for output position:
    int ox = blockIdx.y * TILE + threadIdx.x;  // output column
    int oy = blockIdx.x * TILE + threadIdx.y;  // output row
    if (ox < N && oy < N)
        out[oy * N + ox] = tile[threadIdx.x][threadIdx.y];
}

// Launch:
dim3 block(TILE, TILE);
dim3 grid((N + TILE - 1) / TILE, (N + TILE - 1) / TILE);
transpose<<<grid, block>>>(d_out, d_in, N);

Problem 4: Histogram with Shared Memory

// Compute histogram of unsigned char data (256 bins).
// Avoid contention with shared memory partial histograms.

__global__ void histogram(const unsigned char *data, int *hist, int n) {
    __shared__ int s_hist[256];

    // Clear shared histogram:
    if (threadIdx.x < 256) s_hist[threadIdx.x] = 0;
    __syncthreads();

    // Build local partial histogram (fast shared memory atomics):
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n)
        atomicAdd(&s_hist[data[i]], 1);
    __syncthreads();

    // Merge into global histogram (one atomic per bin per block):
    if (threadIdx.x < 256)
        atomicAdd(&hist[threadIdx.x], s_hist[threadIdx.x]);
}

// Launch:
histogram<<<(n+255)/256, 256>>>(d_data, d_hist, n);

Problem 5: Inclusive Prefix Scan (Kogge-Stone)

// Compute inclusive scan: output[i] = sum(input[0..i])
// Within a single block using shared memory.

__global__ void prefix_scan(float *data, float *output, int n) {
    extern __shared__ float temp[];
    int tid = threadIdx.x;

    temp[tid] = (tid < n) ? data[tid] : 0.0f;
    __syncthreads();

    // Kogge-Stone scan:
    for (int stride = 1; stride < blockDim.x; stride *= 2) {
        float val = temp[tid];
        if (tid >= stride) val += temp[tid - stride];
        __syncthreads();
        temp[tid] = val;
        __syncthreads();
    }

    if (tid < n) output[tid] = temp[tid];
}

Problem 6: Dot Product

// Compute dot product of two float vectors.
// Combines per-thread partial sums with warp reduction.

__global__ void dot_product(const float *a, const float *b,
                             float *result, int n) {
    extern __shared__ float sdata[];
    int tid = threadIdx.x;
    int i   = blockIdx.x * blockDim.x + threadIdx.x;

    // Each thread computes its contribution:
    float val = 0.0f;
    while (i < n) {
        val += a[i] * b[i];
        i += blockDim.x * gridDim.x;   // grid-stride loop
    }
    sdata[tid] = val;
    __syncthreads();

    // Reduction:
    for (int s = blockDim.x / 2; s > 0; s >>= 1) {
        if (tid < s) sdata[tid] += sdata[tid + s];
        __syncthreads();
    }

    if (tid == 0)
        atomicAdd(result, sdata[0]);
}

Part 3 — System Design Questions

Design: Real-time image processing for 100 cameras

  Key considerations:
  • 100 cameras × 1080p × 30fps = huge throughput requirement
  • Target: <50ms end-to-end latency per frame

  Architecture:
  ─────────────────────────────────────────────────────────
  Camera streams → Decode (CPU/NVDEC) → Batch → GPU Pipeline → Output

  Components:
  1. NVDEC hardware decoder (free GPU decode, no CUDA cores used)
  2. cudaMemcpyAsync to GPU with pinned host buffers
  3. Batch multiple camera frames into one kernel launch
     (reduces kernel launch overhead: 100 launches → 1 batched launch)
  4. Use CUDA streams per camera for overlap:
     stream[cam]: decode → upload → process → download
  5. Result dispatch: atomicAdd to a shared output queue

  Memory:
  • Use cudaMallocHost (pinned) for frame buffers
  • Keep processed frames on GPU as long as possible
  • Async download only final results

Design: Matrix multiplication too large for GPU memory

  Problem: A is 100GB×100GB. GPU has 80GB DRAM.

  Solution: Block matrix multiplication
  ─────────────────────────────────────────────────────────
  Split A, B into tiles that fit in GPU memory.
  For C = A × B:
    for each block row of A (A_block):
        upload A_block to GPU
        for each block column of B (B_block):
            upload B_block to GPU
            C_block += A_block × B_block (cuBLAS)
            accumulate result

  Overlap: use streams to upload next block while computing current:
    Stream 0: compute A_block[i] × B_block[j]
    Stream 1: upload A_block[i+1]
    Stream 2: upload B_block[j+1]

  For distributed: use NCCL for multi-GPU, tensor parallelism

Part 4 — Common Candidate Mistakes

  ✗ Not checking CUDA error codes — makes bugs invisible
  ✗ Ignoring memory coalescing — 10× performance loss
  ✗ Using cudaDeviceSynchronize() in a loop — kills async benefits
  ✗ Off-by-one in kernel index: forgetting (N + blockSize - 1) / blockSize
  ✗ Assuming cudaMemcpy is asynchronous (it's not! — it blocks the CPU)
  ✗ Using double when float suffices (double is 2× slower for most ops)
  ✗ Forgetting __syncthreads() after loading shared memory
  ✗ Divergent __syncthreads() — illegal and causes hangs
  ✗ Benchmarking with -G debug flag (10-100× slower than release)
  ✗ Not considering PCIe bottleneck in total pipeline timing

Takeaways

  • Know the warp: 32 threads, lockstep execution, every perf metric is warp-level.
  • Know the memory hierarchy cold: registers (1 cyc), shared (5 cyc), global (800 cyc).
  • Be able to write reduction, transpose, and histogram from memory in an interview.
  • Know the Roofline model: AI = FLOPs/bytes. Memory-bound vs compute-bound determines fixes.
  • Error checking: CUDA_CHECK on every call, CUDA_KERNEL_CHECK after every launch.
  • The most common mistake: ignoring coalescing → check it first in any slow kernel.
  • In production: cuBLAS, cuDNN, CUB are always faster than hand-written kernels for GEMM/reduction.
