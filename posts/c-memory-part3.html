Program 8 — Recursive Call Stack Frames

What happens when you call a function

Every time you call a function in C, the CPU executes a precise sequence of operations to set up the new execution context. This isn't a language abstraction — it's real hardware behaviour. Understanding it explains stack overflows, explains why local variables are fast, and explains the "never return a local variable's address" rule at a level that makes it impossible to forget.

At the hardware level, a function call looks like this:

; Calling function foo(42):
push rbp ; save caller's base pointer on the stack
mov rbp, rsp ; set up new frame's base pointer
sub rsp, 32 ; reserve 32 bytes for local variables
; ... function body ...
mov rsp, rbp ; restore stack pointer (release locals)
pop rbp ; restore caller's base pointer
ret ; jump back to return address (popped from stack)

Three critical observations from this:

  1. Allocation is a single subtract instruction. sub rsp, 32 — one CPU clock cycle — reserves 32 bytes for all local variables. That's why stack allocation is "free." You can't get faster than one instruction.
  2. The stack pointer moves downward. sub rsp, 32 decreases rsp, moving toward lower addresses. This is why the stack grows downward.
  3. Deallocation is also one instruction. mov rsp, rbp instantly "releases" all local variables by resetting the stack pointer. The memory isn't zeroed — it's just marked available by the pointer change. This is why dangling pointers to locals are so dangerous: the bytes are still there physically, but they'll be overwritten by the next function call.

Each call to factorial() pushes a new frame to a lower address (the yellow active frame). At the base case, the recursion unwinds — frames are popped in reverse order as each function returns its result to its caller. Watch the addresses decrease going deeper, then increase as the recursion unwinds. 

#include <stdio.h>

// We'll measure stack growth by comparing addresses of local
// variables across frames. The address decreases with each call.
static void *stack_base = NULL;

long factorial(int n) {
    // This local variable lives on THIS call's stack frame.
    // Its address tells us exactly where this frame is.
    int local_n = n;

    // On the very first call, record the base
    if (!stack_base) stack_base = (void*)&local_n;

    // How far have we descended from the top frame?
    long depth_bytes = (long)stack_base - (long)&local_n;

    printf("  factorial(%d)  |  &local_n=%p  |  depth=%ld bytes from base\n",
           n, (void*)&local_n, depth_bytes);

    if (n <= 1) {
        printf("  BASE CASE — recursion unwinds from here\n\n");
        return 1;
    }

    long result = (long)local_n * factorial(n - 1);
    printf("  factorial(%d) returns %ld\n", n, result);
    return result;
}

int main() {
    // Use main's local as a rough stack base reference
    int main_local = 0;
    stack_base = (void*)&main_local;

    printf("main() local @ %p\n\n", (void*)&main_local);
    printf("Calling factorial(5):\n");

    long result = factorial(5);
    printf("\nfactorial(5) = %ld\n", result);

    // Calculate average frame size
    // (measure by calling factorial again and seeing depth at n=1)
    return 0;
}

Look at the depth numbers: 48, 96, 144, 192, 240 — exactly 48 bytes per frame. That 48 bytes covers: the return address (8 bytes), the saved base pointer (8 bytes), local variables, alignment padding, and any register saves the compiler decided to do. With optimisation enabled (-O2), small recursive functions often use fewer bytes per frame or get tail-call optimised away entirely.

Stack overflow: what actually happens

The OS reserves a fixed amount of virtual address space for the stack — 8 MB on Linux by default. Just below the stack is an unmapped guard page. If the stack pointer ever reaches that guard page, the hardware raises a page fault, the OS detects it's a stack-related fault, and delivers SIGSEGV to the process.

At 48 bytes per frame, you can recurse about 174,000 deep before hitting an 8 MB limit. That sounds like a lot, but a recursive tree traversal or quicksort can easily hit thousands of levels on adversarial inputs. 

// How deep until stack overflow?
// With 8 MB stack and ~50 bytes per frame:
// 8 * 1024 * 1024 / 50 ≈ 167,772 frames

void recurse(int n) {
    char big_buf[512];  // large local array = bigger frame = less depth
    big_buf[0] = n;     // prevent optimisation away
    recurse(n + 1);     // with 512-byte locals: 8MB / 512 ≈ 16,384 levels only
}

// Rule: never put large buffers on the stack.
// Use malloc() for anything over a few KB.
The dangling local pointer bug: Never return a pointer to a local variable. The frame is gone the instant the function returns. The stack memory will be reused by the next function call — your pointer now points into the middle of someone else's locals. This is undefined behaviour and produces one of the most baffling bugs possible: it works sometimes (the bytes haven't been overwritten yet) and crashes others (they have). 
int *broken() {
    int x = 42;
    return &x;    // x lives on THIS frame
}                 // frame destroyed here

int *ok() {
    int *x = malloc(sizeof(int));  // heap allocation
    *x = 42;
    return x;     // heap outlives the frame — caller must free
}

Program 9 — Array Bounds & Safe Buffer

C's design choice: trust the programmer

C was designed in the 1970s for writing operating systems. Speed was everything. Every array access with a bounds check would add a branch instruction. The language designers made a deliberate decision: no runtime bounds checking. If you access arr[100] on an array of 5 elements, the compiler generates the pointer arithmetic, the CPU reads or writes whatever is at that address, and execution continues. No exception. No crash (usually). Just silent, undefined behaviour.

Fifty years later, this decision has been responsible for an enormous fraction of all security vulnerabilities ever discovered — buffer overflows, heap overflows, stack smashing attacks. The design trade-off made sense in 1973. It's harder to justify today. But it's the language you're using, so you need to understand exactly what happens when you go out of bounds.

The array (blue) is surrounded by canary values (green) placed in adjacent memory. Safe accesses (arr[0], arr[2], arr[4]) leave canaries intact. Out-of-bounds accesses (red) overwrite the canaries — silently corrupting data that isn't part of the array. arr[100] in a real program would hit either another variable, heap metadata, or unmapped memory (crash).

What out-of-bounds access actually does

#include <stdio.h>
#include <stdlib.h>

int main() {
    // Set up the situation: an array with known values on either side
    // to see exactly what OOB access would corrupt.
    int canary_before = 0xAAAAAAAA;
    int arr[5]        = {10, 20, 30, 40, 50};
    int canary_after  = 0xBBBBBBBB;

    printf("Memory layout (stack, decreasing address upward on x86-64):\n");
    printf("  canary_before @ %p  = 0x%X\n", (void*)&canary_before, canary_before);
    printf("  arr[0]        @ %p  = %d\n",   (void*)&arr[0], arr[0]);
    printf("  arr[4]        @ %p  = %d\n",   (void*)&arr[4], arr[4]);
    printf("  canary_after  @ %p  = 0x%X\n", (void*)&canary_after, canary_after);

    // Demonstrate the access that would happen (we print the address
    // but don't actually dereference — that's undefined behaviour)
    printf("\nWhat would OOB accesses touch:\n");
    printf("  arr[-1] would access: %p  (near canary_before)\n",
           (void*)(arr - 1));
    printf("  arr[5]  would access: %p  (near canary_after)\n",
           (void*)(arr + 5));
    printf("  arr[50] would access: %p  (somewhere on the stack)\n",
           (void*)(arr + 50));
    // Accessing these addresses reads/writes whatever happens to be there:
    // another local variable, a saved register, the return address...
    // Writing to the return address = classic stack smashing attack

    // Safe accesses
    printf("\nSafe reads:\n");
    for (int i = 0; i < 5; i++)
        printf("  arr[%d] = %d  (@ %p)\n", i, arr[i], (void*)&arr[i]);

    printf("\nCanaries after safe accesses:\n");
    printf("  canary_before = 0x%X  %s\n", canary_before,
           canary_before == 0xAAAAAAAA ? "(intact)" : "CORRUPTED!");
    printf("  canary_after  = 0x%X  %s\n", canary_after,
           canary_after  == 0xBBBBBBBB ? "(intact)" : "CORRUPTED!");

    return 0;
}

Building a Safe Buffer

The fix is to wrap the array in a struct that carries its size and checks every access. This is essentially what C++'s std::vector::at() does (throwing an exception), and what Rust does for every array access by default (panicking with a bounds error). In C, we build it manually: 

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct {
    int  *data;      // the actual heap-allocated array
    int   capacity;  // total slots available
    int   size;      // slots currently used
} SafeBuffer;

// Create a safe buffer with the given capacity, all zeroed
SafeBuffer *sb_create(int capacity) {
    SafeBuffer *sb  = (SafeBuffer *)malloc(sizeof(SafeBuffer));
    sb->data        = (int *)calloc(capacity, sizeof(int));  // calloc zeroes memory
    sb->capacity    = capacity;
    sb->size        = 0;
    return sb;
}

// Returns 1 on success, 0 on bounds error
int sb_set(SafeBuffer *sb, int index, int value) {
    if (index < 0 || index >= sb->capacity) {
        fprintf(stderr, "[SafeBuffer] BOUNDS ERROR: index %d out of [0, %d)\n",
                index, sb->capacity);
        return 0;  // failure — no write occurred
    }
    sb->data[index] = value;
    if (index >= sb->size) sb->size = index + 1;
    return 1;
}

// Returns 1 and fills *out on success, 0 on bounds error
int sb_get(SafeBuffer *sb, int index, int *out) {
    if (index < 0 || index >= sb->size) {
        fprintf(stderr, "[SafeBuffer] BOUNDS ERROR: read index %d out of [0, %d)\n",
                index, sb->size);
        return 0;
    }
    *out = sb->data[index];
    return 1;
}

void sb_free(SafeBuffer *sb) {
    free(sb->data);
    free(sb);
}

int main() {
    SafeBuffer *sb = sb_create(5);
    printf("Created SafeBuffer: capacity=%d\n\n", sb->capacity);

    // Good writes
    printf("Writing values 0..4:\n");
    for (int i = 0; i < 5; i++)
        sb_set(sb, i, (i + 1) * 100);

    // Bad writes — caught at runtime
    printf("\nAttempting out-of-bounds writes:\n");
    sb_set(sb, 5,   9999);   // one past the end
    sb_set(sb, -1,  9999);   // negative
    sb_set(sb, 100, 9999);   // way past

    // Good reads
    printf("\nReading values back:\n");
    for (int i = 0; i < sb->size; i++) {
        int val;
        if (sb_get(sb, i, &val))
            printf("  sb[%d] = %d\n", i, val);
    }

    // Bad read
    printf("\nAttempting out-of-bounds read:\n");
    int dummy;
    sb_get(sb, 10, &dummy);  // caught

    sb_free(sb);
    printf("\nAll OOB accesses caught. No canaries harmed.\n");
    return 0;
}

calloc vs malloc: what's the difference?

calloc(n, size) allocates n * size bytes and zeroes all of them. malloc(n * size) allocates the same memory but leaves the contents as garbage — whatever was in that heap space previously. 

// malloc: fast, contents undefined
int *a = malloc(10 * sizeof(int));
// a[0] might be 0, might be 4882913, might be anything.
// Accessing it before setting it is undefined behaviour.

// calloc: slightly slower (must zero memory), but guaranteed clean
int *b = calloc(10, sizeof(int));
// b[0] through b[9] are guaranteed to be 0.

// Use calloc for:
//   - Arrays you'll iterate before filling completely
//   - Boolean flags / state arrays
//   - Anything where "unset" should mean 0 or false

// Use malloc for:
//   - Buffers you'll fill completely before reading
//   - Performance-critical paths where the zero-fill is wasted work

Tools that catch bounds errors automatically

Building a SafeBuffer is educational but tedious for production code. These tools do it automatically, instrumenting your binary at compile time:

AddressSanitizer (ASan) — recommended for development

Fastest option. Catches buffer overflows, use-after-free, and stack overflows with a ~2x performance overhead. Included in GCC and Clang.

gcc -fsanitize=address -fsanitize=undefined -g program.c && ./a.out

Valgrind — most thorough

Runs your program in a virtual machine. Catches memory errors AND memory leaks. 5–20x slower, but finds issues ASan misses. Best for pre-release testing.

valgrind --tool=memcheck --leak-check=full --track-origins=yes ./a.out

UBSan — undefined behaviour

Catches signed integer overflow, null pointer dereference, misaligned access, and other UB that ASan doesn't cover.

gcc -fsanitize=undefined -g program.c && ./a.out
CUDA bounds checking: GPU kernels are launched across many threads, and it's easy for the last few threads to exceed the array size. The standard guard pattern is mandatory: 
__global__ void kernel(float *arr, int N) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx >= N) return;  // ← without this, OOB write, silent corruption
    arr[idx] = arr[idx] * 2.0f;
}
CUDA doesn't throw exceptions or segfault on OOB — it silently writes to wrong device memory addresses, corrupting whatever was there. The bug often shows up many kernel calls later as mysteriously wrong results.

Program 10 — malloc/free Tracker: Build Your Own Leak Detector

Why memory leaks are serious

A memory leak is simple to define: you malloc() memory and never free() it. The block remains allocated for the lifetime of the process, unreachable, contributing nothing but wasted RAM.

For short-lived programs, leaks don't matter — the OS reclaims all process memory on exit. For long-running processes — web servers, inference workers, database engines — a leak that grows at 1 MB per request will eventually exhaust all available memory and crash. I once debugged a GPU inference server that died every 36 hours. The leak was 3 lines of code that called a library function which returned a malloc'd buffer that the documentation said was "caller's responsibility to free." We weren't freeing it.

The approach: macro-based interception

Rather than using an external tool, we can build a minimal leak detector using one of C's most powerful features: the preprocessor. The key insight is that macros can capture __FILE__ and __LINE__ — the source filename and line number — at the call site, before replacing themselves with function calls. This is how real profilers and leak detectors track "where was this allocated."

Each MALLOC call adds a row to the allocation table marked LIVE. FREE calls flip the row to FREE. The heap usage bar shows real-time byte consumption. At program end, any row still marked LIVE is reported as a LEAK — including the exact line number in the source where the allocation happened. 

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

// ── Allocation tracking table ─────────────────────────────────
// Each malloc call gets one record. We store the address, size,
// the source location, and whether it has been freed.
#define MAX_TRACKED_ALLOCS 256

typedef struct {
    void       *ptr;    // the address malloc returned
    size_t      size;   // bytes requested
    const char *file;   // source file that called MALLOC
    int         line;   // source line that called MALLOC
    int         freed;  // 0=live, 1=freed
} AllocRecord;

static AllocRecord g_table[MAX_TRACKED_ALLOCS];
static int         g_count       = 0;
static size_t      g_total_alloc = 0;  // cumulative bytes ever allocated
static size_t      g_total_freed = 0;  // cumulative bytes freed
static size_t      g_peak_bytes  = 0;  // highest simultaneous live bytes

// ── Our custom malloc ─────────────────────────────────────────
void *tracked_malloc(size_t size, const char *file, int line) {
    // 1. Do the real allocation
    void *ptr = malloc(size);
    if (!ptr) return NULL;

    // 2. Record it in our table
    if (g_count < MAX_TRACKED_ALLOCS) {
        g_table[g_count].ptr   = ptr;
        g_table[g_count].size  = size;
        g_table[g_count].file  = file;
        g_table[g_count].line  = line;
        g_table[g_count].freed = 0;
        g_count++;
    }
    g_total_alloc += size;

    // 3. Update peak usage
    size_t live = g_total_alloc - g_total_freed;
    if (live > g_peak_bytes) g_peak_bytes = live;

    printf("  [ALLOC #%02d]  %p  %6zu bytes  (%s:%d)\n",
           g_count, ptr, size, file, line);
    return ptr;
}

// ── Our custom free ───────────────────────────────────────────
void tracked_free(void *ptr, const char *file, int line) {
    for (int i = 0; i < g_count; i++) {
        if (g_table[i].ptr == ptr && !g_table[i].freed) {
            g_table[i].freed = 1;
            g_total_freed += g_table[i].size;
            printf("  [FREE  #%02d]  %p  %6zu bytes  (%s:%d)\n",
                   i+1, ptr, g_table[i].size, file, line);
            free(ptr);
            return;
        }
    }
    // Not found — either double free or ptr wasn't tracked
    fprintf(stderr, "  [FREE  ???]  %p  NOT TRACKED! "
            "double-free? (%s:%d)\n", ptr, file, line);
}

// ── The magic macros ──────────────────────────────────────────
// __FILE__ expands to the current source filename (a string literal)
// __LINE__ expands to the current line number (an integer)
// Both are evaluated at the CALL SITE — not inside tracked_malloc.
// So when line 87 writes MALLOC(100), the preprocessor substitutes:
//   tracked_malloc((100), "myfile.c", 87)
// The call site's location is captured at compile time.
#define MALLOC(size)  tracked_malloc((size), __FILE__, __LINE__)
#define FREE(ptr)     tracked_free((ptr),    __FILE__, __LINE__)

// ── Leak report ───────────────────────────────────────────────
void print_leak_report() {
    int    leak_count = 0;
    size_t leak_bytes = 0;

    printf("\n╔══════════════════════════════════════╗\n");
    printf("║         MEMORY LEAK REPORT           ║\n");
    printf("╠══════════════════════════════════════╣\n");

    for (int i = 0; i < g_count; i++) {
        if (!g_table[i].freed) {
            printf("║  LEAK #%02d: %zu bytes @ %p  ║\n",
                   leak_count+1, g_table[i].size, g_table[i].ptr);
            printf("║  Allocated at: %s line %d\n",
                   g_table[i].file, g_table[i].line);
            leak_bytes += g_table[i].size;
            leak_count++;
        }
    }

    printf("╠══════════════════════════════════════╣\n");
    printf("║  Total allocated : %8zu bytes    ║\n", g_total_alloc);
    printf("║  Total freed     : %8zu bytes    ║\n", g_total_freed);
    printf("║  Peak live       : %8zu bytes    ║\n", g_peak_bytes);

    if (leak_count == 0) {
        printf("║  Result: NO LEAKS. Clean exit.       ║\n");
    } else {
        printf("║  LEAKED: %8zu bytes (%d alloc(s))  ║\n",
               leak_bytes, leak_count);
    }
    printf("╚══════════════════════════════════════╝\n");
}

// ── Simulated program ─────────────────────────────────────────
int main() {
    printf("=== malloc/free TRACKER DEMO ===\n\n");
    printf("Allocations:\n");

    // Four allocations — we'll "forget" to free one
    int   *data  = MALLOC(10 * sizeof(int));
    char  *name  = MALLOC(64);
    float *vecs  = MALLOC(3  * sizeof(float));
    char  *logbuf= MALLOC(1024);  // ← intentionally leaked

    // Use the memory
    for (int i = 0; i < 10; i++) data[i] = i * i;
    strncpy(name, "tracker-demo", 63);
    vecs[0] = 1.0f; vecs[1] = 0.0f; vecs[2] = 0.0f;
    memset(logbuf, 0, 1024);

    printf("\nFreeing some allocations:\n");
    FREE(data);   // freed
    FREE(name);   // freed
    FREE(vecs);   // freed
    // logbuf intentionally not freed — simulates a leak

    // Run the leak report
    print_leak_report();

    // Actually free for OS cleanliness (bypassing our tracker
    // since we already reported it — double-tracking would confuse stats)
    free(logbuf);
    return 0;
}

Understanding __FILE__ and __LINE__

The power of this technique is in the macros. When the C preprocessor sees MALLOC(1024) at line 160 of tracker.c, it expands it to: 

tracked_malloc((1024), "tracker.c", 160)

The filename and line number are baked in as constants at compile time. Inside tracked_malloc, we just store those strings and integers. When we later report a leak, we can say exactly which line caused it. This is the same technique AddressSanitizer uses — it records allocation stack traces, so when it reports a leak, it shows you the full call chain.

Taking it further: This same interception technique works for any resource. Replace fopen/fclose to find unclosed file handles. Replace pthread_mutex_lock/unlock to find un-released locks. Replace cudaMalloc/cudaFree to track GPU memory leaks. The pattern is always: define a wrapper function that records the operation, then use a macro to call the wrapper with __FILE__ and __LINE__.

Putting It All Together

These ten programs cover a complete mental model of C memory management. Not the high-level "malloc gives you memory" version — the actual version, with addresses, frame depths, padding bytes, and leak sources printed to the terminal so you can see exactly what's happening.

Before writing your first CUDA kernel, you should be able to answer these:

  • Why do stack variables have addresses starting with 0x7fff while heap variables start lower?
  • What does ptr + 3 mean when ptr is a double*?
  • Why is int global; guaranteed to be 0, but int local; is not?
  • Why shouldn't you use a 2D pointer-of-pointers layout for GPU matrix multiplication?
  • What's the right way to use realloc without creating a memory leak on failure?
  • Why does traversing a linked list cause cache misses while traversing an array doesn't?
  • If struct Bad has members summing to 15 bytes, what is sizeof(Bad) and why?
  • How many bytes does each recursive call consume, and how do you measure it?
  • What actually happens in memory when you access arr[n] on an array of size n?
  • How does __LINE__ let you track which line of code made a malloc call?
CUDA doesn't abstract these concepts away — it exposes more of them. cudaMalloc is heap malloc on the device. __shared__ memory is essentially a manually managed cache. Coalesced access is pointer arithmetic done right by thousands of threads simultaneously. Every concept in this series has a direct GPU analogue. Know the C, and the CUDA makes sense.
