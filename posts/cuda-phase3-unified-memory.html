CUDA Mastery Phase 3.4 — Unified Memory: When Convenience Beats Manual Transfers

How cudaMallocManaged() lets you write GPU code without explicit cudaMemcpy calls, what actually happens under the hood with page faults and migration, and when unified memory helps versus when it quietly destroys your performance.

Writing CUDA without unified memory means manually managing two separate copies of every data structure — one on the CPU, one on the GPU — and keeping them in sync with explicit cudaMemcpy calls. For quick experiments and complex data structures like linked lists or trees, this is painful. Unified memory is the answer, but it comes with performance traps you need to understand.

1. The Problem Unified Memory Solves

  Without unified memory: explicit memory management
  ────────────────────────────────────────────────────
  float *h_data = (float*)malloc(N * sizeof(float));
  float *d_data;
  cudaMalloc(&d_data, N * sizeof(float));

  // Initialize on CPU:
  for (int i = 0; i < N; i++) h_data[i] = init_value(i);

  // Upload before kernel:
  cudaMemcpy(d_data, h_data, N * sizeof(float), cudaMemcpyHostToDevice);

  // Run kernel:
  myKernel<<<grid, block>>>(d_data, N);

  // Download results:
  cudaMemcpy(h_data, d_data, N * sizeof(float), cudaMemcpyDeviceToHost);

  // Now use h_data on CPU...

  // Cleanup: free BOTH:
  free(h_data);
  cudaFree(d_data);
  With unified memory: single pointer, automatic migration
  ────────────────────────────────────────────────────────
  float *data;
  cudaMallocManaged(&data, N * sizeof(float));

  // Initialize on CPU (pointer works on CPU!):
  for (int i = 0; i < N; i++) data[i] = init_value(i);

  // Run kernel (pointer works on GPU!):
  myKernel<<<grid, block>>>(data, N);
  cudaDeviceSynchronize();  // wait for kernel

  // Use results on CPU (no explicit copy needed):
  printf("result[0] = %f\n", data[0]);

  // Cleanup: single free:
  cudaFree(data);

2. What Happens Under the Hood

  Unified memory works via the virtual memory system and hardware page faults.

  ┌─────────────────────────────────────────────────────────┐
  │  Unified Memory Architecture                             │
  │                                                          │
  │  Virtual address space:  0x7f0000000000                 │
  │  ┌──────────────────────────────────────┐               │
  │  │  "managed" pages                     │               │
  │  │  • Single virtual address            │               │
  │  │  • Physical copy is on CPU or GPU    │               │
  │  │    (migrated on demand)              │               │
  │  └──────────────────────────────────────┘               │
  │                                                          │
  │  Timeline:                                               │
  │  1. cudaMallocManaged()  → pages allocated on CPU       │
  │  2. CPU writes data[]    → CPU accesses, pages stay CPU │
  │  3. kernel launch        → GPU tries to access pages    │
  │  4. PAGE FAULT on GPU    → driver migrates page CPU→GPU │
  │  5. kernel continues     → GPU reads migrated data      │
  │  6. cudaDeviceSynchronize()                              │
  │  7. CPU reads data[]     → PAGE FAULT on CPU            │
  │  8. driver migrates page GPU→CPU                        │
  └─────────────────────────────────────────────────────────┘

  Each page fault migration costs ~1-2 ms (PCIe transfer + overhead).
  For N pages accessed: N × 1-2ms of migration overhead!

3. Pre-fetching to Eliminate Page Fault Overhead

  // Instead of waiting for page faults, pre-fetch explicitly:
  int device = 0;
  cudaGetDevice(&device);

  // Pre-fetch to GPU before kernel:
  cudaMemPrefetchAsync(data, N * sizeof(float), device, 0);
  //                   ptr   size               dst_dev  stream

  // Now kernel has no page faults:
  myKernel<<<grid, block>>>(data, N);
  cudaDeviceSynchronize();

  // Pre-fetch back to CPU before CPU access:
  cudaMemPrefetchAsync(data, N * sizeof(float), cudaCpuDeviceId, 0);
  cudaDeviceSynchronize();

  printf("result = %f\n", data[0]);   // no page fault!
  Performance comparison (N = 10M floats, simple kernel):
  ────────────────────────────────────────────────────────────
  Explicit cudaMemcpy (baseline):    ~8 ms total
  Unified memory, no prefetch:       ~25 ms  (3× slower, page fault cost)
  Unified memory + prefetch:         ~8.5 ms (near explicit, slight overhead)
  ────────────────────────────────────────────────────────────

4. cudaMemAdvise — Memory Hints

  // Tell the driver this data is mostly read (safe to replicate):
  cudaMemAdvise(data, size, cudaMemAdviseSetReadMostly, device);
  // Driver can now keep copies on BOTH CPU and GPU.
  // CPU reads: served from CPU copy (no migration)
  // GPU reads: served from GPU copy (no migration)
  // Writes: invalidate other copy (write must migrate)

  // Tell the driver data "prefers" to live on GPU:
  cudaMemAdvise(data, size, cudaMemAdviseSetPreferredLocation, device);
  // Initial allocation on GPU memory instead of CPU.
  // CPU accesses: served via page fault + migration or mapped remotely.

  // Tell the driver a device will access this data frequently:
  cudaMemAdvise(data, size, cudaMemAdviseSetAccessedBy, device);
  // Driver maintains a GPU mapping to the data wherever it lives.
  // Reduces page fault cost: direct PCIe access instead of migration.

5. Unified Memory for Complex Data Structures

Unified memory shines for recursive or pointer-based data structures that are hard to manually transfer: 

  // Linked list usable on BOTH CPU and GPU:
  struct Node {
      float value;
      Node *next;   // pointer works on both sides!
  };

  Node *head = nullptr;

  // Build on CPU:
  for (int i = 0; i < 100; i++) {
      Node *n;
      cudaMallocManaged(&n, sizeof(Node));
      n->value = (float)i;
      n->next  = head;
      head = n;
  }

  // Traverse on GPU:
  __global__ void sum_list(Node *head, float *result) {
      if (threadIdx.x == 0 && blockIdx.x == 0) {
          float s = 0.0f;
          for (Node *n = head; n != nullptr; n = n->next)
              s += n->value;
          *result = s;
      }
  }

  float *d_result;
  cudaMallocManaged(&d_result, sizeof(float));
  sum_list<<<1, 1>>>(head, d_result);
  cudaDeviceSynchronize();
  printf("Sum = %f\n", *d_result);

6. Pascal+ vs Older GPUs

  Hardware page fault support (compute capability 6.0+, Pascal+):
  ──────────────────────────────────────────────────────────────────
  Pascal (sm_60) and newer:
    • True hardware page faults on GPU
    • Pages migrate automatically on access
    • Oversubscription: managed memory can exceed GPU DRAM size!
    • Pre-fetch works properly

  Kepler/Maxwell (sm_30 to sm_52):
    • Software-emulated unified memory
    • ALL managed data is migrated to GPU at kernel launch
    • NO on-demand migration
    • NO oversubscription
    • Prefetch is a no-op

  Check your GPU:
  $ nvidia-smi --query-gpu=compute_cap --format=csv
  $ nvcc -arch=sm_86 (RTX 3xxx) or -arch=sm_80 (A100)

7. When to Use Unified Memory

  USE unified memory when:
  ✓ Prototyping / experimenting quickly
  ✓ Complex data structures (trees, graphs, linked lists)
  ✓ Data exceeds GPU memory (oversubscription on Pascal+)
  ✓ Irregular access patterns (iterative algorithms, sparse data)
  ✓ Simplicity matters more than peak performance

  DON'T USE unified memory when:
  ✗ Maximum performance is required (streaming, video processing)
  ✗ Transfer pattern is known in advance (use explicit cudaMemcpy)
  ✗ High-frequency CPU↔GPU data exchange (page fault overhead adds up)
  ✗ Production code where latency is measured
  ✗ pre-Volta GPUs without proper hardware page fault support

8. Profiling Unified Memory

  # Profile page fault counts and migration traffic:
  nsys profile --stats=true --cuda-um-cpu-page-faults=true \
               --cuda-um-gpu-page-faults=true \
               ./my_program

  # Look in the output for:
  # - "CUDA GPU Page Faults" — how many page faults on GPU
  # - "CUDA CPU Page Faults" — how many page faults on CPU
  # High counts → consider using prefetch or explicit transfers

Takeaways

  • cudaMallocManaged() creates a single pointer valid on both CPU and GPU.
  • Under the hood: hardware page faults trigger automatic page migration over PCIe.
  • Page fault migration adds ~1-2ms per page — use cudaMemPrefetchAsync() to avoid this.
  • cudaMemAdvise(SetReadMostly) allows the driver to replicate read-only data.
  • Unified memory enables complex data structures (trees, graphs) on GPU without manual transfer.
  • Hardware page faults require Pascal (compute 6.0+); older GPUs use slower software emulation.
  • Use for prototyping and complex structures; use explicit transfers for production performance.
