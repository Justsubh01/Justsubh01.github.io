Atomic Operations — Thread-Safe Updates Without Locks

Atomics solve the race condition problem: when thousands of threads want to update the same memory location simultaneously, you need a guarantee that each update is indivisible. CUDA provides hardware-accelerated atomics for integers, floats, and custom compare-and-swap operations. Understanding atomics — and when not to use them — is essential for any kernel that aggregates results across threads.

The fundamental problem: two threads read a counter (value=5), both add 1, both write back 6. Final result: 6, not 7. With 10,000 threads all incrementing the same counter, you might get anything from 1 to 10,000. Atomics fix this by making read-modify-write an indivisible hardware operation.

What Are Atomics and Why They're Needed

An atomic operation reads, modifies, and writes a memory location as an indivisible unit. No other thread can see or interrupt the intermediate state. The hardware memory controller serializes concurrent atomics to the same address, guaranteeing correctness regardless of how many threads compete. 

// WRONG: race condition -- result is unpredictable
__global__ void wrongCounter(int* counter, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        // Step 1: read *counter into register
        int old = *counter;
        // -- Another thread may execute steps 1+2 here, overwriting *counter --
        // Step 2: compute new value
        // Step 3: write back
        *counter = old + 1;  // may overwrite the other thread's update!
    }
}
// With N=10000 threads: result is anywhere from 1 to 10000 -- completely broken

// CORRECT: atomic -- hardware guarantees each increment is applied exactly once
__global__ void correctCounter(int* counter, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        // atomicAdd is ONE hardware instruction: read-add-write as an atomic unit
        // No other thread can interleave between the read and the write
        atomicAdd(counter, 1);
    }
}
// With N=10000 threads: result is always exactly 10000 -- guaranteed
What "atomic" means at the hardware level: The memory controller receives an atomic instruction and locks the cache line for that address. All other atomic requests to the same address are queued. The controller completes the read-modify-write, then releases the lock and processes the next queued request. This is why atomics to the same address are serialized: the hardware enforces mutual exclusion, not software.

__global__ void atomicExamples(int* data, unsigned* flags, float* fdata) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    // atomicAdd returns the OLD value before adding
    int old = atomicAdd(&data[0], 1);
    // old = what data[0] was BEFORE this thread incremented it
    // This is useful for claiming a unique slot: if old=5, this thread owns slot 5

    // atomicMax: data[1] ends up as the maximum thread index seen
    atomicMax(&data[1], i);

    // atomicOr: set bit (i%32) in flags[0]
    // After kernel: flags[0] has bits set for every thread index mod 32 that ran
    atomicOr(&flags[0], (1u << (i % 32)));

    // atomicExch: swap in current thread index, get previous occupant
    int prev = atomicExch(&data[2], i);  // last writer's index wins

    // Float atomic add (hardware-native on sm_20+)
    atomicAdd(&fdata[0], 1.0f);  // safe for float accumulation
}

Compare-And-Swap — The Foundation

atomicCAS(addr, compare, val) is the primitive from which all other atomic operations can be built. The semantics:

  1. Read the current value at *addr into old
  2. If old == compare, write val to *addr
  3. Return old regardless

The check-and-write is atomic. If two threads race, only one will see old == compare and succeed; the other gets a different old and knows it must retry. 

// Build custom float atomicMin using CAS:
// (No native hardware atomicMin for floats, so we build it)
__device__ float atomicMinFloat(float* addr, float val) {
    // Reinterpret the float address as an int address
    // (safe because float and int are both 4 bytes and CAS works on int)
    int* addr_as_int = (int*)addr;
    int old = *addr_as_int, assumed;

    do {
        assumed = old;  // remember what we READ (our expectation)

        // Compute the new value we WANT to write:
        float old_float = __int_as_float(assumed);
        float new_float = fminf(old_float, val);
        int   new_int   = __float_as_int(new_float);

        // Attempt the atomic update: "if *addr still equals assumed, write new_int"
        old = atomicCAS(addr_as_int, assumed, new_int);

        // If old == assumed: CAS succeeded, the value was updated. Loop exits.
        // If old != assumed: someone else changed *addr between our read and CAS.
        //   old now holds the NEW current value. Retry with updated 'assumed'.

    } while (assumed != old);  // retry until we successfully CAS

    return __int_as_float(old);  // return the old minimum (before our update)
}

Let me trace a concurrent scenario to make the retry logic clear:

# Initial: *addr = 7.0f, Thread A wants atomicMinFloat(addr, 3.0f) # Thread B wants atomicMinFloat(addr, 5.0f) Thread A: Thread B: old = *addr_as_int (reads 7.0) old = *addr_as_int (reads 7.0) assumed = 7.0 assumed = 7.0 new_float = min(7.0, 3.0) = 3.0 new_float = min(7.0, 5.0) = 5.0 atomicCAS(addr, 7.0, 3.0) <--- Thread A wins the race -- CAS succeeds: *addr becomes 3.0 -- returns old=7.0 while(7.0 != 7.0) = false atomicCAS(addr, 7.0, 5.0) Thread A done. *addr = 3.0 -- CAS FAILS: *addr is now 3.0, not 7.0 -- returns old = 3.0 (current value) while (7.0 != 3.0) = TRUE -- retry! assumed = 3.0 (current value) new_float = min(3.0, 5.0) = 3.0 atomicCAS(addr, 3.0, 3.0) -- succeeds while(3.0 != 3.0) = false Thread B done. *addr still = 3.0 (correct minimum)
Spinlocks in GPU code are dangerous. While CAS can be used to build spinlocks (spin until CAS changes 0 to 1), they are almost never the right solution on a GPU. If the thread holding the lock is context-switched out, all waiting threads spin indefinitely consuming cycles. GPU hardware does not support priority inversion avoidance. Always prefer warp-aggregated atomics, reduction patterns, or shared memory approaches over spinlocks.

Floating-Point Atomics

atomicAdd(float*, float) is hardware-native since sm_20. atomicAdd(double*, double) since sm_60. There is no hardware atomicMin or atomicMax for floats — use the CAS trick shown above. 

// Double precision atomic add (hardware on sm_60+, A100 is sm_80):
__global__ void doublePrecisionAccumulate(double* data, double* result, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        // Note: double atomicAdd is slower than float atomicAdd on most GPUs
        // because doubles share the same memory width as int32 atomics
        // but require more bits. Use reduction patterns for best performance.
        atomicAdd(result, data[i]);
    }
}

Atomic Performance and Contention

The performance of atomics depends entirely on contention — how many threads compete for the same address at the same time. When all threads hit the same address, the memory controller serializes them: N atomic operations on 1 address takes N times longer than N operations on N addresses.

# atomicAdd performance (A100, N=16M threads, all adding to same or different addresses): Contention level | Throughput | vs uncontested ------------------------+-----------------------+---------------- 1 unique address | 0.6 billion ops/sec | 0.03x (terrible!) 32 unique addresses | 2.1 billion ops/sec | 0.11x 1024 unique addresses | 18.7 billion ops/sec | 1.0x (baseline) No atomics (plain add) | 18.4 billion ops/sec | ~1.0x (atomics free at low contention!)

The lesson: at high contention (all threads to 1 address), atomics are 32x slower than compute. At low contention (1 address per 1000 threads), atomics add essentially zero overhead. Design your algorithm to maximize unique addresses.

The atomic golden rule: Maximize unique addresses per atomic operation.

For a histogram with 256 bins and 1M threads: each bin gets ~4000 updates on average. That is manageable — 4000 serialized ops vs 1M parallel ops is a small fraction.

For a global counter (1 address, 1M threads): 1M serialized ops — terrible. Use warp aggregation instead (reduces to ~31K actual atomic ops).

Warp Aggregation — Fast Atomics

Instead of each of the 32 threads in a warp doing its own atomic, the warp collaborates: count how many threads want to add, then have ONE thread do a single atomic adding the total. This reduces atomic contention by up to 32×. 

// Warp-aggregated atomic increment: equivalent to each thread calling atomicAdd(counter, 1)
// but performs only 1 atomic per warp instead of 32
__device__ int atomicAggInc(int* counter) {
    // Step 1: get a bitmask of which threads in this warp are currently executing
    // (some may be predicated off due to divergence)
    unsigned mask = __activemask();

    // Step 2: find the lowest-numbered active lane -- this will be the "leader"
    // __ffs = "find first set bit", returns 1-indexed position, so subtract 1
    int leader = __ffs(mask) - 1;
    int laneId = threadIdx.x % 32;

    int base = 0;
    if (laneId == leader) {
        // Step 3: leader does ONE atomic, adding the count of ALL active threads
        // __popc = "population count" = number of set bits = number of active threads
        base = atomicAdd(counter, __popc(mask));
        // base = old value of counter (before this warp's batch increment)
    }

    // Step 4: broadcast leader's base to all active threads
    base = __shfl_sync(mask, base, leader);

    // Step 5: each active thread computes its unique slot within the batch
    // "how many active threads have a lower lane ID than me?"
    // mask & ((1 << laneId) - 1) = mask restricted to lanes 0..(laneId-1)
    // __popc of that = count of active lanes below me = my offset within the batch
    int myOffset = __popc(mask & ((1 << laneId) - 1));

    return base + myOffset;  // my unique slot in the counter
}

The bit math for computing your position within the batch deserves a concrete example:

# Suppose 6 threads are active in this warp: lanes 0, 2, 3, 5, 7, 9 mask = 0b0000...0010101101 = 0x2AD # For laneId = 5 (one of the active lanes): (1 << laneId) - 1 = (1 << 5) - 1 = 32 - 1 = 0b00011111 (bits 0-4 set) mask & 0b00011111 = 0b0010101101 & 0b00011111 = 0b0000001101 __popc(0b0000001101) = 3 (bits 0, 2, 3 are set -- 3 active lanes below lane 5) # So lane 5's offset = 3, meaning it is the 4th thread (0-indexed: 3) in this batch # lane 0 gets offset=0, lane 2 gets offset=1, lane 3 gets offset=2, lane 5 gets offset=3... # If leader's atomicAdd returns base=1000: # lane 0 claims slot 1000, lane 2 claims 1001, lane 3 claims 1002, lane 5 claims 1003... # All 6 active threads get unique, consecutive slots from one atomic operation. 
// Usage example: scatter active elements to an output array
__global__ void compactPositive(int* data, int* output, int* count, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i >= n) return;

    bool active = (data[i] > 0);
    if (active) {
        // Instead of each active thread doing atomicAdd(count, 1) independently
        // (up to 32 serialized atomics per warp), use warp aggregation:
        // Only 1 atomic per warp, regardless of how many threads are active
        int slot = atomicAggInc(count);  // claim a unique output slot
        output[slot] = data[i];
    }
}
// For 32 active threads per warp: 32x fewer atomics vs naive approach

Case Study: Parallel Histogram

The histogram problem is a perfect stress test for atomics: N threads, each wanting to increment one of B bins. For popular bins, contention can be severe. The solution is to use per-block shared memory histograms, then merge globally. 

// Naive: all N threads atomically update the same B global bins
// For N=16M threads and B=256 bins: each bin averages 62,500 contended updates
__global__ void histogramNaive(const int* data, int* hist, int n, int nBins) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        // High contention for popular bins -- many threads racing for same address
        atomicAdd(&hist[data[i] % nBins], 1);
    }
}

// Optimized: each block builds a local histogram in shared memory,
// then adds the local histogram to the global one
__global__ void histogramShared(const int* data, int* hist, int n, int nBins) {
    // Launch with: histogramShared<<<grid, block, nBins*sizeof(int)>>>(...)
    extern __shared__ int localHist[];

    // Initialize local histogram to zero
    // (each thread initializes multiple bins if nBins > blockDim.x)
    for (int b = threadIdx.x; b < nBins; b += blockDim.x)
        localHist[b] = 0;
    __syncthreads();  // ensure zeroing is complete before accumulation

    // Phase 1: accumulate into LOCAL shared memory histogram
    // Shared memory atomics are much faster than global memory atomics --
    // the L1 cache handles the serialization locally, avoiding PCIe traffic
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        atomicAdd(&localHist[data[i] % nBins], 1);
    }
    __syncthreads();  // ensure all threads done before writing to global

    // Phase 2: add local histogram to global histogram
    // Only gridDim.x * nBins total global atomics (vs N naive atomics)
    // For N=16M, blockSize=256: gridSize=62500, nBins=256
    // Total global atomics: 62500 * 256 = 16M (same count, but no contention per block)
    for (int b = threadIdx.x; b < nBins; b += blockDim.x)
        if (localHist[b] > 0)
            atomicAdd(&hist[b], localHist[b]);  // rare, low-contention updates
}
# Performance (N=16M elements, nBins=256, A100): histogramNaive (global atomics): 8.2 ms histogramShared (shared + global): 0.9 ms (9x faster!) # Why so much faster? # Naive: 16M global atomic ops, up to 62500 threads competing per bin # Shared: 16M shared atomic ops (fast, local) + 16M global ops spread evenly # Shared mem atomics are ~10x faster than global; contention per bin drops # from 62500 (naive) to blockDim.x/nBins = ~1 (shared) per global atomic

Safe Patterns for Common Use Cases

Pattern 1: Claiming unique output slots (scatter)

// Common pattern: filter input array, write active elements to output
// Use atomic counter to claim unique, consecutive output slots
__global__ void filterAndScatter(float* input, float* output,
                                  int* outCount, float threshold, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i >= n) return;

    float val = input[i];
    if (val > threshold) {
        // Claim output slot atomically (use warp aggregation for best performance)
        int slot = atomicAdd(outCount, 1);
        output[slot] = val;
    }
}

// After kernel: *outCount tells you how many elements were written
// output[0..(*outCount-1)] contains the filtered values

Pattern 2: Atomic compare-and-swap for custom operations

// Custom atomic operation: only update if new value is "better" by some criterion
// Example: atomically update if new value has higher absolute value
__device__ float atomicMaxAbs(float* addr, float val) {
    int* iaddr = (int*)addr;
    int  ival  = __float_as_int(fabsf(val));  // compare on |val|
    int  old   = *iaddr, assumed;
    do {
        assumed = old;
        if (__int_as_float(assumed) >= fabsf(val)) break;  // early exit: already larger
        old = atomicCAS(iaddr, assumed, ival);
    } while (assumed != old);
    return __int_as_float(old);
}

Pattern 3: Atomic flag for early termination

// Signal from GPU to CPU: stop launching more kernels
__global__ void searchKernel(int* data, int target, int* found, int* foundIdx, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i >= n) return;

    // Check early-exit flag (after first thread finds it, others can bail)
    if (atomicAdd(found, 0) > 0) return;  // atomic read (add 0 = just read)

    if (data[i] == target) {
        atomicExch(found, 1);     // signal "found"
        atomicMin(foundIdx, i);   // record earliest index
    }
}
// Host: check *h_found after kernel; if 1, read h_foundIdx
