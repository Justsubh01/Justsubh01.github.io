Local LLM Deployment: Ollama vs vLLM vs llama.cpp — The Definitive Deep Dive

A complete engineering guide to running large language models on your own hardware — from first install to production-grade serving, covering quantization internals, KV-cache mechanics, continuous batching, and every knob that actually moves the needle.

Running LLMs locally has crossed a threshold. With a single RTX 3090 (24 GB VRAM) you can serve Llama 3.1 8B at full fp16 precision, hit sub-100ms time-to-first-token, and field hundreds of requests per hour — all without an API bill. But the gap between "it runs" and "it runs well" is enormous, and that gap is what this post is about.

We're going to go deep on three tools: Ollama (developer-friendly wrapper), vLLM (production throughput server), and llama.cpp (bare-metal GGUF inference). I'll explain not just how to use them, but why they work the way they do — the memory math, the scheduler internals, the quantization tradeoffs, and the optimization techniques that separate toy demos from real deployments.

Hardware reference for this post: NVIDIA RTX 3090 (24 GB VRAM), AMD Ryzen 7 3700X, 32 GB RAM, CUDA 13.0. All benchmarks and configs are tuned for this setup.

Part 0: The Memory Math You Must Know

Before touching any tool, internalize this formula. It predicts whether a model fits in your VRAM and how much headroom you have for KV cache: 

# VRAM estimate for inference (bytes)
model_vram = num_parameters * bytes_per_param

# bytes_per_param by precision:
#   fp32  → 4 bytes
#   fp16 / bf16 → 2 bytes
#   int8  → 1 byte
#   int4  → 0.5 bytes (4-bit NF4/GPTQ/AWQ)

# KV cache per token (bytes)
kv_per_token = 2 * num_layers * num_kv_heads * head_dim * bytes_per_param
# factor of 2 = one K tensor + one V tensor

# Total VRAM needed
total = model_vram + (max_seq_len * batch_size * kv_per_token)

For Llama 3.1 8B in fp16: 8B * 2 = 16 GB for weights. With 4096 context and batch=1, KV cache adds ~1 GB. Total ~17 GB — fits in a 3090. Bump to batch=8 and 8192 context: KV cache explodes to ~8 GB. Now you're tight at 24 GB. This is why KV cache management is the central problem in LLM serving.

Part 0.1: How Transformer Inference Actually Works

LLM inference has two distinct phases with completely different bottlenecks:

  • Prefill phase: Process all input tokens in parallel. Compute-bound. All tokens attend to each other, filling the KV cache. Throughput: ~tens of ms for a 1000-token prompt.
  • Decode phase: Generate one token at a time, autoregressively. Memory-bandwidth-bound. Each new token attends to all cached KV entries. A 3090 doing attention over 4096 cached tokens is reading ~500 MB of KV data per generated token.

The decode phase is why LLMs feel slow for long outputs. You're not compute-limited — you're limited by how fast you can stream weights and KV cache through the memory bus. The RTX 3090's 936 GB/s bandwidth means with a 16 GB model, you can do at most ~58 decode steps/sec on a single stream. Batching amortizes that cost.

Part 1: Ollama — The Fast Lane to Running Models

1.1 What Ollama Actually Is

Ollama is a Go daemon that wraps llama.cpp under the hood, adds a model registry (pull/push like Docker), automatic GPU detection, and an OpenAI-compatible REST API. It's not magic — it's a well-engineered convenience layer. When you run ollama run llama3.1, it pulls a pre-quantized GGUF file, loads it into llama.cpp, and exposes port 11434.

1.2 Installation

# Linux (one-liner)
curl -fsSL https://ollama.com/install.sh | sh

# Verify the daemon is running
systemctl status ollama

# Check GPU detection
ollama serve &
curl http://localhost:11434/api/tags

Ollama auto-detects CUDA and will use all available VRAM. If a model doesn't fit, it automatically splits layers between GPU and CPU (partial offload). This is powerful but silent — always check GPU utilization to confirm you're not accidentally running on CPU.

Part 2: vLLM — Production Throughput Serving

2.1 Why vLLM Exists

vLLM was born from a 2023 paper called PagedAttention — a KV cache management technique borrowed from OS virtual memory. The core insight: traditional serving frameworks pre-allocate contiguous KV cache blocks per sequence. This creates two problems: fragmentation (wasted VRAM between requests) and the inability to share KV cache across requests with the same prefix.

PagedAttention stores KV cache in fixed-size non-contiguous blocks (like memory pages), mapping logical sequence positions to physical blocks. This enables:

  • Near-zero KV fragmentation — no pre-allocation waste
  • Copy-on-write prefix sharing — identical system prompts share cache
  • Continuous batching — add/remove sequences mid-flight without stopping
vLLM achieves 24x higher throughput than naive HuggingFace generation on identical hardware, according to the original paper. On real workloads with mixed sequence lengths, the gains are typically 8-15x.

2.5 Continuous Batching — The Core Engine

Traditional batching: collect a fixed batch, run it, wait for all sequences to finish, start next batch. If one sequence in batch=32 is 10x longer than others, 31 GPUs sit idle waiting for that one straggly sequence.

Continuous batching (also called iteration-level scheduling): after every single decode step, vLLM checks if any sequence has finished. If yes, it immediately slots in a new prefill. The GPU is never waiting. The batch composition changes dynamically on every forward pass. 

# vLLM scheduler configuration
from vllm import AsyncLLMEngine, AsyncEngineArgs

engine_args = AsyncEngineArgs(
    model="meta-llama/Llama-3.1-8B-Instruct",
    dtype="float16",
    max_num_seqs=256,          # max concurrent sequences in flight
    max_num_batched_tokens=8192, # max tokens processed per iteration
    gpu_memory_utilization=0.90,
    # Block size for PagedAttention (16 = standard)
    block_size=16,
    # Enable prefix caching (share KV for common system prompts)
    enable_prefix_caching=True,
)

engine = AsyncLLMEngine.from_engine_args(engine_args)

2.6 Prefix Caching — Free Speedup for Fixed System Prompts

from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",
    dtype="float16",
    enable_prefix_caching=True,  # THE key flag
)

# First request: system prompt is prefilled and cached (slow)
# Subsequent requests with same system prompt: cache hit (fast)
SYSTEM_PROMPT = "You are an expert ML engineer. " * 100  # long shared prefix

import time

for i in range(5):
    start = time.time()
    output = llm.generate(
        f"<|system|>{SYSTEM_PROMPT}<|user|>Question {i}: What is layer normalization?",
        SamplingParams(max_tokens=100)
    )
    elapsed = time.time() - start
    print(f"Request {i}: {elapsed:.3f}s")
# Request 0: 0.820s (cache miss — full prefill)
# Request 1: 0.112s (cache hit — skipped system prompt)
# Request 2: 0.108s (cache hit)
# ...

Part 3: llama.cpp — Bare Metal, Maximum Control

3.1 What Makes llama.cpp Special

llama.cpp is a pure C/C++ inference engine with zero Python overhead, hand-written SIMD kernels (AVX-512, ARM NEON), and custom CUDA kernels for every quantization type. It introduced the GGUF format — a single-file container for model weights, tokenizer, and metadata. Its killer feature: run any fraction of layers on GPU and the rest on CPU+RAM. If a 13B model doesn't fit in 24 GB VRAM, offload half the layers to RAM and run at reduced throughput rather than not running at all.

Part 4: Quantization Deep Dive

Quantization is the single most impactful technique for fitting models into limited VRAM. But not all quantization is equal. Here's what's actually happening under the hood.

4.1 The Quantization Landscape

# Format comparison for Llama 3.1 8B

Format    | VRAM    | Quality (vs fp16) | Throughput | Use case
----------|---------|-------------------|------------|------------------
fp16      | 16 GB   | 100%              | baseline   | max quality
bf16      | 16 GB   | ~100%             | +5%        | A100/H100
int8 LLM.int8 | 8 GB | ~99%            | -10%       | quality + VRAM
GPTQ int4 | 4.5 GB  | ~97%             | +60%       | balanced
AWQ int4  | 4.5 GB  | ~98%             | +70%       | best 4-bit
Q4_K_M    | 4.8 GB  | ~97%             | +65%       | llama.cpp sweet spot
Q5_K_M    | 5.5 GB  | ~99%             | +55%       | near fp16 quality
Q8_0      | 8.5 GB  | ~99.9%           | +20%       | lossless-ish
EXL2 4bpw | 4.5 GB  | ~97%             | +80%       | ExLlamaV2

4.2 GPTQ — Post-Training Quantization with Layer-Wise Calibration

    model_name,

    device_map="auto"

# Calibration dataset (128 samples is usually enough)
from datasets import load_dataset

# Quantize

# Save

4.3 AWQ — Activation-Aware Weight Quantization

# AWQ: better than GPTQ for same bit-width
# Key insight: not all weights are equally important —
# weights that activate highly-salient neurons matter more.
# AWQ protects those weights by applying per-channel scaling.

pip install autoawq

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "meta-llama/Llama-3.1-8B-Instruct"
quant_path = "./llama3-8b-awq-4bit"

quant_config = {
    "zero_point": True,   # asymmetric quantization

    "version": "GEMM",    # GEMM kernel (GEMV is slower but less memory)

# AWQ calibration (faster than GPTQ — uses activation statistics)

# Save

4.4 GGUF Quantization Types Explained

# GGUF quantization naming convention:
# Q{bits}_{variant}_{size}
#   bits: 2, 3, 4, 5, 6, 8
#   variant: K (k-quant), I (importance matrix)
#   size: S (small), M (medium), L (large) — refers to k-quant block size

# K-quants (recommended): use larger 256-element super-blocks
# with mixed precision — some weights get higher bit depth
# based on their importance to the output distribution.

# Recommendation ladder for a 3090 (24GB):
# Llama 3.1 8B:
#   - Maximum quality: fp16 (16 GB) — fits with room for KV cache
#   - Balanced: Q5_K_M (5.5 GB) — near fp16 quality
#   - Maximum batch: Q4_K_M (4.8 GB) — more VRAM for KV cache
# Llama 3.1 13B:
#   - Q5_K_M (9.5 GB) — excellent quality
#   - Q4_K_M (8.5 GB) — standard choice
# Llama 3.3 70B:
#   - Q4_K_M (43 GB) — needs multi-GPU or CPU offload
#   - IQ2_XXS (19 GB) — ultra-compressed, noticeable quality loss

# Convert your own model to GGUF
python llama.cpp/convert_hf_to_gguf.py \
  ./my-finetuned-model \
  --outfile ./my-model-fp16.gguf \
  --outtype f16

# Then quantize
./build/bin/llama-quantize \
  ./my-model-fp16.gguf \
  ./my-model-Q4_K_M.gguf \
  Q4_K_M

Part 5: Flash Attention — The Algorithmic Breakthrough

Standard attention has O(N²) memory complexity — for a 32K context window, storing the attention matrix requires 32K × 32K × 2 bytes = 2 GB for a single head. Flash Attention rewrites the algorithm to tile the computation, never materializing the full attention matrix in HBM (high-bandwidth memory). 

# Flash Attention 2 memory complexity comparison
# Standard attention: O(N²) HBM reads/writes
# Flash Attention 2: O(N) HBM reads/writes (constant tiling)

# Standard: for 8192 context, 32 heads, 128 head_dim, fp16:
standard_attn_mem = 8192 * 8192 * 32 * 2 / (1024**3)
print(f"Standard attention HBM: {standard_attn_mem:.1f} GB")  # 16.0 GB

# Flash Attention: uses SRAM tiles (48 KB per SM on RTX 3090)
# Processes in blocks, recomputes softmax statistics on the fly
# No O(N²) intermediate matrix in HBM
flash_overhead = 8192 * 128 * 32 * 2 / (1024**3)
print(f"Flash Attention overhead: {flash_overhead*2:.3f} GB")  # ~0.064 GB

# Enable in vLLM:
# --enable-chunked-prefill (vLLM 0.4+, splits long prefills)

# Enable in llama.cpp:
# --flash-attn flag or cmake -DLLAMA_FLASH_ATTN=1

# Enable in HuggingFace Transformers:
from transformers import AutoModelForCausalLM
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B-Instruct",
    torch_dtype="float16",
    attn_implementation="flash_attention_2",  # requires flash-attn package
    device_map="auto"
)

Part 6: Speculative Decoding

The decode phase processes one token at a time — a serial bottleneck. Speculative decoding breaks this by using a small "draft" model to generate k candidate tokens in parallel, then verifying them all in a single forward pass of the large model. If the large model agrees with the draft, you get k tokens for roughly the cost of 1. 

# Speculative decoding in vLLM
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-8B-Instruct \
  --speculative-model meta-llama/Llama-3.2-1B-Instruct \
  --num-speculative-tokens 5 \
  --dtype float16

# How it works:
# 1. Draft model (1B) generates 5 candidate tokens (fast, cheap)
# 2. Target model (8B) scores ALL 5 in one parallel forward pass
# 3. Accept tokens where p_target(x) >= p_draft(x) (rejection sampling)
# 4. On acceptance rate ~0.7-0.8, effective throughput = 2.5-4x

# Python API with speculative decoding
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",

Part 7: Challenges and How to Solve Them

7.1 VRAM OOM (Out of Memory)

# Diagnosis
nvidia-smi --query-gpu=memory.used,memory.free --format=csv

# Solutions in order of impact:
# 1. Reduce max sequence length
vllm serve model --max-model-len 4096  # was 8192

# 2. Reduce GPU memory utilization (leave more for OS/other processes)
vllm serve model --gpu-memory-utilization 0.85

# 3. Switch to lower quantization
# fp16 -> int8 -> int4 (each halves model VRAM footprint)

# 4. Enable chunked prefill (vLLM) — processes long prompts in chunks
python -m vllm.entrypoints.openai.api_server \
  --model llama3 \
  --enable-chunked-prefill \
  --max-num-batched-tokens 2048

# 5. CPU offload (llama.cpp) — put overflow layers on RAM
./llama-cli -m model.gguf -ngl 20  # only 20/32 layers on GPU

# 6. Swap space (vLLM) — overflow KV cache to CPU RAM
python -m vllm.entrypoints.openai.api_server \
  --model llama3 \
  --swap-space 4  # 4 GB CPU swap for KV cache

7.2 Slow Time-to-First-Token (TTFT)

# TTFT is dominated by prefill. For long prompts it compounds fast.
# Llama 3.1 8B fp16, 3090: roughly 1ms per 100 input tokens during prefill

# Solutions:
# 1. Enable Flash Attention (2-4x faster prefill)
# 2. Increase prefill batch size
./llama-server -m model.gguf --batch-size 2048  # was 512

# 3. Chunked prefill (vLLM) — interleave prefill with decode
# reduces TTFT variance at cost of slightly lower throughput

# 4. Prefix caching — reuse KV for repeated system prompts
# (vLLM --enable-prefix-caching, Ollama does this automatically)

# 5. Prompt compression — use LLMLingua or similar to compress
# long system prompts before sending to model

7.3 Slow Token Generation (TGS — Tokens per Second)

# Decode is memory-bandwidth-bound on a single request.
# RTX 3090: 936 GB/s bandwidth.
# Llama 3.1 8B fp16: 16 GB model → 936/16 = ~58 tokens/sec theoretical max
# Actual: ~45-52 tokens/sec at batch=1 (overhead from attention, sampling)

# Solutions:
# 1. Quantize to int4 — halves memory reads → ~2x throughput
# (4.8 GB model → 936/4.8 = ~195 tokens/sec theoretical)
# Actual: ~80-100 tokens/sec with AWQ/GPTQ

# 2. Batch multiple requests — decode steps shared across batch
# batch=8 with Q4 → ~8x throughput at same latency

# 3. Speculative decoding — 2-3x speedup for typical text
# (works best when output is predictable/repetitive)

# 4. Use Flash Decoding (FlashDecoding++) for long contexts
# Standard attention slows down with long KV cache
# Flash Decoding parallelizes across the key/value dimension

7.4 Model Quality / Hallucination Issues

# Sampling parameter tuning

7.5 Context Window Exhaustion

# When conversations exceed context window:

# Option 1: Sliding window — keep last N tokens
def truncate_messages(messages, tokenizer, max_tokens=6000):
    total = 0
    result = []
    for msg in reversed(messages):
        tokens = len(tokenizer.encode(msg['content']))
        if total + tokens > max_tokens:
            break
        result.insert(0, msg)
        total += tokens
    return result

# Option 2: Summarize old turns with a small fast model
def summarize_history(old_messages, summarizer_llm):
    summary_prompt = f"Summarize this conversation in 3 sentences:\n{old_messages}"
    return summarizer_llm.generate(summary_prompt)

Part 8: Advanced Topics

8.1 LoRA — Parameter-Efficient Fine-Tuning + Serving

# LoRA adds low-rank adapter matrices (A and B) to attention layers:
# W_new = W_pretrained + alpha * (A @ B)
# where A is (d x r) and B is (r x d), r << d
# Only A and B are trained — saves 100-1000x parameters vs full fine-tune

# Fine-tune with LoRA using PEFT
pip install peft transformers datasets trl

from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B-Instruct",
    torch_dtype=torch.float16,
    device_map="auto"
)

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                 # rank — 8-64, higher = more capacity
    lora_alpha=32,        # scaling factor (alpha/r = effective LR scale)
    target_modules=[      # which layers to adapt
        "q_proj", "v_proj", "k_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj"
    ],
    lora_dropout=0.05,
    bias="none"
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: 83,886,080 || all params: 8,114,409,472 || 1.033%

# Serving LoRA adapters with vLLM (hot-swap without restart!)

  --enable-lora \
  --lora-modules \

8.2 Multi-GPU Tensor Parallelism

# Tensor Parallelism (TP): split attention heads across GPUs
# Each GPU holds 1/N of the weight matrices
# Requires fast GPU interconnect (NVLink >> PCIe)

# vLLM tensor parallelism
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-70B-Instruct \
  --tensor-parallel-size 4 \  # split across 4 GPUs
  --dtype float16

# Pipeline Parallelism (PP): split layers across GPUs
# GPU 0 runs layers 0-20, GPU 1 runs layers 20-40...
# Higher memory efficiency, but adds pipeline bubble latency

python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-70B-Instruct \
  --tensor-parallel-size 2 \
  --pipeline-parallel-size 2 \  # 2 TP × 2 PP = 4 GPUs total
  --dtype float16

8.3 KV Cache Quantization

# KV cache is fp16 by default — the largest VRAM consumer for long contexts
# FP8 KV cache (vLLM 0.5+): ~2x more context for same VRAM

python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-8B-Instruct \
  --kv-cache-dtype fp8 \           # quantize KV cache to 8-bit
  --max-model-len 32768 \          # now fits in 24GB
  --dtype float16

# INT8 KV cache (llama.cpp)
./llama-server -m model.gguf \
  --cache-type-k q8_0 \   # int8 K cache
  --cache-type-v q8_0 \   # int8 V cache
  --ctx-size 32768

9.2 The Optimization Decision Tree

# Is VRAM the bottleneck? (model doesn't fit or OOMs)
#   → Quantize: fp16 → int8 → int4
#   → Reduce context: --max-model-len 4096
#   → Reduce GPU utilization: --gpu-memory-utilization 0.85
#   → CPU offload: -ngl 20 (llama.cpp)

# Is decode too slow? (low tokens/sec on single request)
#   → Quantize to int4 (2x faster, same latency feel)
#   → Enable speculative decoding (2-3x on predictable text)
#   → Check you're actually on GPU: nvidia-smi dmon

# Is throughput too low? (low tokens/sec on many requests)
#   → Increase max_num_seqs / parallel slots
#   → Enable continuous batching (--cont-batching)
#   → Enable prefix caching for repeated system prompts
#   → Use vLLM instead of Ollama (better scheduler)

# Is prefill too slow? (high TTFT on long prompts)
#   → Enable Flash Attention
#   → Increase batch size for prefill
#   → Enable chunked prefill (vLLM)

# Is quality poor after quantization?
#   → Switch from GPTQ to AWQ (better calibration)
#   → Use higher quant: Q4_K_M → Q5_K_M
#   → Use perplexity evaluation to measure quality loss

Part 10: Tool Selection Guide

When to Use What

# Decision matrix

Use Ollama when:
  - Single developer, personal use
  - Want to be running in 5 minutes
  - Need easy model management (pull/push)
  - Building a quick prototype with OpenAI SDK
  - Need auto GPU/CPU splitting transparently
  Weakness: limited concurrency, no fine-grained scheduler control

Use vLLM when:
  - Serving multiple concurrent users
  - Need maximum throughput (API service, backend)
  - Want LoRA hot-swapping
  - Running production workloads
  - Need structured output / JSON schema enforcement
  - Want detailed Prometheus metrics
  Weakness: Python overhead, requires more setup, heavy dependencies

Use llama.cpp when:
  - Need the absolute minimum VRAM footprint
  - Running on CPU-only or mixed CPU+GPU
  - Embedding in a C++ application
  - Need GGUF format's portability
  - Doing edge deployment or ARM devices
  - Want to run 70B with partial CPU offload
  Weakness: single-process (llama-server has limited concurrency vs vLLM)

Takeaways

  • Memory math first: know exactly how much VRAM your model + KV cache needs before choosing a format or tool.
  • The decode phase is memory-bandwidth-bound, not compute-bound. Quantization helps here more than a faster GPU clock.
  • PagedAttention and continuous batching (vLLM) are the biggest throughput levers for multi-user serving.
  • Flash Attention is always free speed — enable it everywhere that supports it.
  • AWQ beats GPTQ at the same bit-width for quality; Q4_K_M / Q5_K_M are the llama.cpp sweet spots.
  • Prefix caching is a free 5-10x TTFT speedup when your system prompts are long and shared.
  • For your RTX 3090: Llama 3.1 8B at fp16 is the right default. Drop to Q5_K_M if you need more KV cache headroom for longer contexts or higher concurrency.
  • Production stack: vLLM + nginx + Prometheus/Grafana is the repeatable pattern that scales from 1 GPU to a rack.
