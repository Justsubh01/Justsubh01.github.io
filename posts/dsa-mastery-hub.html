DSA Mastery Series — FAANG Interview Complete Guide

Not another "here are 500 problems, good luck" guide. This is a pattern-first, deeply explained series that teaches you how to think about problems — so you can walk into any interview and recognize what's being asked within 60 seconds. 200+ code examples, real aha-moments, and an 8-week path from zero to offer.

Let me be honest with you

I watched a friend grind 600 Leetcode problems over 6 months. He failed 4 Google interviews in a row. Another friend solved exactly 120 problems — but studied them by pattern — and got offers from Meta and Stripe in his first attempt.

The difference wasn't work ethic. It was strategy.

Most people approach DSA prep like this: open Leetcode, pick a random problem, get stuck after 10 minutes, look at the solution, feel bad, repeat. After 6 months they've "done" 300 problems but can't solve anything they haven't seen before.

The pattern-first approach is different. You learn that there are roughly 14 patterns that cover about 90% of what FAANG actually asks. Once you internalize those patterns, a problem you've never seen before becomes: "Oh, this is just sliding window with a hash map" or "This is Dijkstra with a twist on the state." That recognition — that's what gets you the offer.

The real interview question is never the problem itself. It's: can you see through the surface description and identify the underlying structure? A Google interviewer asking about "rotten oranges spreading in a grid" is really asking: do you know multi-source BFS?

The "aha" that changes everything

Here's a concrete example. Two Sum is usually the first problem anyone solves:

  # Given [2, 7, 11, 15], target = 9
  # Find indices of the two numbers that sum to target.

  # Most beginners write this:
  for i in range(len(nums)):
      for j in range(i+1, len(nums)):
          if nums[i] + nums[j] == target:
              return [i, j]
  # O(n^2). Fine. It works.

  # The pattern-thinker immediately sees:
  # "I need to know if (target - current) exists in what I've seen."
  # → Hash map. One pass.

  seen = {}
  for i, num in enumerate(nums):
      complement = target - num
      if complement in seen:
          return [seen[complement], i]
      seen[num] = i
  # O(n). Always.

That jump — from nested loops to "store what I've seen, look up the complement" — is a pattern. And the same exact jump applies to hundreds of other problems: 3Sum, 4Sum, pair with closest sum, subarray with target sum, two strings with matching characters. One mental model, dozens of problems.

This series is about building those mental models, not collecting problem solutions.

How to use this table

In an interview, the moment you read the problem, skim the trigger phrases above. The problem says "find the minimum window that contains all characters of t"? That's sliding window with a frequency map — before you've even thought about code. Problem says "you can take 1 or 2 steps, how many ways to reach the top"? That's dynamic programming — it's Fibonacci in disguise. Start here, not at the code.

Why most people get stuck on DP (and how this series fixes it)

Dynamic Programming has a reputation for being impossible. People see a DP problem, freeze, and write "I'm not good at DP" in their prep notes. Here's the actual reason DP is hard: nobody teaches you how to define the state.

Every DP tutorial starts at the recurrence. But defining the state — what does dp[i] mean? — is 80% of the work. Once you nail that, the recurrence almost writes itself. 

  Coin Change — most people's mental block:
  ─────────────────────────────────────────────────────────────
  Coins = [1, 5, 10], Amount = 12

  Wrong mental model: "I'll try every combination of coins."
  → That's backtracking → O(amount^coins) → too slow.

  Right mental model: define dp[a] = min coins to make amount a.
  Then: for each coin c, dp[a] = min(dp[a], dp[a-c] + 1)
  Because: "if I use coin c once, I need dp[a-c] coins for the rest."

  dp[0]  = 0   (0 coins needed to make 0)
  dp[1]  = 1   (one coin of value 1)
  dp[2]  = 2   (two coins of value 1)
  dp[5]  = 1   (one coin of value 5)
  dp[6]  = 2   (5 + 1)
  dp[10] = 1   (one coin of value 10)
  dp[11] = 2   (10 + 1)
  dp[12] = 3   (10 + 1 + 1)

  The state dp[a] is the key. Everything else is just bookkeeping.

The DP post in this series walks through 15 problems using this exact framework — state first, recurrence second, code last. By the end, you'll be able to define a DP state for problems you've never seen.

Quick Reference: Complexity Cheat Sheet

  Time Complexity at a Glance:
  ────────────────────────────────────────────────────────────────────
  O(1)        Hash map lookup, array index, heap peek
  O(log n)    Binary search, balanced BST ops, heap push/pop
  O(n)        Linear scan, BFS/DFS, sliding window, prefix sum build
  O(n log n)  Merge sort, heap sort, Dijkstra O((V+E) log V)
  O(n²)       Nested loops, naive string match, some DP
  O(2ⁿ)       Subsets enumeration, backtracking (before pruning)
  O(n!)       All permutations

  n size → what complexity you need:
  ────────────────────────────────────────────────────────────────────
  n ≤ 12        O(n!) is fine       (permutations, brute force)
  n ≤ 25        O(2ⁿ) is fine       (backtracking, bitmask)
  n ≤ 500       O(n³) is fine       (Floyd-Warshall, naive DP)
  n ≤ 5,000     O(n²) is fine       (bubble sort, O(n²) DP)
  n ≤ 100,000   O(n log n) needed   (merge sort, Dijkstra)
  n ≤ 10⁶       O(n) needed         (sliding window, BFS)
  n ≤ 10⁹       O(log n) needed     (binary search, math)

  The 14 Patterns — Quick Trigger:
  ────────────────────────────────────────────────────────────────────
  "find pair that sums to X"         → Two Pointers or Hash Map
  "longest/shortest subarray with.." → Sliding Window (variable)
  "max/min in every window of size k"→ Sliding Window + Deque
  "detect cycle in linked list"      → Fast & Slow Pointers
  "missing/duplicate in 1..n"        → Cyclic Sort
  "merge overlapping intervals"      → Sort by start + greedy merge
  "k-th largest / top-k elements"    → Min-heap of size k
  "all combinations / permutations"  → Backtracking
  "how many ways to reach target"    → Dynamic Programming
  "shortest path, no weights"        → BFS
  "shortest path, positive weights"  → Dijkstra (heap)
  "prerequisite / dependency order"  → Topological Sort (Kahn's)
  "are nodes A and B connected?"     → Union-Find
  "prefix / autocomplete matching"   → Trie

  Key one-liners:
  ────────────────────────────────────────────────────────────────────
  Prefix sum:   prefix[i] = prefix[i-1] + arr[i]
  Range sum:    sum(l, r) = prefix[r+1] - prefix[l]
  Binary search: mid = lo + (hi - lo) // 2   ← never (lo+hi)//2
  Heap (size k): push all elements, pop when len > k
  DP state:     dp[i] = "answer for subproblem of size i"
  BFS by level: snapshot = len(queue), process exactly that many

Where to start

If you're a complete beginner: start at 1.1. Don't skip Phase 1. Understanding why O(n²) is catastrophically worse than O(n log n) for large inputs is the foundation everything else builds on.

If you're intermediate (can solve easy problems, struggle with mediums): jump to Phase 3 — Two Pointers & Sliding Window. These patterns alone will unlock a large chunk of mediums immediately. Then come back to fill gaps.

If you have an interview in 2 weeks: read Phase 5 first to understand the meta-strategy, then hammer Phase 3 and Phase 4 with the Leetcode problems listed there.
