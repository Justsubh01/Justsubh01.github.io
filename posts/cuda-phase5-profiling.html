CUDA Profiling — Finding and Fixing Performance Bottlenecks

Good intuition can get you to 30% of GPU peak. Getting to 80%+ requires profiling. nsys shows you the big picture (where time goes). ncu tells you exactly why a kernel is slow. The Roofline model tells you what the theoretical ceiling is. Together, they form a systematic optimization workflow.

The Roofline model for an A100: diagonal = memory-bandwidth-bound (slope = 2 TB/s), flat ceiling = compute-bound (19.5 TFLOPS FP32). Your kernel's arithmetic intensity and throughput plotted on this chart tells you exactly what to optimize: more data reuse (move right) or more compute (move up).

The Roofline Model

The Roofline model answers: "Given my algorithm's arithmetic intensity, what is the maximum achievable performance on this GPU?"

Arithmetic Intensity (AI) = FLOPs performed / bytes transferred from global memory. It measures how much compute you get per byte of memory traffic.

KernelFLOPsBytesAI (FLOPs/byte)Bound by
Vector Add (C=A+B)N adds3N floats0.08Memory
SAXPY (y=ax+y)2N ops3N floats0.17Memory
Stencil 2D (5-pt)5N ops~4N floats0.31Memory
Conv 3x39N ops~2N floats2.25Memory/boundary
Tiled MatMul (T=32)2N^32N^2 floatsN/4Compute for large N

A100 ridge point: 19.5 TFLOPS / 2 TB/s = 9.75 FLOPs/byte. Kernels below the ridge point are memory-bound. Above it, compute-bound. Vector add (AI=0.08) is 122× below the ridge — it will never be compute-bound.

What the Roofline tells you to do: Memory-bound kernel? Increase data reuse (tiling, caching) to shift AI right. Compute-bound kernel? Optimize arithmetic (use Tensor Cores, FP16, fused operations). Already at the roof? You're done — no algorithmic improvement possible.

Nsight Systems (nsys) — Timeline View

nsys gives you a system-wide timeline: CPU threads, GPU kernels, memory copies, API calls, NVTX ranges. Use it first to understand where time goes. 

# Basic profiling
nsys profile --stats=true ./myprogram

# With specific trace types
nsys profile   --trace=cuda,nvtx,osrt   --output=profile_report   ./myprogram

# View in GUI:
nsys-ui profile_report.nsys-rep

# Or get text summary:
nsys stats profile_report.nsys-rep
# nsys text summary output: CUDA Kernel Statistics (nanoseconds): Time(%) Total Time Instances Avg Name ------- ---------- --------- --- ---- 78.3% 1,248,432 1000 1,248 vectorAdd 12.1% 193,024 500 386 reduction 5.4% 86,112 100 861 matmul 4.2% 67,008 50 1,340 transpose CUDA Memory Operations: Time(%) Total Time Operations Avg Name ------- ---------- ---------- --- ---- 61.2% 982,400 1000 982 CUDA memcpy HtoD 35.8% 573,376 1000 573 CUDA memcpy DtoH 3.0% 48,128 100 481 CUDA memcpy DtoD

From this output: vectorAdd takes 78% of kernel time — that's where to focus. Also: memcpy takes 1.5ms vs 1.2ms kernel time — you're transfer-bound. Consider streams to overlap them.

NVTX Ranges for Custom Profiling

#include <nvtx3/nvToolsExt.h>

void myFunction() {
    nvtxRangePush("Data Preparation");
    prepareData();
    nvtxRangePop();

    nvtxRangePush("GPU Computation");
    myKernel<<<grid, block>>>(d_data, N);
    cudaDeviceSynchronize();
    nvtxRangePop();
}
// These show up as labelled regions in the nsys timeline

Nsight Compute (ncu) — Kernel Deep Dive

ncu profiles individual kernels with hardware performance counters. It shows you occupancy, memory throughput, compute throughput, instruction mix, and stall reasons. 

# Full profiling of all kernels (slow: 5-20x slower due to counter collection)
ncu ./myprogram

# Profile specific kernel only
ncu --kernel-name "vectorAdd" ./myprogram

# Show summary metrics
ncu --set full ./myprogram

# Export for GUI
ncu -o ncu_report ./myprogram
ncu-ui ncu_report.ncu-rep

# Specific metrics
ncu --metrics   sm__throughput.avg.pct_of_peak_sustained_elapsed,  l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum,  smsp__thread_inst_executed_pred_on_per_inst_executed.ratio   ./myprogram

The Optimization Workflow

A systematic process that works for any kernel:

  1. Profile with nsys first. Find the hottest kernel (most time).
  2. Profile that kernel with ncu. Look at the Summary section.
  3. Identify the bottleneck:
    • Memory throughput > 70%? Memory-bound. Optimize memory access.
    • SM throughput > 70%? Compute-bound. Optimize arithmetic.
    • Occupancy < 50%? Resource-bound. Check registers and shared memory.
  4. Apply one fix at a time. Re-profile after each change.
  5. Stop when you hit the Roofline ceiling.
The most important rule of optimization: Never optimize without profiling. Intuition is wrong more than 50% of the time for GPU code. The bottleneck is almost never where you expect it. Profile first, always.

Diagnosing Memory-Bound Kernels

# Key ncu metrics for memory-bound diagnosis:
ncu --metrics   l1tex__t_sectors_pipe_lsu_mem_global_op_ld.sum,  l1tex__t_requests_pipe_lsu_mem_global_op_ld.sum,  l1tex__t_bytes_pipe_lsu_mem_local_op_ld.sum,  l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld.sum   ./myprogram
# Metrics to look for: sectors/request ratio: 1.0 => perfectly coalesced (1 cache line per warp) 32.0 => fully uncoalesced (32 cache lines per warp) local memory load bytes > 0: => register spilling to global DRAM! Reduce registers or add __launch_bounds__ shared bank conflicts > 0: => add +1 padding to 2D shared arrays DRAM throughput < 50% of peak: => consider whether L2 cache is absorbing hits (check L2 metrics separately)

Diagnosing Compute-Bound Kernels

# SM utilization and instruction mix:
ncu --metrics   sm__throughput.avg.pct_of_peak_sustained_elapsed,  smsp__inst_executed_pipe_fma.avg.pct_of_peak_sustained_active,  smsp__inst_executed_pipe_tensor.avg.pct_of_peak_sustained_active,  smsp__inst_executed_pipe_alu.avg.pct_of_peak_sustained_active   ./myprogram
# FP32 compute bound: FMA pipe near 100% FMA throughput: 94.2% of peak # excellent Tensor Core usage: 0.0% # opportunity: could use Tensor Cores for matmul # If tensor cores available for your operation (matmul, conv): # Use cublasGemmEx with CUDA_R_16F for 4x throughput vs FP32 # Or use __mma_sync() for manual Tensor Core control

Occupancy Profiling

# Check what limits occupancy:
ncu --metrics   sm__warps_active.avg.pct_of_peak_sustained_active,  launch__occupancy_limit_warps,  launch__occupancy_limit_registers,  launch__occupancy_limit_shared_mem,  launch__occupancy_limit_blocks   ./myprogram
# Low occupancy due to registers: Achieved Occupancy: 25.0% Occupancy Limit (registers): 25.0% # registers are the bottleneck Registers per thread: 128 # Fix: reduce register usage, or add __launch_bounds__(blockSize, minBlocks) # or --maxrregcount=64 compile flag # Low occupancy due to shared memory: Achieved Occupancy: 33.3% Occupancy Limit (shared mem): 33.3% # shared memory is the bottleneck Shared Memory per Block: 64 KB # 3 blocks max per SM (3 * 64 = 192 KB) # Fix: reduce shared memory per block, or use smaller tiles
