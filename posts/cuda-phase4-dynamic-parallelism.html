CUDA Mastery Phase 4.5 — Dynamic Parallelism: Kernels Launching Kernels

How CUDA dynamic parallelism lets GPU kernels spawn their own child kernels — enabling adaptive algorithms where the GPU decides its own work distribution without any CPU involvement between steps.

Normally, kernels are launched from the CPU. The CPU decides the grid size, block size, and when to launch. But what if the amount of work isn't known until the GPU is already running? Dynamic parallelism solves this: threads on the GPU can launch new kernels directly, without going back to the CPU.

1. What Dynamic Parallelism Is

  Without dynamic parallelism:
  ─────────────────────────────────────────────────
  CPU: launch kernel A
  GPU: kernel A runs, decides it needs more work
  GPU: kernel A finishes (can't launch anything)
  CPU: (reads result, decides how much work to do)
  CPU: launch kernel B with appropriate grid size
  GPU: kernel B runs

  Total: 2 round-trips CPU↔GPU (each ~tens of microseconds)

  With dynamic parallelism:
  ─────────────────────────────────────────────────
  CPU: launch parent kernel A
  GPU: kernel A runs
  GPU: kernel A itself launches child kernel B (no CPU involved!)
  GPU: child kernel B runs
  GPU: parent A (optionally) waits for B to complete
  GPU: parent A finishes
  CPU: receives final result

  Total: 1 CPU launch, GPU handles everything internally

2. Basic Syntax

  // Compile with: nvcc -arch=sm_35 -rdc=true -o prog prog.cu
  //   -rdc=true = "relocatable device code" — required for dynamic parallelism

  __global__ void child_kernel(float *data, int n) {
      int idx = blockIdx.x * blockDim.x + threadIdx.x;
      if (idx < n)
          data[idx] *= 2.0f;
  }

  __global__ void parent_kernel(float *data, int n) {
      // Thread 0 of block 0 launches the child:
      if (threadIdx.x == 0 && blockIdx.x == 0) {
          int childGrid  = (n + 255) / 256;
          int childBlock = 256;

          // Launch from device code — same syntax as from host!
          child_kernel<<<childGrid, childBlock>>>(data, n);

          // Wait for child to complete (device-side synchronize):
          cudaDeviceSynchronize();
          // ↑ This is the GPU calling cudaDeviceSynchronize(), NOT the CPU!
      }
  }

  // From host:
  parent_kernel<<<1, 1>>>(d_data, N);

3. Parent-Child Relationship

  Memory visibility:
  ──────────────────────────────────────────────────────────────
  • Child kernel can access the SAME global memory as parent
  • Shared memory of the parent is NOT visible to child (different blocks)
  • Local memory is private and not shared

  Synchronization:
  ──────────────────────────────────────────────────────────────
  • Parent does NOT automatically wait for children
  • Must explicitly call cudaDeviceSynchronize() from device code
    (or use streams and events on the device side)
  • When parent thread block completes, all child kernels launched
    by that block MUST have completed (implicit sync at block exit)

  Grid level:
  ──────────────────────────────────────────────────────────────
  • Each thread can launch a child kernel independently
  • This can create massive fan-out: 1M threads each launching 1M child threads
    → potential for 10^12 threads (handle with care!)
  • Depth limit: 24 levels of nesting

4. Adaptive Algorithms — Real Use Case

  // Example: sparse matrix-vector multiplication
  // Rows have variable number of non-zeros — hard to balance with fixed grid

  __global__ void spmv_adaptive(int *row_ptr, int *col_idx,
                                 float *values, float *x, float *y, int n) {
      int row = blockIdx.x * blockDim.x + threadIdx.x;
      if (row >= n) return;

      int nnz = row_ptr[row + 1] - row_ptr[row];  // non-zeros in this row

      if (nnz <= 32) {
          // Small row: handle inline with one thread
          float sum = 0.0f;
          for (int j = row_ptr[row]; j < row_ptr[row + 1]; j++)
              sum += values[j] * x[col_idx[j]];
          y[row] = sum;
      } else {
          // Large row: launch a child kernel with a full block
          // GPU decides the work distribution without CPU involvement!
          process_large_row<<<1, 256>>>(
              row_ptr[row], row_ptr[row+1],
              col_idx, values, x, &y[row]);
          cudaDeviceSynchronize();
      }
  }

5. Recursive Algorithms

  // Quicksort on GPU using dynamic parallelism:
  __global__ void quicksort(float *data, int lo, int hi) {
      if (lo >= hi) return;  // base case

      // Partition (done by a single thread for simplicity):
      if (threadIdx.x == 0 && blockIdx.x == 0) {
          int pivot_idx = partition(data, lo, hi);

          // Launch child kernels for each half in parallel:
          if (pivot_idx - 1 > lo)
              quicksort<<<1, 1>>>(data, lo, pivot_idx - 1);
          if (pivot_idx + 1 < hi)
              quicksort<<<1, 1>>>(data, pivot_idx + 1, hi);

          cudaDeviceSynchronize();
      }
  }

  // From host:
  quicksort<<<1, 1>>>(d_data, 0, N - 1);
  cudaDeviceSynchronize();

  // NOTE: This is illustrative. In practice, GPU quicksort needs careful
  // optimization to avoid excessive kernel launch overhead. CUB or thrust::sort
  // is the production choice for GPU sorting.

6. Performance Considerations

  Child kernel launch overhead:
  ─────────────────────────────────────────────────────────────
  A child kernel launch from device code costs ~10–50 microseconds.
  This is 10–50× more than a host-side kernel launch (~1 µs typical).

  When dynamic parallelism makes sense:
  ✓ Amortized work per child is large (milliseconds, not microseconds)
  ✓ Work distribution truly not known until runtime
  ✓ Reduces CPU round-trips in iterative algorithms
  ✓ Recursive algorithms (trees, subdivision, AMR)

  When it does NOT make sense:
  ✗ Fine-grained child kernels (tiny work, high launch overhead ratio)
  ✗ When work is known at compile time or host side
  ✗ For embarrassingly parallel problems (just launch from host!)
  ✗ Performance-critical inner loops
  ─────────────────────────────────────────────────────────────

  Rule of thumb: each child kernel should do at least 1ms of work
  to justify the ~50µs launch overhead (50× overhead = 2% efficiency loss).

7. Requirements and Compilation

  Requirements:
  • Compute capability 3.5 or higher (Kepler+)
  • Must compile with -rdc=true (relocatable device code)
  • Must link with -lcudadevrt (CUDA device runtime library)

  Compilation:
  # Single file:
  nvcc -arch=sm_70 -rdc=true -o prog prog.cu -lcudadevrt

  # Multi-file:
  nvcc -arch=sm_70 -rdc=true -c file1.cu -o file1.o
  nvcc -arch=sm_70 -rdc=true -c file2.cu -o file2.o
  nvcc -arch=sm_70 file1.o file2.o -o prog -lcudadevrt

Takeaways

  • Dynamic parallelism: GPU kernels can launch child kernels without CPU involvement.
  • Use it when work distribution isn't known until the GPU is already running.
  • Child kernels can access the same global memory as the parent.
  • Explicit synchronization with device-side cudaDeviceSynchronize() is required.
  • Child kernel launch overhead is ~50µs — only worth it for large chunks of work.
  • Best for: recursive algorithms, adaptive refinement, sparse/irregular workloads.
  • Requires -rdc=true and compute capability 3.5+.
